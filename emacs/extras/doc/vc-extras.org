This manual describes the features and customization options for the Emacs Lisp file =vc-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=vc-extras.el= extends Emacs' built-in =vc.el= with commands for
managing GitHub repositories through the =gh= CLI tool.  It depends on
the =paths= package for directory constants and on =transient= for its
command menu.

The package provides a complete repository lifecycle from within Emacs:
creating repositories on GitHub, cloning them locally (with submodule
support), renaming them (both locally and on GitHub), and deleting local
copies.  It supports multiple GitHub accounts through a profile system
that associates each account with a local directory, so cloned
repositories automatically land in the right place.

A distinctive feature is the ability to "split" the =.git= directory out
of the repository's working tree into a centralized location.  This is
useful when the working tree resides in a synced folder (such as
Dropbox) and you want to avoid syncing the often-large =.git= directory.
The splitting replaces the =.git= directory with a small text file
containing a =gitdir:= pointer, a mechanism natively supported by Git.

All major operations are accessible through the ~vc-extras-menu~
transient ([[#h:transient-menu][Transient menu]]).

* User options
:PROPERTIES:
:CUSTOM_ID: h:user-options
:END:

** GitHub account configuration
:PROPERTIES:
:CUSTOM_ID: h:github-accounts
:END:

#+vindex: vc-extras-gh-executable
#+vindex: vc-extras-github-account-personal
#+vindex: vc-extras-github-account-work
#+vindex: vc-extras-personal-repo-dir
#+vindex: vc-extras-work-repo-dir
The user option ~vc-extras-gh-executable~ specifies the path to the
=gh= CLI executable.  It defaults to the result of ~executable-find~, so
it is typically auto-detected.  If =gh= is not in your =PATH=, set this
to the full path.

The user option ~vc-extras-github-account-personal~ specifies the GitHub
username for the personal profile.  The default value is ="benthamite"=.

The user option ~vc-extras-github-account-work~ specifies the GitHub
username or organization name for the work profile.  The default value
is ="tlon-team"=.

The user option ~vc-extras-personal-repo-dir~ specifies the local
directory where personal repositories are stored.  It defaults to the
value of ~paths-dir-personal-repos~.

The user option ~vc-extras-work-repo-dir~ specifies the local directory
where work repositories are stored.  It defaults to the value of
~paths-dir-tlon-repos~.

** Profile system
:PROPERTIES:
:CUSTOM_ID: h:profile-system
:END:

#+vindex: vc-extras-profiles
The user option ~vc-extras-profiles~ defines a list of GitHub profiles.
Each profile is a property list with three keys: ~:name~ (a symbol like
~personal~ or ~work~), ~:account~ (the GitHub username string), and
~:dir~ (the local directory path for repositories belonging to that
account). The Customize interface renders one editable field per key,
making the schema explicit when adding a new profile.

The profile system drives most of the package's behavior.  When cloning
a repository, the account determines the target directory.  When
listing local repositories for deletion or renaming, all profile
directories are searched.  The default value defines two profiles
matching the personal and work accounts described above.

** Git directory splitting
:PROPERTIES:
:CUSTOM_ID: h:split-repo-option
:END:

#+vindex: vc-extras-split-repo
The user option ~vc-extras-split-repo~ controls whether the =.git=
directory should be moved to a separate centralized location when
cloning a repository ([[#h:splitting-git-directory][Splitting the git directory]]).

The accepted values are:

- ~nil~: never split.
- ~prompt~ (the default): ask the user each time.
- ~t~ (or any other non-~nil~ value): always split.

This option only affects the behavior of ~vc-extras-clone-repo~.  You
can always split an existing repository manually with
~vc-extras-split-local-repo~.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Creating repositories
:PROPERTIES:
:CUSTOM_ID: h:creating-repos
:END:

#+findex: vc-extras-create-repo
The command ~vc-extras-create-repo~ creates a new repository on GitHub
using the =gh= CLI.  It prompts for a name, an account (from the
profiles in ~vc-extras-profiles~), a description, and whether the
repository should be private.  After creation, it offers to clone the
new repository locally using ~vc-extras-clone-repo~
([[#h:cloning-repos][Cloning repositories]]).

** Cloning repositories
:PROPERTIES:
:CUSTOM_ID: h:cloning-repos
:END:

#+findex: vc-extras-clone-repo
The command ~vc-extras-clone-repo~ clones a GitHub repository into the
local directory associated with the repository's account.  When called
without arguments, it fetches the list of repositories from GitHub (via
=gh repo list=) and presents them for completion.

The clone uses =git clone --recurse-submodules= to ensure submodules
are initialized.  After cloning, the command:

1. Initializes and updates submodules, then checks out the default
   branch (=main= or =master=) in each submodule.
2. Optionally splits the =.git= directory based on the
   ~vc-extras-split-repo~ user option ([[#h:split-repo-option][Git directory splitting]]).
3. Optionally prompts to add the repository to Forge (via
   ~forge-extras-track-repository~), unless the NO-FORGE argument is
   non-~nil~.

With a prefix argument, the command prompts for the target clone
directory.  The choices include the account's default directory, the
current elpaca sources directory (if available), and an option to
select any other directory.

The clone runs asynchronously and reports progress via messages.

** Renaming repositories
:PROPERTIES:
:CUSTOM_ID: h:renaming-repos
:END:

#+findex: vc-extras-rename-repo
The command ~vc-extras-rename-repo~ renames a repository both locally
and on GitHub.  It prompts for the repository to rename (with
completion from local repositories), the new name, and the account if
it cannot be determined automatically.

The command performs four operations:

1. Renames the local directory.
2. Renames the split =.git= directory (if one exists) and updates the
   =.git= pointer file.
3. Renames the repository on GitHub via the =gh= API.
4. Updates the Git remote URL in the local configuration.

Repository name arguments are shell-quoted when passed to =gh= and =git=
commands.  If any step fails (e.g., the remote rename returns an error),
the command reports a warning but continues with the remaining steps.

** Deleting local repositories
:PROPERTIES:
:CUSTOM_ID: h:deleting-repos
:END:

#+findex: vc-extras-delete-local-repo
The command ~vc-extras-delete-local-repo~ deletes a local Git
repository.  It prompts for the repository name with completion from
all local Git repositories found in the profile directories.

The command deletes both the main repository directory (if it exists and
is a Git repository) and the corresponding split =.git= directory in
~paths-dir-split-git~ (if it exists).  A message lists the directories
that were actually deleted.

** Splitting the git directory
:PROPERTIES:
:CUSTOM_ID: h:splitting-git-directory
:END:

#+findex: vc-extras-split-local-repo
The command ~vc-extras-split-local-repo~ moves the =.git= directory
from a repository to a centralized location under ~paths-dir-split-git~
and replaces it with a =.git= file containing a =gitdir:= pointer.
This is useful when the working tree is in a synced folder (e.g.,
Dropbox) and you want to avoid syncing the potentially large =.git=
directory.

If the repository has submodules, the command also updates their =.git=
pointer files and runs ~git submodule sync --recursive~ to ensure
everything remains consistent.

** Checking authentication
:PROPERTIES:
:CUSTOM_ID: h:checking-auth
:END:

#+findex: vc-extras-check-gh-authenticated
The command ~vc-extras-check-gh-authenticated~ verifies that the =gh=
CLI is properly authenticated by running =gh auth status=.  It displays
a success message or signals an error with instructions to authenticate.

** Stripping diff markers
:PROPERTIES:
:CUSTOM_ID: h:strip-diff-markers
:END:

#+findex: vc-extras-strip-diff-markers
The command ~vc-extras-strip-diff-markers~ removes leading =+= or =-=
characters from each line in the active region.  This is useful when you
paste diff output into a buffer and want to clean it up into plain text.

** Transient menu
:PROPERTIES:
:CUSTOM_ID: h:transient-menu
:END:

#+findex: vc-extras-menu
The command ~vc-extras-menu~ presents a transient menu providing quick
access to all major =vc-extras= commands: creating, cloning (with and
without directory confirmation), splitting, renaming, deleting, and
checking authentication status.

* Functions
:PROPERTIES:
:CUSTOM_ID: h:functions
:END:

** Repository information
:PROPERTIES:
:CUSTOM_ID: h:repo-information
:END:

#+findex: vc-extras-is-git-dir-p
#+findex: vc-extras-has-submodules-p
#+findex: vc-extras-get-github-remote
#+findex: vc-extras-get-account-of-name
#+findex: vc-extras-get-account-prop
#+findex: vc-extras-get-repo-dir
#+findex: vc-extras-resolve-repo-dir
#+findex: vc-extras-list-local-candidates
The function ~vc-extras-is-git-dir-p~ returns non-~nil~ if a directory
is a Git repository.  It uses ~locate-dominating-file~ to check for a
=.git= entry, which correctly handles both regular =.git= directories
and =gitdir:= pointer files created by ~vc-extras-split-local-repo~
([[#h:splitting-git-directory][Splitting the git directory]]).

The function ~vc-extras-has-submodules-p~ checks whether a =.gitmodules=
file exists in a directory, indicating that the repository has
submodules.

The function ~vc-extras-get-github-remote~ constructs the HTTPS GitHub
remote URL for a repository name and account.

The function ~vc-extras-get-account-of-name~ determines which GitHub
account owns a repository by querying =gh repo list= for each profile.

The function ~vc-extras-get-account-prop~ retrieves a property (such as
~:dir~) for a given account from ~vc-extras-profiles~.

The function ~vc-extras-get-repo-dir~ constructs the local path for a
repository, optionally pointing to the =.git= or split =.git=
directory.

The function ~vc-extras-resolve-repo-dir~ looks up a repository's local
directory from a candidates alist or constructs it from the account.

The function ~vc-extras-list-local-candidates~ returns an alist of
=(NAME . DIR)= pairs for all local Git repositories found in the profile
directories.

** GitHub CLI operations
:PROPERTIES:
:CUSTOM_ID: h:gh-operations
:END:

#+findex: vc-extras-gh-create-repo
#+findex: vc-extras-ensure-gh-exists
#+findex: vc-extras-gh-list-repos
#+findex: vc-extras-gh-list-repos-in-account
#+findex: vc-extras-gh-list-repos-in-all-accounts
The function ~vc-extras-gh-create-repo~ executes the =gh repo create=
command with the given name, account, description, and visibility.

The function ~vc-extras-ensure-gh-exists~ signals an error if =gh= is
not installed.

The function ~vc-extras-gh-list-repos~ lists repositories for a
specific account or all accounts.  It delegates to
~vc-extras-gh-list-repos-in-account~ and
~vc-extras-gh-list-repos-in-all-accounts~.
