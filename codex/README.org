* Overview

This directory contains the version-controlled parts of my [[https://developers.openai.com/codex/][Codex]] setup.
The cross-tool synchronization policy is documented in =../agents/README.org=.
This file covers the Codex-specific activation files and local conventions.

Claude and Codex artifacts are peer implementations of shared agent behavior.
Codex gets explicit counterpart files where Codex syntax, hook payloads,
discovery rules, or instruction-loading semantics require them.

* Current layout

#+begin_example
codex/

├── config.toml            <- live ~/.codex/config.toml symlink target
├── hooks.json             <- live ~/.codex/hooks.json symlink target

Repo-root =.codex/= is reserved for project-local Codex config. Do not use
=~/My Drive/dotfiles/.codex/= as a target for live global =~/.codex/= symlinks;
that makes Codex load the same file as both global and dotfiles-project-local
configuration. The topology table in =../agents/README.org= is the
authoritative cross-tool reference.

* Instructions

Codex reads global user instructions from =AGENTS.override.md= and =AGENTS.md=
in =$CODEX_HOME=, so the live =~/.codex/AGENTS.md= points to the tracked
=codex/AGENTS.md= file.

Project-level Codex instructions live in =AGENTS.md= files next to their
Claude =CLAUDE.md= peers. The old =project_doc_fallback_filenames = ["CLAUDE.md"]=
setting is intentionally not used, so every project-local =CLAUDE.md= should
have a sibling =AGENTS.md= counterpart. The Claude/Codex sync hooks treat these
instruction files as project-local peers in the git repo being edited or
committed.

The tracked =codex/AGENTS.md= is the Codex-side peer of =claude/CLAUDE.md=. It
uses Codex-safe syntax: the title is tool-specific, Claude's
=@context/secrets.md= import is expanded into an explicit secrets rule, and
relative context/tool paths are absolute so they resolve correctly from any
project. =bin/ai-config-sync= normalizes those tool-specific forms before
checking instruction equivalence. The paired global instructions also ask the
agent to alert the user when they direct abusive language at the assistant/model,
mention moral uncertainty about possible model sentience, and request a civil,
task-focused restatement. They also require completion-claim discipline: agents
must not say a bug is fixed, resolved, working, or done unless the exact
reported user-visible behavior has been verified after the change, and final
answers after changes must include an explicit =Verification:= or =Not verified
end-to-end:= sentence.

It also records the manual Chrome profile routing used by browser-only service
flows: =Default= for personal projects and =Profile 2= for Epoch/work projects
(=pablo@epoch.ai=).

=codex/config.toml= enables experimental goals support under =[features]=. It
also sets =suppress_unstable_features_warning = true= so Codex does not print
the startup warning on every launch.

* Skills

=codex/skills/= contains the tracked Codex global skills. Most paired skills
share the same bodies and auxiliary files across tools. The sync check removes
tool-specific frontmatter keys such as
=user-invocable=, =allowed-tools=, =argument-hint=, and =model= before
comparison. Tool-specific skills may
intentionally diverge when agents store state differently; these cases are
recorded in =ai-config-sync.json=.
The cross-tool master inventory of global, programmatic, system, and
project-local skills lives in =../agents/README.org=, so agents can discover
skills even when they are local to another project. Reusable skills that
graduate out of project-local roots should be added here, paired in
=claude/skills/= where possible, and recorded in =ai-config-sync.json=.

For paired skills that keep small state files in their skill directories, update
both tool copies to the same contents unless =ai-config-sync.json= records an
explicit tool-specific divergence. Project-local skill pairs follow the same
paired-state rule as global skills.

Codex discovers global skills from =~/.codex/skills/*/SKILL.md=. The live
directory is a symlink to the tracked Codex tree. Run:

#+begin_src sh
codex/bin/install-skills.sh
#+end_src

The script ensures =~/.codex/skills= points at =codex/skills/= and applies any
native skill-disable settings recorded by =bin/skill-prune=. Local reusable
Codex skills belong in =codex/skills/=, not as untracked directories under
=~/.codex/skills=. Programmatic-only skills are not available in ordinary Codex
sessions. They live in =programmatic-skills/= directories so
=agent-run-skill= can find and run them explicitly from Emacs without adding
them to every interactive Codex session. The installer exposes those directories
at =~/.codex/programmatic-skills= because =agent-run-skill= discovers Codex
skills from that live path. Codex-managed hidden system skills, when present,
live under ignored =codex/skills/.system/= runtime data and are not part of the
reusable cross-tool skill parity model.
Private skills that should remain hidden from ordinary interactive sessions
live under =codex/private-skills/= with paired Claude copies under
=claude/private-skills/=. Those trees and =ai-config-sync.private.json= are
git-crypt-encrypted, so neither the content nor the private sync manifest is
publicly readable. =bin/ai-config-sync= loads the encrypted private manifest
locally and still checks parity. Public private-skill directories use opaque
IDs; the real skill names live only in the encrypted private manifest. Private
skills are exposed to local Codex sessions by =bin/install-private-skills=,
which creates gitignored shims under =~/.codex/skills= from the encrypted
manifest.

Skill frontmatter descriptions are loaded for routing, so keep them concise:
preserve user phrasing, scope boundaries, and important near-miss exclusions,
but move implementation details to the skill body.

To reduce context pressure from stale skills, run:

#+begin_src sh
bin/skill-prune apply --days 60
#+end_src

The prune command scans recent Claude and Codex transcripts. It disables only
paired skills for which neither tool shows usage inside the threshold window;
if either Claude or Codex used the skill recently, both tools keep it enabled.
It includes global paired skills and dynamically discovered project-local skill
pairs under =~/My Drive=; override the discovery roots with
=SKILL_PRUNE_LOCAL_ROOTS=, using the platform path separator for multiple roots.
Disabled skills are recorded in =skills-disabled.json= and applied to Codex as
=[[skills.config]]= entries with =enabled = false= in =codex/config.toml=.
Re-enable a disabled skill with
=bin/skill-prune enable NAME=; add =--keep= when a renamed or otherwise
intentionally retained skill should remain enabled despite missing history.

Project-specific skills belong beside their project instead of this global
tree:

- Dotfiles maintenance: =.codex/skills/= at the dotfiles repo root. They are
  available while working in dotfiles without loading globally in unrelated
  projects.
- Emacs-only maintenance: =~/My Drive/dotfiles/emacs/.codex/skills/=.
- macOS maintenance: =~/My Drive/dotfiles/macos/.codex/skills/=.
- Tlön issue triage: =~/.config/emacs-profiles/8.3.0-dev/elpaca/sources/tlon/.codex/skills/=.
- Notes workflows: =~/My Drive/notes/.codex/skills/=.
- Batch-only notes workflows: the matching =programmatic-skills= roots under
  =~/My Drive/notes/.codex/=.
- Google Drive cleanup: =~/My Drive/.codex/skills/=.
- Epoch-only workflows: =~/My Drive/Epoch/.codex/skills/=.

The sync reminder and commit guard apply to these project-local pairs too:
changing one local skill reminds the agent to update the other, and committing
one side without an equivalent counterpart is blocked. Each project-local skill
root documents its skills in that project's =README.org= under a dedicated
=Skills= section.

When switching Emacs/elpaca profiles, the emacs project-local
=migrate-profile= skill migrates project-local agent roots that live inside
profile package clones: =<old-profile>/elpaca/{sources,repos}/<package>/.codex/=
is copied into the verified new-profile package path without overwriting
existing target files.

=bin/codex-app-handoff= opens the current Codex thread in the desktop app via
the app's =codex://threads/<THREAD_ID>= deeplink. Use it from CLI/Emacs turns
when a task needs App-only Browser Use tools; the next turn must be sent from
the Codex App for the Browser plugin / Node REPL tools to appear. The command
also copies a short continuation prompt to the Emacs kill ring and macOS
clipboard, because the existing-thread deeplink opens the thread but does not
submit a new message.

When changing a skill:

1. Edit the side requested by the user.
2. Update the counterpart in the same session.
3. Stage both sides. For global skills, update =ai-config-sync.json= with a
   clear artifact-specific status or exemption if a counterpart is intentionally
   absent.

When renaming a global Codex skill, rename the Codex directory, update its
=name= frontmatter value, and follow the synchronization policy in
=../agents/README.org=. When removing a global Codex skill, remove the Codex
copy and update =ai-config-sync.json= according to that policy. Project-local
Codex replacements should live in the target project's =.codex/skills/=
directory.

Skill files must be named uppercase =SKILL.md=.  This matters on the Google
Drive mount and for Codex skill discovery.

The =twitter= skill is a programmatic Codex workflow for direct Twitter/X
lookup, search, and save requests via the active skill copy's
=lib/twitterapi.sh= curl wrapper around twitterapi.io. It supports tweet,
thread/reply, timeline, profile, follower/following, user-search, and keyword

=twitter-discover=, and =twitter-vet= workflows; and requires response,

The =twitter-digest= skill is a programmatic Codex workflow for fetching recent
tweets from a named curated account list, triaging them against the list rubric,
saving an org digest, and updating last-run state. It uses the active
=fetch-tweets.sh= helper and active =twitterapi.sh= wrapper rather than MCP

timestamps only into the current tool's runtime tree, keeps curated list files
synchronized through the sync policy, uses the shared =twitter-vet= registry,
fails loudly when no listed account fetch succeeds, and verifies fetch
markers, saved digest, last-run timestamp, diff cleanliness, and staged paths
before committing digest, timestamp, list, and vet registry changes.

The =twitter-discover= skill is a programmatic Codex workflow for iterative,
graph-based discovery of high-value Twitter accounts in any niche. It
interviews the user for criteria, acquires seed accounts via search, expands
through hot promise scores from following-list overlap, uses the active
Codex skill root for wrappers and saved state, keeps API responses in a
per-session temporary directory, verifies its parsing helper, re-reads saved
logs/lists/registries, and optionally exports high-scoring accounts to a
=twitter-digest= list.

The =twitter-vet= skill is a programmatic Codex workflow for vetting, scoring,
ranking, or deciding whether Twitter/X accounts belong on curated lists. It
requires a handle, list name, rubric, and optional =vet-threshold=; uses the
active =twitter= skill wrapper for read-only tweet sampling; applies a quick
filter plus a full 1-10 scoring pass with the shared epistemic quality gate; and
keeps a shared vet registry so Codex callers do not split history. It auto-adds
accounts only for existing digest lists, returns scores to =twitter-discover=

commits.

The =handoff= skill saves next-session instructions to

The =verify= skill is a Codex workflow for running an explicit
user-requested verification loop when the user asks to verify, close the loop,
define success criteria before work, or rigorously check an output whose correctness is not
automatically answerable. It generates concrete criteria before acting,
meta-verifies criteria quality, proceeds without a user checkpoint when criteria
are clear, asks only when criteria require user/domain judgment or change
scope/risk, evaluates each criterion, loops on failures, and reports
pass/fail/uncertain evidence. Ordinary coding and debugging use narrower
project checks, and Superpowers owns the ordinary final completion gate, unless
this meta-verification workflow was explicitly requested.

The =walk-list= skill is a Codex workflow for strict item-by-item
processing of JSON arrays, newline-delimited JSON, or newline-separated text
files. It moves the source file into the protected walk-list store, leaves a
stub with quoted commands generated from the same =walk.py= path that started
the walk, validates positive concurrency and non-negative stale-claim
thresholds, and distinguishes strict isolation/concurrency work from ordinary
summarizing, filtering, sorting, or bulk list transformations.

The =overnight-todos= skill is a Codex batch workflow that pulls active
org-roam TODOs (no scheduled or deadline date) via
=org-roam-extras-dump-actionable-todos=, classifies each via =triage.py=,
and in =--mode act= dispatches per-TODO subagents through =walk-list= to do
work end-to-end on those that can be completed without user input.
=triage.py= is the single source of truth for the title-pattern heuristics
— tune there rather than duplicating in =SKILL.md=. Title-only patterns
drop obvious user-bound TODOs ("Ask Pablo …", "Decide …") and scope-blown
verbs ("Translate /course/", "Refactor …", "Organize weekly meetings …")
straight to the blocker list without subagent cost. Action priority
combines the explicit org priority (1=highest … 9=lowest) with an
ease-of-processing score, and hard rails (=--max-tasks=, =--time-budget=,
externally-visible actions are always blockers with a draft as the next
step) cap each run. Cross-run state lives in
=~/.claude/overnight-todos-data/state.json= via =ledger.py= (per-machine
data dir shared with Claude, outside the dotfiles tree; reports under
=~/.claude/overnight-todos-data/runs/=). Each TODO's last verdict, ease,
attempt timestamp, and a SHA-256 hash of the heading body are recorded so
previously-blocked unchanged items are filtered out of subsequent runs
(skip window defaults to 14 days). Editing the heading body invalidates
the hash and re-queues the TODO automatically. The orchestrator runs
=org-roam-db-sync= before each dump and each COMPLETED subagent calls
=org-roam-db-update-file= on the touched file so just-completed TODOs do
not re-appear in the next run. Verification-only COMPLETED actions must
append a =Verified [YYYY-MM-DD]: <evidence>= line to the heading body
before marking DONE so the audit trail lives in-file. An append-only changelog at
=~/.claude/overnight-todos-data/history.md= captures every verdict across
runs, durable independently of the per-run reports. The output report
ranks remaining blockers by ease of unblock and includes a "still
blocked since last run" section so the user can clear the cheapest
gates first.

* Global Activation

=~/.codex/config.toml= points to =codex/config.toml=, and
=~/.codex/hooks.json= points to =codex/hooks.json=.  Updating the tracked files
therefore updates the live global Codex configuration.

Codex shell hooks are registered for both legacy =Bash= tool events and the
current =exec_command=/=functions.exec_command= tool names. The hook scripts
read both =command= and =cmd= payload fields so commit gates, security guards,
and Elisp verification markers work in CLI/API sessions as well as older
Bash-shaped payloads. Hook entries intentionally omit =statusMessage= because
Codex renders those progress labels in the TUI, and in persistent Emacs
terminal buffers they remain as scrollback after the final assistant answer.

The shell hook chain starts with =block-sensitive-read.sh=, which blocks
commands that would read or search known credential-bearing paths such as
=.mcp.json=, =.zshenv-secrets=, password stores, private-key directories,
OAuth token/client-secret JSON, generic credential JSON, and =.env= files.
It allowlists content-free metadata checks (=ls=, =stat=, =file=, =wc=),
quiet/count/name-only searches (=grep=/=rg= with =-q=, =-c=, =-l=, or =-L=),
existence tests, and auth-aware tools (=pass=, =op=, =git-crypt=,
=security=). This closes the gap where a command could contain no literal
secret but still print one by reading a credential config.

The =UserPromptSubmit= hook runs =civility-guard.sh= before Emacs lifecycle
dispatch. It blocks obvious abusive prompts directed at the assistant, asks for
a rewrite on the first violation, and locks the session on a second violation
until the reset phrase =I commit to a civil reset.= is submitted. The matcher is
empty because =UserPromptSubmit= has no tool name.

The =Stop= hook runs =completion-claim-guard.sh= before Emacs lifecycle
dispatch. It scans the latest final answer for words such as "fixed", "done",
or "resolved" and warns when the answer lacks an explicit verification receipt.
When it fires, it returns Codex's supported Stop continuation JSON
(=decision: block= with a =reason=) rather than =hookSpecificOutput=, because
Codex Stop hooks reject PostToolUse-style context output. Because =Stop= fires
after the final answer is emitted, this is an audit/reminder control rather
than a pre-final block.

The Elisp documentation gate accepts =doc/*.org= manuals, root =README.org=
manuals, and root =README.md= as a fallback only when the repo has no Org
manual. This keeps standalone packages such as =emacs-slack= covered without
treating Markdown READMEs as the preferred manual format for repos that already
have Org documentation.

=codex/config.toml= sets the default model, reasoning effort, fast service
tier, approval policy, sandbox mode, light TUI theme, and personality; enables Codex hooks,
plugins, the goals UI, terminal resize reflow, and local notice preferences
such as hiding the rate-limit model nudge; records trusted project roots; and
records model-availability and hook-trust state written by Codex.
When =agent= starts Codex with an account-specific =CODEX_HOME= such as
=~/.codex-epoch=, Codex may write account-home hook trust entries into this
tracked config. Those entries are kept here with the ordinary global
=~/.codex/hooks.json= trust entries so account-switched Codex sessions can run
the same trusted hook set without repeated prompts.
Trusted project roots are Codex-local allowlist entries for directories that can
run without extra trust prompts; they do not have a Claude settings counterpart.
The tracked allowlist includes the nested =dotfiles/emacs= project because it is
opened independently from the dotfiles root during Emacs configuration work, and
may include active Elpaca source clones that are edited as canonical package
checkouts, plus active work repos that should run without repeated trust
prompts.

=~/.codex/AGENTS.md= points to =codex/AGENTS.md= so Codex gets a tracked global
instruction file with Codex-safe path references. This is needed for user-level
instructions; project-level Codex instructions are handled by sibling
=AGENTS.md= files.

=~/.codex/skills= points directly to =codex/skills/= via
=codex/bin/install-skills.sh= because Codex does not load arbitrary skill paths
from =codex/config.toml=. =bin/skill-prune= disables stale skills natively with
=[[skills.config]]= entries in =codex/config.toml= instead of moving skill
directories. Project-local Codex skills are stored under the project's own
=.codex/skills/= directory. Codex-managed hidden system skills, if present, are
ignored runtime data under =codex/skills/.system/=.
