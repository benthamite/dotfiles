This manual describes the features and customization options for the Emacs Lisp file =mu4e-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=mu4e-extras.el= extends =mu4e=, the Emacs email client powered by the
=mu= indexer, with commands and configuration helpers tailored to a
Gmail-based workflow.  It depends on =el-patch= for patching internal
=mu4e= functions and on =mu4e= itself.

The package solves several practical problems that arise when using
=mu4e= with Gmail.  The personal account syncs via =mbsync= (IMAP),
while the Epoch AI account syncs via =gmail-maildir-sync=, a custom
tool that uses the Gmail API directly (necessary because the Epoch
Google Workspace admin has disabled IMAP).  Both sync backends run in
parallel during mail updates.  Gmail's IMAP behavior can reset the read
status of refiled and sent messages after synchronization.  This package
tracks such messages in a queue and re-applies the read flag after each
sync cycle, preventing the annoying appearance of already-read messages
as unread.

For daily email processing, the package provides one-step refile, trash,
and move commands that skip the confirmation prompt.  This removes
friction from the most common email actions in both headers and view
modes.

The package also handles multiple email contexts (personal vs. work,
HTML vs. plain text), sets up maildir shortcuts, integrates with Org
capture for processing actionable emails, and provides commands for
opening messages in Gmail's web interface.  A patch to
~mu4e--view-get-urls-num~ removes the unnecessary prompt when a message
contains only a single URL.

** Folder configuration
:PROPERTIES:
:CUSTOM_ID: h:folder-config
:END:

#+vindex: mu4e-extras-inbox-folder
#+vindex: mu4e-extras-daily-folder
#+vindex: mu4e-extras-newsletters-folder
#+vindex: mu4e-extras-epoch-inbox-folder
#+vindex: mu4e-extras-epoch-sent-folder
#+vindex: mu4e-extras-epoch-drafts-folder
#+vindex: mu4e-extras-epoch-refiled-folder
#+vindex: mu4e-extras-epoch-trash-folder
The user option ~mu4e-extras-inbox-folder~ specifies the maildir path
for the main inbox folder.  It is used by ~mu4e-extras-set-shortcuts~
and ~mu4e-extras-set-bookmarks~ to configure maildir shortcuts and
bookmarks ([[#h:setup-helpers][Setup helpers]]).  The default value is an empty string; you
must set this to match your maildir layout (e.g., ="/Inbox"=).

The user option ~mu4e-extras-daily-folder~ specifies the maildir path
for a "daily" folder, also used by ~mu4e-extras-set-shortcuts~.  The
default value is an empty string.

The user option ~mu4e-extras-newsletters-folder~ specifies the maildir
path for a newsletters folder.  The default value is an empty string.
This option is available for use in custom configurations but is not
referenced by any default function in the package.

The user option ~mu4e-extras-epoch-inbox-folder~ specifies the maildir
path for the Epoch AI inbox folder, used by
~mu4e-extras-set-shortcuts~ and ~mu4e-extras-set-bookmarks~.  The
default value is an empty string.

The user options ~mu4e-extras-epoch-sent-folder~,
~mu4e-extras-epoch-drafts-folder~, ~mu4e-extras-epoch-refiled-folder~,
and ~mu4e-extras-epoch-trash-folder~ specify the maildir paths for the
corresponding Epoch AI folders.  They are used by
~mu4e-extras-set-shortcuts~ to add uppercase-key maildir shortcuts for
Epoch folders.  The default value for each is an empty string.

** Reply behavior
:PROPERTIES:
:CUSTOM_ID: h:reply-behavior
:END:

#+vindex: mu4e-extras-wide-reply
The user option ~mu4e-extras-wide-reply~ controls the default behavior
when replying to messages with multiple recipients.  It accepts three
values:

- ~prompt~ (the default): ask the user whether to reply to all.
- ~t~: always reply to all recipients.
- ~nil~: always reply only to the sender.

This option is consulted by ~mu4e-extras-compose-reply~
([[#h:composing-messages][Composing messages]]).  When the original message has only a single
recipient (excluding the sender), the option has no effect and a normal
reply is always used.

** Reaction emojis
:PROPERTIES:
:CUSTOM_ID: h:reaction-emojis
:END:

#+vindex: mu4e-extras-reaction-emojis
The user option ~mu4e-extras-reaction-emojis~ specifies the list of
emoji choices presented when reacting to a message with
~mu4e-extras-react-to-message~ ([[#h:gmail-reactions][Gmail reactions]]).  The default value
includes common emojis such as thumbs up, heart, and check mark.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Refiling, trashing, and moving without confirmation
:PROPERTIES:
:CUSTOM_ID: h:no-confirm-actions
:END:

#+findex: mu4e-extras-headers-refile
#+findex: mu4e-extras-view-refile
#+findex: mu4e-extras-headers-trash
#+findex: mu4e-extras-view-trash
#+findex: mu4e-extras-headers-move
#+findex: mu4e-extras-view-move
#+findex: mu4e-extras-mark-execute-all-no-confirm
The standard =mu4e= mark-and-execute workflow requires an explicit
confirmation step.  The commands in this section bypass that
confirmation, performing the action immediately on the message at point.

The commands ~mu4e-extras-headers-refile~ and ~mu4e-extras-view-refile~
refile the message at point in headers mode and view mode respectively.
Similarly, ~mu4e-extras-headers-trash~ and ~mu4e-extras-view-trash~
trash the message, while ~mu4e-extras-headers-move~ and
~mu4e-extras-view-move~ prompt for a target folder and move the message
immediately.

The command ~mu4e-extras-mark-execute-all-no-confirm~ executes all
pending marks in the current buffer without asking for confirmation.

** Marking as read and refiling
:PROPERTIES:
:CUSTOM_ID: h:mark-read-refile
:END:

#+findex: mu4e-extras-headers-mark-read-and-refile
The command ~mu4e-extras-headers-mark-read-and-refile~ combines two
actions: it marks the message at point as read and refiles it.  The
message's ID is added to an internal queue so that its read status can
be re-applied after the next Gmail synchronization ([[#h:read-status-management][Read status management]]).

This command exists because Gmail's IMAP implementation sometimes resets
the read flag on refiled messages during synchronization.  By queuing
the message, the package ensures the flag is restored automatically.

** Initializing and updating mail
:PROPERTIES:
:CUSTOM_ID: h:init-update
:END:

#+findex: mu4e-extras-init
#+findex: mu4e-extras-update-all-mail-and-index
#+findex: mu4e-extras-check-all-mail
#+findex: mu4e-extras-reindex-db
The command ~mu4e-extras-init~ reinitializes the =mu= database.  It
stops any running mail update, quits =mu4e=, and then runs the =muinit=
script asynchronously.  Use this when the database is corrupted or after
major configuration changes.

The command ~mu4e-extras-update-all-mail-and-index~ fetches all mail
including archives from all accounts, then indexes with =mu=.  It runs
=mbsync gmail-all= for the personal account and
=gmail-maildir-sync pull --include-all= for the Epoch account in
parallel (the Epoch account uses the Gmail API instead of IMAP, since
the Epoch Google Workspace admin has disabled IMAP).  With a prefix
argument, it runs in the background; otherwise, it shows a progress
window.

The command ~mu4e-extras-check-all-mail~ performs the same operation as
~mu4e-extras-update-all-mail-and-index~ but always runs in the
background.  It is intended for periodic full syncs that can take longer.

The command ~mu4e-extras-reindex-db~ kills any running =mu= process and
then runs =mu index= to rebuild the index.

** Composing messages
:PROPERTIES:
:CUSTOM_ID: h:composing-messages
:END:

#+findex: mu4e-extras-compose-reply
#+findex: mu4e-extras-compose-new-externally
The command ~mu4e-extras-compose-reply~ initiates a reply to the message
at point.  If the sender is the current user, it calls
~mu4e-compose-supersede~ to edit the sent message.  When there are
multiple recipients, the behavior depends on the ~mu4e-extras-wide-reply~
user option ([[#h:reply-behavior][Reply behavior]]).

The command ~mu4e-extras-compose-new-externally~ opens Gmail's compose
screen in the default web browser, bypassing =mu4e= entirely.

** Gmail reactions
:PROPERTIES:
:CUSTOM_ID: h:gmail-reactions
:END:

#+findex: mu4e-extras-react-to-message
The command ~mu4e-extras-react-to-message~ reacts to the email at point
with a Gmail emoji reaction.  It prompts the user to select an emoji
from ~mu4e-extras-reaction-emojis~ ([[#h:reaction-emojis][Reaction emojis]]), then constructs a
MIME message per Google's email reaction format specification (using a
=text/vnd.google.email-reaction+json= part) and sends it to the
original sender.

The command automatically determines the sending account by checking
which of the user's email addresses appears in the message's To or Cc
fields.  Epoch messages are sent via the Gmail API (through
=gmail-maildir-sync send=); personal and Tlon messages are sent via
SMTP.

** Viewing messages in Gmail
:PROPERTIES:
:CUSTOM_ID: h:gmail-commands
:END:

#+findex: mu4e-extras-open-gmail
#+findex: mu4e-extras-view-in-gmail
#+findex: mu4e-extras-gmail-base
#+findex: mu4e-extras-gmail-message-id
#+findex: mu4e-extras-gmail-thread-id
#+vindex: mu4e-extras-gmail-sync-state-db
The function ~mu4e-extras-gmail-base~ returns the base Gmail URL for the
account that a message belongs to.  It uses the email address in the URL
path (e.g. =https://mail.google.com/mail/u/pablo@epoch.ai/=) so that
Gmail automatically switches to the correct account.  When the message
belongs to the Epoch account (detected via its maildir prefix), it uses
~EPOCH_EMAIL~; otherwise it falls back to ~PERSONAL_GMAIL~.

The command ~mu4e-extras-open-gmail~ opens the Gmail inbox in the
default web browser.

The command ~mu4e-extras-view-in-gmail~ opens the message at point in
Gmail.  For messages synced via ~gmail-maildir-sync~ (Epoch account), it
reads the =X-Gmail-Message-Id= header from the message file, looks up
the corresponding thread ID from the ~gmail-maildir-sync~ SQLite state
database (~mu4e-extras-gmail-sync-state-db~), and opens the thread
directly using a ~#all/<thread-id>~ URL.  For other messages (personal
account), it falls back to searching by RFC 822 message ID.  This is
useful when you need to perform an action that =mu4e= does not support,
such as viewing the full HTML rendering or accessing Gmail-specific
features.

** Message file operations
:PROPERTIES:
:CUSTOM_ID: h:message-files
:END:

#+findex: mu4e-extras-copy-message-path
#+findex: mu4e-extras-open-message-file
The command ~mu4e-extras-copy-message-path~ copies the filesystem path
of the current message to the kill ring.  This is useful when you need
to pass the path to an external tool or paste it into a shell command.
It uses ~mu4e-extras-get-message-file~ internally
([[#h:message-file-access][Message file access]]).

The command ~mu4e-extras-open-message-file~ opens the file on disk that
stores the current message.  It uses ~dired-jump~ to display the file
in its parent directory in Dired mode.  This is useful for inspecting
raw message files or managing mail storage.

** Org capture integration
:PROPERTIES:
:CUSTOM_ID: h:org-capture
:END:

#+findex: mu4e-extras-view-org-capture
The command ~mu4e-extras-view-org-capture~ captures the message at point
using =org-capture=.  By default it uses the template key ="e"=, but if
the message body contains the string =[org-capture key: KEY]=, it uses
KEY instead.  This allows automated email workflows to specify which
capture template should process the message.

Without a prefix argument, the message is refiled after capture (via
~mu4e-extras-view-refile~).  With a prefix argument, the message is
captured but not refiled, leaving it in the inbox.

This command works in both ~mu4e-view-mode~ and ~mu4e-headers-mode~.

** Copying information
:PROPERTIES:
:CUSTOM_ID: h:copying-info
:END:

#+findex: mu4e-extras-copy-number-in-title
The command ~mu4e-extras-copy-number-in-title~ extracts a numeric value
from the subject line of the message at point and copies it to the kill
ring.  This is useful for quickly capturing amounts from transactional
emails (e.g., invoices, receipts) without opening the message.

* Functions
:PROPERTIES:
:CUSTOM_ID: h:functions

** Read status management
:PROPERTIES:
:CUSTOM_ID: h:read-status-management
:END:

#+findex: mu4e-extras-reapply-read-status
#+findex: mu4e-extras-reapply-read-status-set-timer
#+findex: mu4e-extras-add-refiled-to-mark-as-read-queue
#+findex: mu4e-extras-add-sent-to-mark-as-read-queue
#+vindex: mu4e-extras-mark-as-read-queue
The variable ~mu4e-extras-mark-as-read-queue~ holds a list of message
IDs that need to be re-marked as read after the next Gmail
synchronization.  Messages are added to this queue by two functions:

The function ~mu4e-extras-add-refiled-to-mark-as-read-queue~ adds the
current message to the queue, refiles it, and executes all marks.  It is
called by ~mu4e-extras-headers-mark-read-and-refile~
([[#h:mark-read-refile][Marking as read and refiling]]).

The function ~mu4e-extras-add-sent-to-mark-as-read-queue~ adds the sent
message's ID to the queue.  It is intended for ~message-sent-hook~.
Gmail automatically saves a copy of sent messages, and the synced-back
copy appears as unread; this function ensures it is marked as read.

The function ~mu4e-extras-reapply-read-status~ iterates through the
queue and marks each message as seen using ~mu4e--server-move~ with the
=+S-u= flags.  It then clears the queue.

The function ~mu4e-extras-reapply-read-status-set-timer~ sets a
30-second timer to call ~mu4e-extras-reapply-read-status~ if the queue
is not empty.  This delay allows the synchronization to complete before
re-applying the read status.

** Gmail flag correction
:PROPERTIES:
:CUSTOM_ID: h:gmail-flags
:END:

#+findex: mu4e-extras-gmail-fix-flags
The function ~mu4e-extras-gmail-fix-flags~ translates standard =mu4e=
marks into Gmail-specific label changes.  For each mark and message
pair, it applies the appropriate tag operations: removing =\Inbox= and
adding =\Trash= for trash, removing =\Inbox= and adding =\Refiled= for
refile, and toggling =\Starred= for flag/unflag.  Epoch messages are
skipped, since they are synced via the Gmail API directly (through
=gmail-maildir-sync=) rather than via =mbsync=, so IMAP flag
manipulation does not apply to them.

** Setup helpers
:PROPERTIES:
:CUSTOM_ID: h:setup-helpers
:END:

#+findex: mu4e-extras-set-shortcuts
#+findex: mu4e-extras-set-bookmarks
#+findex: mu4e-extras-set-face-locally
#+findex: mu4e-extras-send-via-gmail-api
#+findex: mu4e-extras-set-contexts
#+findex: mu4e-extras-set-index-params
The function ~mu4e-extras-set-shortcuts~ adds predefined entries to
~mu4e-maildir-shortcuts~.  Personal folders use lowercase keys (~i~ for
inbox, ~y~ for daily) and Epoch folders use uppercase keys (~I~ for
inbox, ~T~ for sent, ~D~ for drafts, ~R~ for refiled, ~X~ for trash).

The function ~mu4e-extras-set-bookmarks~ adds an "All inboxes" bookmark
(key ~a~) to ~mu4e-bookmarks~ that queries both the personal and Epoch
inbox folders, providing a combined view of all incoming mail.

The function ~mu4e-extras-set-face-locally~ remaps the ~shr-text~ face
to a smaller height (0.9) in ~mu4e-view-mode~ buffers.  This is
intended for ~mu4e-view-mode-hook~ to improve the readability of HTML
emails rendered with =shr=.

The function ~mu4e-extras-send-via-gmail-api~ sends the current message
by piping it to =gmail-maildir-sync send=, which uses the Gmail API
directly.  Unlike ~message-send-mail-with-sendmail~, it does not inject
sendmail-specific flags (=-oi=, =-f=, =-t=) that =gmail-maildir-sync=
does not accept.

The function ~mu4e-extras-set-contexts~ defines and sets
~mu4e-contexts~ for six combinations: personal HTML, personal plain
text, Epoch HTML, Epoch plain text, Tlon HTML, and Tlon plain text.
Each context uses environment variables (=PERSONAL_GMAIL=,
=EPOCH_EMAIL=, =WORK_EMAIL=) for email addresses and references Org Msg
signature variables from =org-msg-extras.el=.  The Epoch contexts also
set account-specific folder paths (sent, drafts) and use the custom
function ~mu4e-extras-send-via-gmail-api~ to send messages via
=gmail-maildir-sync send= instead of SMTP, since the Epoch Google
Workspace admin has disabled IMAP/SMTP.  The plain text contexts
disable ~org-msg-mode~ on entry and re-enable it on leave.

#+findex: mu4e-extras-set-account-folders
The function ~mu4e-extras-set-account-folders~ sets ~mu4e-refile-folder~
and ~mu4e-trash-folder~ to functions that route messages to the correct
account-specific folder based on the message's maildir path.

The function ~mu4e-extras-set-index-params~ adjusts indexing parameters
(~mu4e-index-lazy-check~ and ~mu4e-index-cleanup~) based on Emacs idle
time.  When the user has been idle for longer than
~mu4e-update-interval~, it assumes the user is away and performs a
thorough index (lazy check off, cleanup on).  When idle time is short,
it performs a fast index (lazy check on, cleanup off).

** Archiving refiled messages
:PROPERTIES:
:CUSTOM_ID: h:archiving-refiled
:END:

#+findex: mu4e-extras-archive-refiled
The function ~mu4e-extras-archive-refiled~ archives messages that have
the "refile" label by running a full mail sync with =mbsync gmail-all=
and performing a thorough index.  This is intended for periodic cleanup
to keep the archive up to date.

** Context matching
:PROPERTIES:
:CUSTOM_ID: h:context-matching
:END:

#+findex: mu4e-extras-msg-is-personal-p
#+findex: mu4e-extras-msg-is-epoch-p
#+findex: mu4e-extras-msg-is-tlon-p
#+findex: mu4e-extras-msg-belongs-to-epoch-p
#+findex: mu4e-extras-msg-is-html-p
#+findex: mu4e-extras-refile-folder
#+findex: mu4e-extras-trash-folder
The function ~mu4e-extras-msg-is-personal-p~ returns non-~nil~ if the
message's =:to= field matches either the personal Gmail address or the
personal email address (from environment variables).

The function ~mu4e-extras-msg-is-epoch-p~ returns non-~nil~ if the
message's =:to= field matches the Epoch AI email address.

The function ~mu4e-extras-msg-is-tlon-p~ returns non-~nil~ if the
message's =:to= field matches the Tlon work email address or if the
=:reply-to= field matches the team's Google Group.

The function ~mu4e-extras-msg-belongs-to-epoch-p~ returns non-~nil~ if
the message's maildir path starts with =/epoch/=.  This is used by the
folder routing functions below.

The function ~mu4e-extras-refile-folder~ returns =/epoch/Refiled= for
Epoch messages and =/Refiled= for all others.

The function ~mu4e-extras-trash-folder~ returns =/epoch/Trash= for
Epoch messages and =/Trash= for all others.

The function ~mu4e-extras-msg-is-html-p~ returns non-~nil~ if the
message contains a =text/html= MIME part, as determined by parsing the
raw message file.

** Reaction helpers
:PROPERTIES:
:CUSTOM_ID: h:reaction-helpers
:END:

#+findex: mu4e-extras-reaction-from-address
#+findex: mu4e-extras-reaction-to-email
#+findex: mu4e-extras-reaction-build-mime
#+findex: mu4e-extras-reaction-send
The function ~mu4e-extras-reaction-from-address~ determines the
sending account for a reaction by checking which of the user's email
addresses (Epoch, Tlon, or personal) appears in the message's To or
Cc fields.

The function ~mu4e-extras-reaction-to-email~ extracts the bare email
address of the original sender.

The function ~mu4e-extras-reaction-build-mime~ constructs a
=multipart/alternative= MIME message containing three parts: a plain
text fallback, a =text/vnd.google.email-reaction+json= part with the
emoji and format version, and an HTML fallback.

The function ~mu4e-extras-reaction-send~ dispatches the reaction
message to the appropriate transport: the Gmail API (via
=gmail-maildir-sync send=) for Epoch messages, and SMTP for all
others.

** Message file access
:PROPERTIES:
:CUSTOM_ID: h:message-file-access
:END:

#+findex: mu4e-extras-get-message-file
The function ~mu4e-extras-get-message-file~ returns the filesystem path
of the message at point.  It checks ~mu4e-compose-parent-message~ first
(for use during composition) and falls back to ~mu4e-message-at-point~.

** Single-URL auto-selection
:PROPERTIES:
:CUSTOM_ID: h:single-url-patch
:END:

#+findex: mu4e--view-get-urls-num
The function ~mu4e--view-get-urls-num~ is patched using =el-patch= to
automatically select the URL when a message contains exactly one link.
The standard behavior would prompt the user to enter the number "1",
which is unnecessary.  When there are multiple URLs, the original
prompting behavior is preserved.
