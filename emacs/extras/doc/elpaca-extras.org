This manual describes the features and customization options for the Emacs Lisp file =elpaca-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=elpaca-extras.el= extends the =elpaca= package manager with commands that make it easier to iterate on packages without restarting Emacs. When you update or rebuild a package, its old definitions remain loaded in the running session. These extensions automate the tedious process of unloading stale features and reloading the updated versions, and they also provide filtered lock-file writing.

The main capabilities are:

- *Reload after update.* A command that updates a package and automatically reloads its features once the build finishes ([[#h:update-and-reload][Update and reload]]).
- *Reload after rebuild.* The same workflow for package rebuilds ([[#h:rebuild-and-reload][Rebuild and reload]]).
- *Manual reload.* A standalone reload command for when you have made local edits to a package's source ([[#h:reload][Reload]]).
- *Filtered lock files.* Writing Elpaca lock files while excluding specific packages ([[#h:lock-file][Lock file management]]).

* User options
:PROPERTIES:
:CUSTOM_ID: h:user-options
:END:

** ~elpaca-extras-write-lock-file-excluded~
:PROPERTIES:
:CUSTOM_ID: h:elpaca-extras-write-lock-file-excluded
:END:

#+vindex: elpaca-extras-write-lock-file-excluded
The user option ~elpaca-extras-write-lock-file-excluded~ specifies a list of package identifiers (symbols) that should be omitted when writing a lock file with ~elpaca-extras-write-lock-file-excluding~ ([[#h:lock-file][Lock file management]]). This is useful for packages that are installed from local sources or that you do not want pinned to a specific revision.

The default value is ~nil~ (no exclusions). The type is a list of symbols.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Reload
:PROPERTIES:
:CUSTOM_ID: h:reload
:END:

#+findex: elpaca-extras-reload
After making local edits to an installed package or after a manual rebuild, the old feature definitions remain active in the running Emacs session. The command ~elpaca-extras-reload~ unloads and reloads the features provided by a specified package.

It works by scanning the package's installation directory for =.el= files, extracting the feature names from their ~provide~ forms, forcefully unloading those features, and then requiring them again. By default, only features that were already loaded are reloaded. With a prefix argument, all features provided by the package are loaded regardless of their previous state.

*Important:* because ~unload-feature~ is called with the ~force~ flag, any data stored in the package's symbols will be lost. If the package saves state on deactivation or Emacs exit, you should trigger that save before reloading.

** Update and reload
:PROPERTIES:
:CUSTOM_ID: h:update-and-reload
:END:

#+findex: elpaca-extras-update-and-reload
The command ~elpaca-extras-update-and-reload~ combines updating a package via Elpaca with an automatic reload. It prompts for a queued package, triggers ~elpaca-update~, and installs a temporary hook on ~elpaca-post-queue-hook~ to detect when the build finishes. If the build succeeds, it calls ~elpaca-extras-reload~ ([[#h:reload][Reload]]) and reports the result. If the build fails, it directs you to the =*elpaca-log*= buffer for details.

This provides a convenient single-step workflow: update, build, unload old definitions, and load the new ones.

** Rebuild and reload
:PROPERTIES:
:CUSTOM_ID: h:rebuild-and-reload
:END:

#+findex: elpaca-extras-rebuild-and-reload
The command ~elpaca-extras-rebuild-and-reload~ is analogous to ~elpaca-extras-update-and-reload~ ([[#h:update-and-reload][Update and reload]]) but uses ~elpaca-rebuild~ instead of ~elpaca-update~. This is useful when you have made local source changes and want to rebuild the package from those changes rather than fetching upstream updates.

** Lock file management
:PROPERTIES:
:CUSTOM_ID: h:lock-file
:END:

#+findex: elpaca-extras-write-lock-file-excluding
The command ~elpaca-extras-write-lock-file-excluding~ writes an Elpaca lock file to the specified PATH while filtering out packages listed in ~elpaca-extras-write-lock-file-excluded~ ([[#h:elpaca-extras-write-lock-file-excluded][User options]]). When called interactively, it prompts for the output file path.

The optional ELPACAS argument accepts a queue-like list as used by ~elpaca-write-lock-file~. When nil, the current queue is used. This makes it possible to call the function programmatically with a custom package set.
