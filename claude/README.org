*** Directory topology
:PROPERTIES:
:CUSTOM_ID: h:directory-topology
:ID:       EB7422C8-E371-4F9B-AE1F-D6B6473F11BB
:END:

Three layers of directories work together: this dotfiles directory (source of truth), the shared =~/.claude/= hub, and per-account overlay directories.

*Source of truth* (version-controlled in this repo):

#+begin_example
~/My Drive/dotfiles/claude/           ← global config source of truth
├── CLAUDE.md                         ← global conventions for all projects
├── settings.json                     ← hooks, model, permissions, theme
├── settings.local.json               ← machine-specific permissions (mpv, brew, osascript…)
├── hooks/                            ← 12 hook scripts
├── skills/                           ← 44 skill directories
├── bin/batch-test.sh                 ← elpaca-aware emacs --batch test runner
├── bin/sync-mcp-servers.sh           ← MCP server sync utility
└── README.org                        ← this file
#+end_example

*Shared hub* (=~/.claude/=), mostly symlinks into the above:

#+begin_example
~/.claude/
├── CLAUDE.md          → dotfiles/claude/CLAUDE.md
├── settings.json      → dotfiles/claude/settings.json
├── settings.local.json → dotfiles/claude/settings.local.json
├── skills/            → dotfiles/claude/skills/
├── projects/          ← per-project memory, trust entries (managed by Claude Code)
├── memory/            ← global memory (managed by Claude Code)
├── history.jsonl      ← command history (managed by Claude Code)
└── plugins/           ← installed plugins (managed by Claude Code)
#+end_example

*Per-account overlays* (=~/.claude-personal/=, =~/.claude-work/=): see [[*Multi-account setup][multi-account setup]] below.

*Per-project config* (=<project>/.claude/=):

#+begin_example
<project>/.claude/
├── settings.json       ← project-level hooks (optional)
├── settings.local.json ← project-specific permissions (accumulated interactively)
├── memory/             ← project memory (moved here by symlink-project-memory.sh hook)
└── skills/             ← project-specific skills (optional)
#+end_example

*MCP servers* are defined in =~/.claude.json= (canonical) and synced to account directories by =claude/bin/sync-mcp-servers.sh=.

*** Settings hierarchy
:PROPERTIES:
:CUSTOM_ID: h:settings-hierarchy
:ID:       ABB0C5CA-1B40-47BF-8746-BFFE1F4E420D
:END:

Claude Code loads settings from multiple levels, merged together. Each level can define permissions, hooks, and other options. Permissions are additive across levels. Hooks run from all levels that define them.

| Level | =settings.json= | =settings.local.json= | Purpose |
|-------+------------------+-----------------------+---------|
| User (=~/.claude/=) | Model, theme, global hooks, global permissions | Machine-specific tool whitelists | Applies to all projects |
| Project (=<project>/.claude/=) | Project-specific hooks | Project-specific permissions | Applies only in that project |

=settings.json= is version-controlled; =settings.local.json= is gitignored (machine-specific or accumulated interactively).

The global =settings.json= uses =bypassPermissions= mode: all tools run without approval prompts. The confirmation prompt is suppressed via =skipDangerousModePermissionPrompt=.

*** =CLAUDE.md= loading
:PROPERTIES:
:CUSTOM_ID: h:claude-md-loading
:ID:       D03891CF-C905-480A-9690-62B4F8A7273D
:END:

Claude Code loads =CLAUDE.md= files from several locations. When working in a project, it loads:

1. =~/.claude/CLAUDE.md= — user-level global conventions (symlinked to =dotfiles/claude/CLAUDE.md=)
2. =<project>/CLAUDE.md= — project-level instructions (if present)

Each file should contain distinct content. The global one defines behavioral conventions (verify work, fix root causes, commit immediately, etc.). Project-level ones define project-specific context.

** Multi-account setup
:PROPERTIES:
:CUSTOM_ID: h:multi-account
:ID:       239E0918-2383-460C-8C9C-2ABC4E3D344D
:END:

Two OAuth accounts (personal and work) are supported without re-authenticating. Each account uses a separate =CLAUDE_CONFIG_DIR= (=~/.claude-personal/= and =~/.claude-work/=) to keep its OAuth credentials and identity isolated, while sharing everything else via symlinks back to =~/.claude/=.

Shared (symlinked into each account dir):
- =settings.json=, =settings.local.json= — preferences and permissions
- =skills/= — custom skills
- =memory/= — persistent memory
- =projects/= — trusted folders, project config, session logs
- =history.jsonl= — command history
- =plugins/= — installed plugins

Not symlinked:
- =CLAUDE.md= — loaded from =~/.claude/CLAUDE.md= at user level; account dirs do not have their own copy (to avoid loading the same instructions twice)

Separate per account:
- =.claude.json= — OAuth identity, user ID, and MCP servers (synced from =~/.claude.json=)
- Keychain credentials — the actual OAuth tokens

The active account is stored in =~/.claude-current-account=. From Emacs, =claude-code-extras-start-or-switch= uses the persisted account automatically (configured via =claude-code-extras-accounts=). Use =claude-code-extras-select-account= to change it. From the terminal, the shell aliases =claude-personal= and =claude-work= are available.

*MCP servers:* Claude Code reads =.claude.json= from =$CLAUDE_CONFIG_DIR= (not =$HOME=) when that variable is set. The canonical source of truth is =~/.claude.json=; run =claude/bin/sync-mcp-servers.sh= after adding or removing any MCP server there.

*** Chrome integration and multi-account

The Claude in Chrome extension only supports *one account at a time*. The account logged into the extension must match the account of the Claude Code session trying to use it. If they don't match, Chrome tools return "No Chrome extension connected" (or "Browser extension is not connected").

*Current state:* three Claude Code accounts (personal, work, tlon) but only one Chrome extension login. Whichever account is logged into the extension is the only one that can use =/chrome=.

*To switch the Chrome extension account:*
1. Open the extension settings (click the Claude extension icon in Chrome toolbar)
2. Click "Log out"
3. Log in with the desired account (e.g. =tlon.shared@gmail.com=)
4. *Restart Chrome* — the native messaging bridge caches the old session and won't pick up the new account without a restart

*Troubleshooting "No Chrome extension connected":*
1. Check which account the extension is logged into (extension settings, top right)
2. Check which account the Claude Code session is using (=echo $CLAUDE_CONFIG_DIR=, then check =.claude.json= → =oauthAccount.emailAddress=)
3. If they don't match, switch the extension account (see above)
4. If they match but it still fails: restart Chrome, then start a *new* Claude Code session (resumed sessions preserve old MCP config)

*Emacs-specific:* non-interactive sessions (Emacs) require the =--chrome= flag, set via =(setq claude-code-program-switches '("--chrome"))=. Without it, Claude Code silently skips Chrome MCP startup.

*Architecture:*
- Chrome extension (ID =fcoeoabgfenejglbffodgkkbkcdhcgfn=) communicates via native messaging
- Native host manifest: =~/Library/Application Support/Google/Chrome/NativeMessagingHosts/com.anthropic.claude_code_browser_extension.json=
- Native host script: =~/.claude-tlon/chrome/chrome-native-host= (points to the installed =claude= binary with =--chrome-native-host=)
- Bridge socket: =/tmp/claude-mcp-browser-bridge-pablostafforini/<pid>.sock=
- Keychain credentials per account: =Claude Code-credentials-<sha256(config_dir)[:8]>=

** CLAUDE.md
:PROPERTIES:
:ID:       9B404E8C-F109-4188-9F3E-1F1A9A64A37D
:END:

Project-level instructions referencing the =dotfiles-context= skill. The global =~/.claude/CLAUDE.md= (symlinked here) defines conventions that apply across all projects:

- Fix root causes, not symptoms; no silent fallbacks. Prefer robust solutions over hacks, even if they take longer.
- Verify fixes before declaring them done.
- Commit all changes immediately; prefer amending when iterating on the same logical change (unless already pushed).
- Pre-commit checklist (hook-enforced): batch-test Elisp before committing, stage =doc/*.org= with =.el= changes, stage =README.org= with =claude/= changes.
- Use subagents and teams liberally; prefer the most powerful model.
- Use =trash= instead of =rm -rf=; never send signals to Emacs without confirmation.
- Never take actions visible to others (push, PR, post) without explicit confirmation.

*** twitter-digest
:PROPERTIES:
:ID:       F677671C-6C97-4BC6-B2F5-A4E0E2457C92
:END:

Fetches recent tweets from a named list of accounts, triages them against the list's description (used as a relevance rubric), and surfaces only notable content. Uses a shell script (=fetch-tweets.sh=) that curls the twitterapi.io REST API directly and extracts only the fields needed for triage (~130 chars/tweet), bypassing MCP tools whose raw JSON responses (~120K chars/account) would consume enormous context. Lists are stored in =~/.claude/skills/twitter-digest/lists/= and track the last-run timestamp to avoid re-surfacing old tweets. =fetch-tweets.sh= validates list names against path traversal before constructing file paths. Digests are saved to =digests/= for archival, with session metadata (cost, duration, session ID) in the org frontmatter. Discovered accounts (surfaced via retweets) are checked against the vet registry and excluded if previously rejected or below threshold; remaining accounts are vetted using the shared =twitter-vet= procedure (including the epistemic quality gate) and auto-added to the list if they score >= 7. Commits all changed files (digests, last-run timestamps, list updates, vet registry) at the end. Pairs with =/loop= for periodic digests.

*** twitter-vet
:PROPERTIES:
:ID:       A7B3E1D4-5F92-4C8A-B6E0-9D2F4A1C7E53
:END:

Shared account vetting procedure used by =twitter-digest= and =twitter-discover=. Defines a two-phase evaluation: a free quick filter (follower bounds, bio relevance, tweet quality) and a full scoring pass (fetch 20 tweets, score 1-10 against the list's rubric). Includes a universal *epistemic quality gate* that penalizes engagement-optimized framing, overclaiming, and hype, and rewards calibrated confidence, caveats, and uncertainty acknowledgment. Accounts failing the epistemic gate are penalized 2-3 points regardless of topical fit. Accounts scoring above the threshold (default 7, overridable via =vet-threshold= frontmatter) are auto-added to the list; borderline scores are reported; low scores are skipped silently. Commits changed files (list updates, vet registry) when run standalone; defers to the caller's commit step when invoked from =twitter-digest=.

*** twitter-discover

* Hooks
:PROPERTIES:
:ID:       4D862953-7E8A-4154-BC16-919ED6B092A5
:END:

Ten hook scripts handle workflow invariants and Emacs integration. Most are defined in the global =settings.json= and use early-exit guards to skip inapplicable projects. One (=require-readme-update.sh=) is defined in the dotfiles project-level =.claude/settings.json= since it only applies to this repo. The emacsclient hooks (=notify-emacs-session-end.sh=, =notify-emacs-notification.sh=, =load-elisp-after-edit.sh=) escape interpolated values before embedding them in Elisp strings to prevent injection.

** require-elisp-test-before-commit.sh (PreToolUse)
:PROPERTIES:
:ID:       4E0CDC5D-C436-4701-AC99-E29F950318B4
:END:

Blocks =git commit= when Elisp files (=*.el= or =emacs/config.org=) are staged but no test has been run in the current session. Forces the agent to verify changes before committing. The error message suggests a ready-to-paste =batch-test.sh= command with the detected package name. Also detects the case where =git add= and =git commit= appear in the same bash command (which would otherwise bypass the staged-file check by running before staging occurs).

** track-elisp-test.sh (PostToolUse)
:PROPERTIES:
:ID:       85C6B926-9F3F-431A-9650-F504CEE12B4A
:END:

Creates a marker file (=/tmp/claude-elisp-tested-{SESSION_ID}=) whenever =emacs --batch= runs. This marker is what the pre-commit hook checks for. The marker is consumed (deleted) after a successful commit, so each commit requires a fresh test. Also verifies that the test loaded files from the working tree, not stale =.elc= from =elpaca/builds/=: if the command output contains evidence of a stale load (e.g. "using older file" or =elpaca/builds/*.elc=), the marker is NOT created and a warning is printed, so the commit hook will still block.

** symlink-project-memory.sh (PostToolUse)
:PROPERTIES:
:ID:       086B58F4-949F-4C88-B855-687EAD5F3B63
:END:

On the first Edit or Write in a project, moves Claude Code's memory directory from its default location (=~/.claude/projects/<encoded-path>/memory/=) into the project's =.claude/memory/= directory, replacing the original with a symlink. This keeps memory local and version-controllable.

** require-readme-update.sh (PreToolUse, project-level)
:PROPERTIES:
:ID:       F4D78958-4194-4DB3-9624-C04EC7790960
:END:

Defined in the dotfiles project-level =.claude/settings.json= (not the global settings) since it only applies to this repo. Blocks =git commit= when files in =claude/= (skills, hooks, settings, CLAUDE.md) are staged but =claude/README.org= is not. Also detects when =git add= and =git commit= appear in the same bash command (which would otherwise bypass the staged-file check). Ensures documentation stays in sync with configuration changes. Project-specific skills (under =.claude/skills/=) are excluded from this check, since they belong to their project's documentation, not the dotfiles README. Runtime data paths (=digests/=, =last-run/=, =twitter-vet/vetted/=) are also excluded, since these are skill output files that don't warrant a README update.

** remind-claude-readme-update.sh (PostToolUse)
:PROPERTIES:
:ID:       2D9A0367-26C4-40B8-9B9C-06346D89238B
:END:

Companion to =require-readme-update.sh=. Fires globally after any Write or Edit, resolves the target path through symlinks, and outputs a reminder if the file is under =dotfiles/claude/= but the agent is working in a different project. This closes the gap where the commit-blocking hook never fires because the agent never switches to the dotfiles repo to commit. Silent when editing =README.org= itself or when already in the dotfiles repo.

** require-doc-update.sh (PreToolUse)
:PROPERTIES:
:CUSTOM_ID: h:require-doc-update
:ID:       BB6FE0BD-FC74-4BEB-89AB-4FE902A84055
:END:

Blocks =git commit= when =.el= files are staged but no =doc/*.org= or =README.org= manual update is included. Applies only in repos that have a =doc/= directory or a =README.org= at the root. Ensures documentation stays in sync with code changes.

** load-elisp-after-edit.sh (PostToolUse)
:PROPERTIES:
:CUSTOM_ID: h:load-elisp-after-edit
:ID:       F07EF468-2770-42DE-B282-9BCB6E624EF5
:END:

Fires on every Edit or Write of an =.el= file under =elpaca/sources/= or =dotfiles/emacs/extras/=. Identifies the elpaca package (by matching the file's basename against package names for extras, or by source directory prefix for standalone packages), then runs =elpaca-rebuild= + =elpaca-wait= + =elpaca-extras-reload= synchronously via =emacsclient=. This byte-compiles the source into =elpaca/builds/=, waits for completion, and reloads all features. Skips byte-compiled files and test files. Note: for dotfiles extras edits, the rebuild may compile from a stale elpaca source if the edit hasn't been committed yet; the =post-commit= hook triggers a second rebuild from the correct source (see =emacs/extras/doc/elisp-development-workflow.org=).

** elpaca-run.sh (utility)
:PROPERTIES:
:ID:       54F62C33-E882-4B75-A4F0-A8963E5AF345
:END:

Resolves the active elpaca profile dynamically (via =emacsclient -e 'init-current-profile'=) and runs a script from the specified elpaca package. Searches both =repos/= and =sources/= directories to handle different elpaca layouts. Used by the Notification and Stop hooks to invoke =claude-code-hook-wrapper= without hardcoding the profile name. The =emacsclient= call has a 5-second timeout to prevent hangs when the Emacs server is unresponsive.

** notify-emacs-session-end.sh (Stop)
:PROPERTIES:
:ID:       8A2F3E91-4D5C-4B8F-9E3A-7C6D2A1F0B94
:END:

Forwards the session-end event to Emacs via =claude-code-handle-hook=. Runs through =fire-and-forget.sh= so the CLI doesn't block on =emacsclient=. Works together with =agent-log-sync-on-session-end= to trigger log syncing when a session ends.

** claude-code-hook-wrapper (Notification, Stop)
:PROPERTIES:
:ID:       44E20BF0-9B6B-420E-858F-98CBF18D13F8
:END:

Invoked via =elpaca-run.sh= on Notification and Stop events. Delegates to the =claude-code= Emacs package's hook wrapper, which updates the Emacs-side state (e.g. mode-line indicators) in response to Claude Code lifecycle events.
