This manual describes the features and customization options for the Emacs Lisp file =eww-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=eww-extras.el= extends EWW (Emacs Web Wowser), the built-in Emacs web
browser, with capabilities for generating files from web pages, managing
readability mode, navigating URL hierarchies, and integrating with other
packages.  It depends on the =paths= and =el-patch= packages.

The most substantial feature is the ability to generate PDF and HTML
files from any URL using headless Chrome.  This requires Chrome to be
installed and uses copies of the user's Chrome profile data for
authentication, enabling downloads from sites that require login.  The
generated files are placed in the downloads directory and can optionally
trigger a callback function (e.g., to attach the file to a BibTeX
entry).

The package also enhances EWW's readability mode.  By default, every
page rendered in EWW is automatically converted to its "readable" form
(stripping navigation, ads, and other clutter).  A file-backed exception
list allows excluding specific domains where readability mode produces
poor results.  Adding exceptions is a single command away.  The package
also patches an upstream bug in ~eww-score-readability~ that causes
CSS-heavy pages to render as empty buffers ([[#h:patches][Patches]]).

For navigation, the package provides Vimium-style commands: editing the
current URL, opening kill ring content, and moving up or to the root of
the URL hierarchy.  It also provides commands for switching between EWW
and xwidget-webkit for different rendering needs.

Integration with other packages includes handling YouTube URLs via
=elfeed-tube= (for transcript extraction) and =mpv= (for video
playback), and adding the current page to a bibliography via
=zotra-extras=.

* User options
:PROPERTIES:
:CUSTOM_ID: h:user-options
:END:

** Readability exceptions
:PROPERTIES:
:CUSTOM_ID: h:readability-exceptions
:END:

#+vindex: eww-extras-readable-exceptions
#+vindex: eww-extras-readable-exceptions-file
The user option ~eww-extras-readable-exceptions~ holds a list of URL
patterns (strings treated as regular expressions) for which
~eww-readable~ should not be automatically applied.  The default value
is an empty list.  This variable is populated at load time from the file
specified by ~eww-extras-readable-exceptions-file~ and updated whenever
you add a new exception via
~eww-extras-add-domain-to-readable-exceptions~
([[#h:adding-exceptions][Adding readability exceptions]]).

The user option ~eww-extras-readable-exceptions-file~ specifies the
path to the file containing readability exception patterns, one per
line.  The default value is =etc/eww-readable-exceptions.txt= within
the Emacs configuration directory (via ~paths-dir-dotemacs~).  You would
change this if you want to store the exceptions file elsewhere.

** Chrome data directory
:PROPERTIES:
:CUSTOM_ID: h:chrome-data-option
:END:

#+vindex: eww-extras-chrome-data-dir
The user option ~eww-extras-chrome-data-dir~ specifies the path to the
main Google Chrome profile directory.  The default value is
=~/Library/Application Support/Google/Chrome/=.  This directory is used
as the source when creating profile copies for headless Chrome sessions
([[#h:chrome-data-management][Chrome data management]]).  You should only change this if Chrome stores
its data in a non-standard location.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Browsing local files
:PROPERTIES:
:CUSTOM_ID: h:browsing-files
:END:

#+findex: eww-extras-browse-file
The command ~eww-extras-browse-file~ opens a local file in EWW.  It
tries several sources for the file path: an explicit FILE argument, the
filename at point, the current buffer's file, or a prompt.  This is
useful for previewing local HTML files or documents in the EWW renderer.

** Generating files from URLs
:PROPERTIES:
:CUSTOM_ID: h:generating-files
:END:

#+findex: eww-extras-url-to-pdf
#+findex: eww-extras-url-to-html
The command ~eww-extras-url-to-pdf~ generates a PDF file from a URL
using headless Chrome's print-to-PDF feature.  The URL is determined
from context (the current EWW page, kill ring, or a prompt).  The output
file is saved to the downloads directory (~paths-dir-downloads~) with a
name derived from the BibTeX key (if called from Ebib or BibTeX mode)
or the buffer name.

The command ~eww-extras-url-to-html~ generates an HTML file from a URL
using headless Chrome's DOM dump feature.  When called from a
bibliographic context, the file name is based on the BibTeX key;
otherwise, it uses a slugified version of the page title.

Both commands accept an optional CALLBACK function that is called with
the output file path and BibTeX key upon completion.  This enables
workflows like automatically attaching the generated file to a BibTeX
entry.

The commands require Chrome profile copies to exist for authentication.
Use ~eww-extras-chrome-copy-data-dirs~ to create or update them
([[#h:chrome-data-management][Chrome data management]]).  The processes run asynchronously; check the
=*eww-extras download file process*= buffer for detailed output.

** Chrome data management
:PROPERTIES:
:CUSTOM_ID: h:chrome-data-management
:END:

#+findex: eww-extras-chrome-copy-data-dirs
#+findex: eww-extras-chrome-delete-data-dirs
The command ~eww-extras-chrome-copy-data-dirs~ creates or updates two
copies of the Chrome profile directory.  Two copies are needed to allow
simultaneous PDF and HTML generation (one headless Chrome instance per
copy).  The copies use =rsync= for efficiency: the first run may take a
while, but subsequent updates are fast.

Chrome must be completely closed before running this command, because
headless Chrome cannot access a profile that is locked by a running
instance.  The command verifies this and signals an error if Chrome is
still running.

The command ~eww-extras-chrome-delete-data-dirs~ deletes both profile
copies after prompting for confirmation.  This can free substantial disk
space.

** Adding readability exceptions
:PROPERTIES:
:CUSTOM_ID: h:adding-exceptions
:END:

#+findex: eww-extras-add-domain-to-readable-exceptions
The command ~eww-extras-add-domain-to-readable-exceptions~ prompts for a
domain pattern and adds it to the readability exceptions file.  The
domain of the current EWW page (or a URL in the kill ring) is offered as
the default input.  After adding the exception, it reloads the
exceptions list and refreshes the current page.

This command provides a quick way to permanently opt out of readability
mode for sites where it strips important content.

** Vimium-like navigation
:PROPERTIES:
:CUSTOM_ID: h:vimium-navigation
:END:

#+findex: eww-extras-edit-current-url
#+findex: eww-extras-open-with-recent-kill-ring
#+findex: eww-extras-go-up-url-hierarchy
#+findex: eww-extras-go-to-root-url-hierarchy
The command ~eww-extras-edit-current-url~ pre-fills the minibuffer with
the current EWW URL and lets you edit it or enter a new search query.
With a prefix argument, the result opens in a new EWW buffer.

The command ~eww-extras-open-with-recent-kill-ring~ opens the most
recent kill ring entry as a URL in EWW.  With a prefix argument, it
opens in a new EWW buffer.

The command ~eww-extras-go-up-url-hierarchy~ navigates one level up in
the URL path.  For example, from =example.com/a/b/c= it navigates to
=example.com/a/b=.

The command ~eww-extras-go-to-root-url-hierarchy~ navigates to the root
domain of the current URL, stripping all path components.

** Switching renderers
:PROPERTIES:
:CUSTOM_ID: h:switching-renderers
:END:

#+findex: eww-extras-open-with-xwidget
#+findex: eww-extras-open-with-eww
The command ~eww-extras-open-with-xwidget~ opens the current EWW page
in xwidget-webkit, providing a full WebKit rendering.  This is useful
when EWW's text-mode rendering is insufficient for a page's layout.
The command must be called from an EWW buffer.

The command ~eww-extras-open-with-eww~ performs the reverse operation:
it takes the URL from the current xwidget-webkit session and opens it
in EWW.  This is useful when you want EWW's readability mode or
text-based navigation for a page you are viewing in xwidget-webkit.

** Bibliography integration
:PROPERTIES:
:CUSTOM_ID: h:bibliography-integration
:END:

#+findex: eww-extras-add-entry
The command ~eww-extras-add-entry~ adds the current EWW page's URL to a
bibliography file using ~zotra-extras-add-entry~.  This command is
available only in ~eww-mode~ buffers.

* Functions
:PROPERTIES:
:CUSTOM_ID: h:functions
:END:

** Readability management
:PROPERTIES:
:CUSTOM_ID: h:readability-management
:END:

#+findex: eww-extras-readable-autoview
#+findex: eww-extras-set-readable-exceptions-from-file
The function ~eww-extras-readable-autoview~ is added to
~eww-after-render-hook~.  It automatically calls ~eww-readable~ unless
the current URL matches a pattern in ~eww-extras-readable-exceptions~ or
the page has no source content to process.  If the readability heuristic
selects a non-visible node (producing an empty buffer), the function
falls back to displaying the full page.  This guards against the
upstream scoring bug described in [[#h:patches][Patches]].

The function ~eww-extras-set-readable-exceptions-from-file~ reads the
exception patterns from ~eww-extras-readable-exceptions-file~ and
populates the ~eww-extras-readable-exceptions~ variable.  It is called
automatically at load time and after adding a new exception.

** YouTube handling
:PROPERTIES:
:CUSTOM_ID: h:youtube-handling
:END:

#+findex: eww-extras-browse-youtube
The function ~eww-extras-browse-youtube~ is designed as a handler for
YouTube URLs (e.g., in ~browse-url-handlers~).  When called with a
YouTube URL, it kills the current EWW buffer, ensures the external
video player is running (defaulting to =mpv=), and fetches the video
transcript using ~elfeed-tube-fetch~.  This provides a workflow where
you read the transcript in Emacs while watching the video in an external
player.

** File generation internals
:PROPERTIES:
:CUSTOM_ID: h:file-generation-internals
:END:

#+findex: eww-extras-url-to-file
#+findex: eww-extras-url-to-file-make-command
#+findex: eww-extras-url-to-file-sentinel
#+findex: eww-extras-run-callback
The function ~eww-extras-url-to-file~ is the core logic shared by
~eww-extras-url-to-pdf~ and ~eww-extras-url-to-html~.  It determines
the URL and output file name from context, constructs and starts the
headless Chrome process, and sets up the process sentinel.  An optional
~key~ argument lets callers supply the BibTeX key directly, bypassing
the buffer-derived lookup.

The function ~eww-extras-url-to-file-make-command~ builds the shell
command for headless Chrome, including GPU disabling flags and a
15-second timeout.

The function ~eww-extras-url-to-file-sentinel~ creates a process
sentinel that checks the exit status.  It runs the callback on success,
or if the file was created despite a non-zero exit status (which can
happen with certain Chrome warnings).

The function ~eww-extras-run-callback~ safely executes the optional
callback function with the output file and BibTeX key as arguments.

:PROPERTIES:
:CUSTOM_ID: h:patches
:END:

** Readability scoring
:PROPERTIES:
:CUSTOM_ID: h:readability-scoring-patch
:END:

#+findex: eww-score-readability
The built-in function ~eww-score-readability~ determines which part of a
web page contains the main textual content for ~eww-readable~.  It
excludes ~script~, ~head~, and ~comment~ tags from scoring, but does not
exclude ~style~ tags.  CSS text inside =<style>= elements is therefore
counted as readable content, which can cause CSS-heavy nodes (such as
Wikipedia's =div.navbox-styles=) to win the readability contest over the
actual article.  The result is an empty buffer because the selected node
contains only invisible CSS.

This package uses ~el-patch~ to add ~style~ to the exclusion list in
~eww-score-readability~, giving =<style>= elements a score of -2 (the
same penalty applied to ~script~ and ~head~).  As an additional safety
net, ~eww-extras-readable-autoview~ detects when the readable render
produces an empty buffer and falls back to displaying the full page
([[#h:readability-management][Readability management]]).
