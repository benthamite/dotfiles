This manual describes the features and customization options for the Emacs Lisp file =gptel-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=gptel-extras.el= extends =gptel=, the Emacs package that integrates various Large Language Models---ChatGPT, Gemini, Claude, Perplexity, local models, and others---into Emacs buffers. While =gptel= provides the core infrastructure for sending prompts and receiving responses, =gptel-extras.el= adds workflows and conveniences that make day-to-day use more productive.

The package depends on =gptel= (version 0.7.1 or later) and =paths= (for directory path resolution).

The extensions fall into several thematic areas:

- *Aider-style context management.* Commands for generating a repository map (via the external =aider= tool), caching it with configurable time-to-live and git-aware invalidation, and automatically injecting both the map and a project conventions file into the =gptel= context before each request ([[#h:aider-context][Aider context management]]).

- *Conversation management.* Automatic saving of new =gptel= buffers to a dedicated directory, continuation of conversations across linked buffers, and a workaround for unresponsive =gptel= sessions ([[#h:conversation-management][Conversation management]]).

- *ChatGPT import.* Bulk import of conversations exported from ChatGPT into Org files ([[#h:chatgpt-import][ChatGPT import]]).

- *Commit summarization.* Sending selected commit diffs from a Magit log buffer to an LLM for changelog-style summaries ([[#h:commit-summarization][Commit summarization]]).

- *Search integration.* A command that simultaneously opens a web search and queries a configured AI model (e.g., Perplexity Sonar) with the same query ([[#h:search][Search integration]]).

- *Email drafting.* Integration with =mu4e= and =org-msg= for LLM-assisted email reply drafting ([[#h:mu4e-integration][Email drafting with mu4e]]).

- *Gptel tools.* A collection of tools---filesystem editing, bibliography search, and entry addition---that =gptel= can invoke via its tool-use feature, plus a macro for defining tool presets ([[#h:tools][Tools]]).

- *Model-specific workarounds.* Automatic Mullvad VPN activation for Gemini requests and a command for fixing garbled characters in LLM output ([[#h:model-workarounds][Model-specific workarounds]]).

- *General convenience commands.* Toggling default major mode, navigating to the end of a buffer before sending, rewriting function definitions via Embark, and more ([[#h:misc-commands][General convenience commands]]).

* User options
:PROPERTIES:
:CUSTOM_ID: h:user-options
:END:

** Aider context options
:PROPERTIES:
:CUSTOM_ID: h:aider-context-options
:END:

#+vindex: gptel-extras-add-repo-map-to-context
The user option ~gptel-extras-add-repo-map-to-context~ controls whether the repository map is automatically added to the =gptel= context before every request sent via ~gptel-send~. When non-nil, the function ~gptel-extras-add-context-files~ generates (or retrieves from cache) the repository map and injects it into the context ([[#h:automatic-context-injection][Automatic context injection]]).

The default value is ~nil~. Enable this option when you are using =gptel= for coding tasks within a project and want the LLM to have an overview of the codebase structure, similar to Aider's built-in repo map feature.

This is a boolean option.

#+vindex: gptel-extras-add-conventions-to-context
The user option ~gptel-extras-add-conventions-to-context~ controls whether a =conventions.md= file from the project root is automatically added to the =gptel= context before every request. When non-nil, ~gptel-extras-add-context-files~ calls ~gptel-extras-add-conventions-to-context~ to include this file ([[#h:automatic-context-injection][Automatic context injection]]).

The default value is ~nil~. Enable this option if your project has a =conventions.md= file documenting coding standards and you want the LLM to respect them.

This is a boolean option.

#+vindex: gptel-extras-repo-map-cache-ttl
The user option ~gptel-extras-repo-map-cache-ttl~ specifies the time-to-live for cached repository maps, in seconds. After this duration, the cached map is considered stale and will be regenerated on the next request.

The default value is ~3600~ (one hour). Increase this value for large, stable repositories where the structure rarely changes; decrease it for rapidly evolving codebases.

This is an integer option.

#+vindex: gptel-extras-repo-map-invalidate-on-git-changes
The user option ~gptel-extras-repo-map-invalidate-on-git-changes~ controls whether the repository map cache is invalidated when the git HEAD reference changes, regardless of the TTL. When non-nil, any new commit, branch switch, or rebase that alters HEAD triggers regeneration of the map.

The default value is ~t~. Disable this option only if you want the map to persist strictly based on the TTL, ignoring git activity.

This is a boolean option.

** Email drafting options
:PROPERTIES:
:CUSTOM_ID: h:mu4e-options
:END:

#+vindex: gptel-extras-mu4e-draft-reply-prompt
The user option ~gptel-extras-mu4e-draft-reply-prompt~ specifies the prompt sent to the LLM when drafting an email reply with ~gptel-extras-mu4e-draft-reply~ ([[#h:mu4e-integration][Email drafting with mu4e]]). The prompt instructs the model to draft a message based on the attached email thread, complete any partial draft already present, and produce only the body text (no headers or signature).

The default value includes the user's full name (from ~user-full-name~) to help the LLM identify the sender. Customize this option to adjust the drafting instructions, tone, or language.

This is a string option.

** Search options
:PROPERTIES:
:CUSTOM_ID: h:search-options
:END:

#+vindex: gptel-extras-search-model
The user option ~gptel-extras-search-model~ specifies the backend and model to use for the AI component of ~gptel-extras-search-and-ask-model~ ([[#h:search][Search integration]]). The value is a cons cell of the form =(BACKEND . MODEL)=, where =BACKEND= is a string naming a =gptel= backend and =MODEL= is a symbol identifying the model.

The default value is =("Perplexity" . sonar)=. Change this option if you prefer a different search-oriented model or if the Perplexity backend is not configured.

#+vindex: gptel-extras-search-prefix
The user option ~gptel-extras-search-prefix~ specifies the URL prefix for the web search opened by ~gptel-extras-search-and-ask-model~. The query string is appended (URL-encoded) to this prefix.

The default value is ="https://www.google.com/search?q="=. Change this to use a different search engine (e.g., DuckDuckGo, Bing).

This is a string option.

** Gemini and VPN options
:PROPERTIES:
:CUSTOM_ID: h:gemini-options
:END:

#+vindex: gptel-extras-gemini-mullvad-disconnect-after
The user option ~gptel-extras-gemini-mullvad-disconnect-after~ specifies the number of minutes after which the Mullvad VPN connection, activated for a Gemini request, is automatically disconnected ([[#h:model-workarounds][Model-specific workarounds]]).

The default value is ~1~. Increase this if your Gemini requests take longer than one minute to complete.

This is an integer option.

** Directory options
:PROPERTIES:
:CUSTOM_ID: h:directory-options
:END:

#+vindex: gptel-extras-dir
The user option ~gptel-extras-dir~ specifies the directory where =gptel= conversation buffers are automatically saved by ~gptel-extras-save-buffer~ ([[#h:saving-buffers][Saving buffers]]). The directory is sorted chronologically via a =.dir-locals.el= file, since the typical use case is to locate a recently modified conversation.

The default value is derived from ~paths-dir-notes~ with =gptel/= appended. Change this if you store your notes elsewhere.

This is a directory option.

#+vindex: gptel-extras-chatgpt-import-dir
The user option ~gptel-extras-chatgpt-import-dir~ specifies the directory where imported ChatGPT conversations are saved by ~gptel-extras-import-chatgpt-conversations~ ([[#h:chatgpt-import][ChatGPT import]]).

The default value is derived from ~paths-dir-notes~ with =chatgpt/= appended.

This is a directory option.

** Alert options
:PROPERTIES:
:CUSTOM_ID: h:alert-options
:END:

#+vindex: gptel-extras-alert-when-finished
The user option ~gptel-extras-alert-when-finished~ controls whether an alert is displayed after =gptel= finishes inserting a response. It accepts three types of values:

- ~t~: always display an alert.
- ~nil~: never display an alert.
- A list of model symbols (e.g., ~(gemini-pro claude-opus-4-6)~): display an alert only when the response comes from one of the listed models.

The default value is ~nil~. This is useful when using slow models where you might switch to another task while waiting for the response. The alert relies on the =alert= package.

** Commit summarization
:PROPERTIES:
:CUSTOM_ID: h:commit-summarization
:END:

#+findex: gptel-extras-summarize-commit-diffs
#+vindex: gptel-extras-changelog-file
When reviewing a series of commits in a =magit-log-mode= buffer, you may want a quick, high-level summary of what changed. The command ~gptel-extras-summarize-commit-diffs~ takes the active region in the log buffer, extracts the diffs for all commits within that region, and sends them to the LLM with a prompt requesting an Org-mode-formatted changelog summary.

The summary is displayed in a dedicated =*Commit Summary*= buffer. The prompt is structured using the template in the constant ~gptel-extras-changelog-file~ (which points to the changelog template file in the extras directory), helping the LLM produce output organized by package or feature.

With a prefix argument (=C-u=), the command includes full diff content with unified context rather than the default three-line unified diff. This can help the LLM understand broader context around each change.

Tool use (~gptel-use-tools~) is disabled for this request, since it is a programmatic call that should produce a plain-text summary without invoking any tools.

This command must be invoked from a =magit-log-mode= buffer with an active region spanning the commits of interest.

** Aider context management
:PROPERTIES:
:CUSTOM_ID: h:aider-context
:END:

These commands manage the repository map and conventions file that provide project-level context to the LLM. The repository map is generated by the external =aider= tool (=aider --show-repo-map=) and cached in memory with configurable TTL and git-aware invalidation ([[#h:aider-context-options][Aider context options]]).

*** Showing and managing the repository map
:PROPERTIES:
:CUSTOM_ID: h:repo-map-commands
:END:

#+findex: gptel-extras-show-repo-map
#+vindex: gptel-extras-repo-map-buffer-name-formatter
The command ~gptel-extras-show-repo-map~ generates (or retrieves from cache) the repository map for the current project and displays it in a dedicated buffer. The buffer name follows the format string ~gptel-extras-repo-map-buffer-name-formatter~ (default =*Repo Map: %s*=, where =%s= is the project name). Use this to inspect the map before deciding whether to include it in the context.

#+findex: gptel-extras-add-repo-map-to-context
The command ~gptel-extras-add-repo-map-to-context~ generates the repository map, removes any previously added map from the =gptel= context, and then adds the current map with explanatory preamble text. The preamble instructs the LLM to use the map for understanding project structure and to request relevant files.

#+findex: gptel-extras-add-conventions-to-context
The command ~gptel-extras-add-conventions-to-context~ adds the =conventions.md= file from the current project root to the =gptel= context. The project root is determined by ~gptel-extras-get-repo~ ([[#h:functions][Functions]]).

#+findex: gptel-extras-remove-repo-map-from-context
The command ~gptel-extras-remove-repo-map-from-context~ removes the repository map buffer from the =gptel= context without deleting the buffer itself.

#+findex: gptel-extras-invalidate-repo-map-cache
The command ~gptel-extras-invalidate-repo-map-cache~ removes the cached repository map for the current project, forcing regeneration on the next request. This is useful after large structural changes that the git-aware invalidation might not capture (e.g., adding many new files without committing).

*** Toggling automatic context injection
:PROPERTIES:
:CUSTOM_ID: h:toggle-aider
:END:

#+findex: gptel-extras-toggle-aider-files
The command ~gptel-extras-toggle-aider-files~ toggles both ~gptel-extras-add-repo-map-to-context~ and ~gptel-extras-add-conventions-to-context~ simultaneously. If both are currently disabled, they are enabled, and vice versa. This provides a quick way to switch between coding mode (with full project context) and general conversation mode (without it).

** Conversation management
:PROPERTIES:
:CUSTOM_ID: h:conversation-management
:END:

*** Continuing in a new buffer
:PROPERTIES:
:CUSTOM_ID: h:continue-in-new-buffer
:END:

#+findex: gptel-extras-continue-in-new-buffer
Long conversations can become unwieldy and expensive to process. The command ~gptel-extras-continue-in-new-buffer~ creates a new =gptel= buffer whose heading is derived from the current buffer's top-level heading, with a number appended or incremented (e.g., "My topic" becomes "My topic 2", "My topic 2" becomes "My topic 3").

The command also inserts a "Continued here" link (using Org IDs) at the end of the original buffer, so you can easily navigate between parts of the conversation. The current buffer must have ~gptel-mode~ active.

*** Saving buffers
:PROPERTIES:
:CUSTOM_ID: h:saving-buffers
:END:

#+findex: gptel-extras-save-buffer
The function ~gptel-extras-save-buffer~ is designed as =:after= advice on the main ~gptel~ function. When =gptel= is called interactively to create a new buffer (rather than reusing an existing file), this function automatically saves the buffer to ~gptel-extras-dir~ ([[#h:directory-options][Directory options]]). The filename is a slugified version of the buffer name with the appropriate extension (=.org= or =.md= depending on the major mode).

In Org mode, the function also inserts the buffer name as a level-1 heading at the top. During the initial save, ~gptel--save-state~ is temporarily removed from ~before-save-hook~ to avoid errors from the incomplete gptel state at that early stage.

*** Recovering from unresponsive sessions
:PROPERTIES:
:CUSTOM_ID: h:kill-reopen
:END:

#+findex: gptel-extras-kill-buffer-then-reopen-file
Occasionally, =gptel= may stop processing requests in a buffer for unclear reasons. The command ~gptel-extras-kill-buffer-then-reopen-file~ works around this by saving the buffer, killing it, and reopening the file. In Org mode buffers, it also unfolds all headings after reopening.

This command can be added to ~gptel-post-response-functions~ for automatic recovery, or invoked manually with =M-x gptel-extras-kill-buffer-then-reopen-file=.

** ChatGPT import
:PROPERTIES:
:CUSTOM_ID: h:chatgpt-import
:END:

#+findex: gptel-extras-import-chatgpt-conversations
The command ~gptel-extras-import-chatgpt-conversations~ processes the =conversations.json= file included in a ChatGPT data export. It creates one Org file per conversation in ~gptel-extras-chatgpt-import-dir~ ([[#h:directory-options][Directory options]]), using a slugified version of the conversation title as the filename.

Each file contains the conversation title as a level-1 heading, with user and assistant messages as level-2 headings beneath it. Markdown content from ChatGPT is converted to Org syntax via Pandoc (using ~org-extras-convert-markdown-to-org~). A link to the original conversation on the ChatGPT website is included when the conversation ID is available.

With a prefix argument (=C-u=), archived conversations are included in the import. By default, archived conversations are skipped.

** Email drafting with mu4e
:PROPERTIES:
:CUSTOM_ID: h:mu4e-integration
:END:

#+findex: gptel-extras-mu4e-draft-reply
The command ~gptel-extras-mu4e-draft-reply~ uses the LLM to draft a reply to the email currently being composed in =mu4e= (via =org-msg=). The command adds the entire current buffer as context, positions the cursor after the email headers, and sends the prompt defined in ~gptel-extras-mu4e-draft-reply-prompt~ ([[#h:mu4e-options][Email drafting options]]).

If you have already started writing a partial draft, the LLM will attempt to complete it. The prompt instructs the model to produce only the body text, without headers or signature.

With a prefix argument (=C-u=), the command lets you edit the prompt before sending it, which is useful for providing additional instructions (e.g., requesting a particular tone).

Before sending, the command calls ~gptel-extras-warn-when-context~ ([[#h:context-warning][Context warning]]) to alert you if there is existing context that might interfere. After the request is sent, all context is removed.

Tool use (~gptel-use-tools~) is disabled for this request, since the reply drafting is a programmatic call that should produce plain text without invoking any tools.

** Search integration
:PROPERTIES:
:CUSTOM_ID: h:search
:END:

#+findex: gptel-extras-search-and-ask-model
The command ~gptel-extras-search-and-ask-model~ prompts for a search query, then performs two actions in parallel:

1. Opens the query in a web browser using ~gptel-extras-search-prefix~ ([[#h:search-options][Search options]]).
2. Creates a new =gptel= buffer, sets the backend and model to ~gptel-extras-search-model~ buffer-locally, and sends the query as a prompt.

This is useful for quick research where you want both traditional web results and an AI-generated answer. The command calls ~gptel-extras-warn-when-context~ before proceeding ([[#h:context-warning][Context warning]]).

** General convenience commands
:PROPERTIES:
:CUSTOM_ID: h:misc-commands
:END:

*** Toggling default major mode
:PROPERTIES:
:CUSTOM_ID: h:toggle-major-mode
:END:

#+findex: gptel-extras-toggle-major-mode
The command ~gptel-extras-toggle-major-mode~ toggles ~gptel-default-mode~ between ~org-mode~ and ~markdown-mode~. New =gptel= buffers will use whichever mode is active after the toggle. This does not affect existing buffers.

*** Navigating and sending
:PROPERTIES:
:CUSTOM_ID: h:goto-end-and-send
:END:

#+findex: gptel-extras-goto-end-and-send
The command ~gptel-extras-goto-end-and-send~ moves point to the end of the current buffer and then calls ~gptel-send~. This is convenient when you have typed a prompt somewhere in the middle of a buffer and want to ensure the full conversation is sent from the end.

*** Rewriting function definitions
:PROPERTIES:
:CUSTOM_ID: h:rewrite-defun
:END:

#+findex: gptel-extras-rewrite-defun
The command ~gptel-extras-rewrite-defun~ marks the function definition at point (using ~mark-defun~) and then invokes ~gptel-rewrite~ on the selected region. This is intended for use with =embark=: binding ~gptel-rewrite~ directly to an Embark action would not work because it requires an active region.

If point is not on a function definition, the command signals an error.

*** Fixing garbled characters
:PROPERTIES:
:CUSTOM_ID: h:fix-garbled-chars
:END:

#+findex: gptel-extras-fix-garbled-chars
The command ~gptel-extras-fix-garbled-chars~ replaces =let/= with =let*= and ==(= with =`(= from point to the end of the buffer. These substitutions fix common encoding artifacts that some LLMs introduce when generating Emacs Lisp code.

*** Adding a buffer as context
:PROPERTIES:
:CUSTOM_ID: h:add-buffer
:END:

#+findex: gptel-extras-add-buffer
The command ~gptel-extras-add-buffer~ adds the entire contents of a buffer to the =gptel= context. When called interactively, it adds the current buffer. When called from Lisp, pass a buffer object as the argument.

This is useful when you want the LLM to consider the full content of a file or buffer without manually selecting a region. It is used internally by ~gptel-extras-mu4e-draft-reply~ ([[#h:mu4e-integration][Email drafting with mu4e]]).

*** Context warning
:PROPERTIES:
:CUSTOM_ID: h:context-warning
:END:

#+findex: gptel-extras-warn-when-context
The command ~gptel-extras-warn-when-context~ checks whether the =gptel= context is non-empty and, if so, prompts for confirmation before proceeding. If the user declines, the command offers to clear the context entirely. This prevents accidentally sending a request with stale or irrelevant context.

This function is called by ~gptel-extras-mu4e-draft-reply~ ([[#h:mu4e-integration][Email drafting with mu4e]]) and ~gptel-extras-search-and-ask-model~ ([[#h:search][Search integration]]).

*** Connecting to MCP servers
:PROPERTIES:
:CUSTOM_ID: h:connect-servers
:END:

#+findex: gptel-extras-connect-to-all-servers
The command ~gptel-extras-connect-to-all-servers~ connects to all configured MCP (Model Context Protocol) servers via ~gptel-mcp-connect~, then immediately applies the =tools-clear= preset to deactivate all tools. This is useful during initialization: it establishes the server connections so that tools are available on demand, without activating any tools by default.

:PROPERTIES:

:END:

** Automatic context injection
:PROPERTIES:
:CUSTOM_ID: h:automatic-context-injection
:END:

#+findex: gptel-extras-add-context-files
The function ~gptel-extras-add-context-files~ is added as =:before= advice on ~gptel-send~. Before every request, it checks the user options ~gptel-extras-add-repo-map-to-context~ and ~gptel-extras-add-conventions-to-context~ ([[#h:aider-context-options][Aider context options]]) and, when enabled, calls the corresponding commands to inject context.

** Post-response hooks
:PROPERTIES:
:CUSTOM_ID: h:post-response
:END:

#+findex: gptel-extras-alert-when-finished
The function ~gptel-extras-alert-when-finished~ is added to ~gptel-post-response-functions~. After =gptel= inserts a response, it checks ~gptel-extras-alert-when-finished~ ([[#h:alert-options][Alert options]]) and, if the condition is met, displays an alert via the =alert= package.

#+findex: gptel-extras-generate-latex-previews
The function ~gptel-extras-generate-latex-previews~ is intended for ~gptel-post-response-functions~. When the current buffer is in Org mode and its ~default-directory~ matches ~gptel-extras-dir~, the function generates LaTeX previews by calling ~org-latex-preview~. This provides automatic rendering of mathematical notation in =gptel= conversation files.

** Model-specific workarounds
:PROPERTIES:
:CUSTOM_ID: h:model-workarounds
:END:

#+findex: gptel-extras-set-mullvad
The function ~gptel-extras-set-mullvad~ is added as =:around= advice on ~gptel-curl-get-response~. When ~gptel-model~ is ~gemini-pro~, it activates a Mullvad VPN connection (via ~mullvad-connect-to-website~) before the request is sent, with an automatic disconnect timer set to ~gptel-extras-gemini-mullvad-disconnect-after~ minutes ([[#h:gemini-options][Gemini and VPN options]]). This bypasses location restrictions that Gemini imposes in certain regions.

The VPN connection is made silently (no user prompt). The =mullvad= package must be installed and configured separately.

*** Fast JSON logging
:PROPERTIES:
:CUSTOM_ID: h:fast-json-logging
:END:

#+findex: gptel-extras--fast-log
The function ~gptel-extras--fast-log~ is added as =:around= advice on ~gptel--log~. The upstream ~gptel--log~ function calls ~json-pretty-print~, which uses ~replace-region-contents~ internally.  That function runs the O(n²) =compareseq= diff algorithm, which freezes Emacs on large JSON payloads (such as full LLM request/response bodies).

This advice replaces the pretty-printing step with a direct parse-then-insert approach: the JSON data is parsed with ~json-parse-string~ and re-inserted with ~json-encode~ (with ~json-encoding-pretty-print~ bound to =t=), avoiding the diff entirely.

** Running queries programmatically
:PROPERTIES:
:CUSTOM_ID: h:run-query
:END:

#+findex: gptel-extras-run-query
The function ~gptel-extras-run-query~ creates a new =gptel= buffer from a query string, optionally setting a specific backend and model. It is used internally by ~gptel-extras-search-and-ask-model~ ([[#h:search][Search integration]]) and can be called from Lisp for any programmatic query dispatch.

The buffer name is a slugified version of the query with the appropriate file extension. If =BACKEND= or =MODEL= are nil, the global defaults are used.

Tool use (~gptel-use-tools~) is disabled for this request, since it is a programmatic call that should produce a plain-text response without invoking any tools.

** Setting backend and model
:PROPERTIES:
:CUSTOM_ID: h:set-backend-and-model
:END:

#+findex: gptel-extras-set-backend-and-model
The function ~gptel-extras-set-backend-and-model~ sets ~gptel-backend~ and ~gptel-model~ buffer-locally in the current Org buffer. Pass the backend name as a string and the model as a symbol. If either argument is nil, the current global value is used.

This is a helper function (not a command) used by ~gptel-extras-run-query~ ([[#h:run-query][Running queries programmatically]]) and ~gptel-extras-search-and-ask-model~ ([[#h:search][Search integration]]).

** Buffer validation
:PROPERTIES:
:CUSTOM_ID: h:buffer-validation
:END:

#+findex: gptel-extras-ensure-gptel-mode
The function ~gptel-extras-ensure-gptel-mode~ signals an error if ~gptel-mode~ is not active in the current buffer. It is used as a guard by commands such as ~gptel-extras-continue-in-new-buffer~ ([[#h:continue-in-new-buffer][Continuing in a new buffer]]) that require an active =gptel= session.

** Repository identification
:PROPERTIES:
:CUSTOM_ID: h:repo-identification
:END:

#+findex: gptel-extras-get-repo
The function ~gptel-extras-get-repo~ determines the repository path for the current request. In Org mode, it checks the =GPTEL_REPO= property at the document level; if not set, it prompts the user and stores the selection. In other modes, it uses ~project-root~ if a project is detected, falling back to a directory prompt.

#+findex: gptel-extras-get-repo-map
The function ~gptel-extras-get-repo-map~ returns the repository map for the current project, using the cached version if it is still valid (checked via ~gptel-extras-repo-map-cache-valid-p~). When the cache is stale or empty, it calls =aider --show-repo-map= to generate a fresh map and stores it along with the current timestamp and git HEAD reference.

** Preset macro
:PROPERTIES:
:CUSTOM_ID: h:preset-macro
:END:

#+findex: gptel-extras-make-tool-presets
The macro ~gptel-extras-make-tool-presets~ creates two =gptel= presets for a given set of tools: a base preset that sets the active tools to exactly the specified list, and an additive preset (with =+= appended to the name) that appends the tools to whatever tools are already active. This is useful for defining reusable tool configurations.

For example:

#+begin_src emacs-lisp
(gptel-extras-make-tool-presets tools-web-search
  "web search tools"
  '(web_search browse_url))
#+end_src

This creates presets named =tools-web-search= (sets tools to the two specified) and =tools-web-search+= (adds them to existing tools).

* Tools
:PROPERTIES:
:CUSTOM_ID: h:tools
:END:

=gptel-extras.el= registers several tools that =gptel= can invoke via its tool-use feature. These tools allow the LLM to interact with the filesystem and bibliography.

** Filesystem tools
:PROPERTIES:
:CUSTOM_ID: h:filesystem-tools
:END:

Three filesystem tools are provided, forming a fallback chain for file modification:

#+findex: gptel-extras-edit-file
1. *edit_file* (~gptel-extras-edit-file~): The primary file editing tool. It applies a list of edits to a file, where each edit specifies an ~:old_string~ to find and a ~:new_string~ to replace it with. The tool first attempts an exact match; if that fails, it uses fuzzy matching with normalized whitespace, case-insensitive search, and flexible regex strategies.

2. *apply_diff*: Applies a unified diff (patch) to a file using the external =patch= command. Useful when the LLM generates changes in diff format. Supports configurable patch options (=-p0=, =-p1=, etc.).

3. *replace_file_contents*: Replaces the entire content of a file. This is the last resort, as it requires sending the full file content and is therefore token-inefficient.

All three tools modify files directly on disk without user confirmation.

** BibTeX tools
:PROPERTIES:
:CUSTOM_ID: h:bibtex-tools
:END:

#+findex: gptel-extras-citar-search
The tool *search_bibliography* (~gptel-extras-citar-search~) searches the user's bibliography via =citar=. It splits the search string into space-separated terms and returns entries matching all terms as whole words. The search is case-insensitive and supports pagination via =limit= and =offset= parameters.

Each result is a pair of =(FORMATTED-CITATION . CITE-KEY)=. The default limit is 10 results; use -1 for no limit.

#+findex: gptel-extras-add-bib-entry
The tool *add_bib_entry* (~gptel-extras-add-bib-entry~) adds a bibliographic entry for a URL, ISBN, or DOI to a specified BibTeX file. It delegates to ~zotra-extras-add-entry~ from =zotra-extras.el= and returns the citation key of the newly added entry.

** Robustifying tool presets
:PROPERTIES:
:CUSTOM_ID: h:append-fix
:END:

#+findex: gptel-extras--modify-value-append-fix
The function ~gptel-extras--modify-value-append-fix~ is added as =:around= advice on ~gptel--modify-value~. It normalizes =:append= specifications in preset plists so that tool names (strings or symbols) are properly wrapped in a list when appending to an existing list value. Without this fix, appending tool names could be misinterpreted as string appends, causing errors.
