**** Directory topology
:PROPERTIES:
:CUSTOM_ID: h:directory-topology
:ID:       EB7422C8-E371-4F9B-AE1F-D6B6473F11BB

Three Claude-specific layers work together: this dotfiles directory (source of
truth), the shared =~/.claude/= hub, and per-account overlay directories. The
authoritative cross-tool path-role table lives in =../agents/README.org=.

*Source of truth* (version-controlled in this repo):

#+begin_example
~/My Drive/dotfiles/claude/           ← global config source of truth
├── CLAUDE.md                         ← global conventions for all projects
├── hooks/                            ← hook scripts
├── skills/                           ← skill directories
├── bin/batch-test.sh                 ← elpaca-aware emacs --batch test runner
├── bin/sync-mcp-servers.sh           ← MCP server sync utility
├── bin/gmail.py                      ← Gmail API CLI (--account epoch|personal; attachments supported for send/draft/reply)
├── bin/sheets.py                     ← Sheets API CLI (--account epoch|personal)
├── bin/slack.py                      ← Slack web-API CLI, including permalink/thread fetches
├── bin/_gworkspace_auth.py           ← shared OAuth helper for gmail.py / sheets.py
├── ../bin/ai-config-sync             ← Claude/Codex sync audit and hook checker
└── README.org                        ← this file
#+end_example

Note: =settings.json= is intentionally *not* tracked here. Claude Code writes to =~/.claude/settings.json= directly (via =/theme=, =/effort=, =/permissions=, plugin toggles, etc.), and those atomic writes would break any symlink back to this directory. The file is the canonical live config; edit it in place.

*Shared hub* (=~/.claude/=):

#+begin_example
~/.claude/
├── CLAUDE.md          → dotfiles/claude/CLAUDE.md (symlink)
├── skills/            → dotfiles/claude/skills/ (symlink)
├── settings.json      ← canonical, mutable, not tracked
├── settings.local.json ← machine-specific, not tracked
├── projects/          ← trust entries, session logs (managed by Claude Code)
├── history.jsonl      ← command history (managed by Claude Code)
└── plugins/           ← installed plugins (managed by Claude Code)
#+end_example

*Per-account overlays* (=~/.claude-personal/=, =~/.claude-tlon/=, =~/.claude-epoch/=): see [[*Multi-account setup][multi-account setup]] below.

*Per-project config* (=<project>/.claude/=):

#+begin_example
<project>/.claude/
├── settings.json       ← project-level hooks (optional)
├── settings.local.json ← project-specific permissions (accumulated interactively)
└── skills/             ← project-specific skills (optional)
#+end_example

=../agents/README.org=. In particular, repo-root =.claude/= and =.codex/= paths
are project-local by definition and must not be used as targets for global
=~/.claude/= or =~/.codex/= symlinks.

The global =CLAUDE.md= points agents there when MCP-specific context is needed, so they do not need to read this whole README just to configure or debug an MCP server.

*Service access tools.* The global =CLAUDE.md= lists the local tools used to connect to external services: Anna's Archive, Twitter/X, Gmail, Google Sheets, Slack, Google Calendar, Google Docs/Drive, and GitHub. These local tools are the default path for Claude workflows; MCP servers and hosted connectors are exceptions documented in =context/mcp-servers.md= and service-specific context files.

**** Settings hierarchy
:PROPERTIES:
:CUSTOM_ID: h:settings-hierarchy
:ID:       ABB0C5CA-1B40-47BF-8746-BFFE1F4E420D
:END:

Claude Code loads settings from multiple levels, merged together. Each level can define permissions, hooks, and other options. Permissions are additive across levels. Hooks run from all levels that define them.

| Level | =settings.json= | =settings.local.json= | Purpose |
|-------+------------------+-----------------------+---------|
| User (=~/.claude/=) | Model, theme, global hooks, global permissions | Machine-specific tool whitelists | Applies to all projects |
| Project (=<project>/.claude/=) | Project-specific hooks | Project-specific permissions | Applies only in that project |

The user-level =~/.claude/settings.json= and
=~/.claude/settings.local.json= files are not version-controlled; both are
edited in place in =~/.claude/=.

The global =settings.json= uses =auto= mode (Anthropic's classifier-backed permission mode that approves most actions while flagging risky ones) inside the =permissions= object. All MCP servers are explicitly listed in =permissions.allow= as a workaround for [[https://github.com/anthropics/claude-code/issues/5140][anthropics/claude-code#5140]]. Safety hooks (=guard-external-actions.sh=, =block-destructive-command.sh=, etc.) provide additional guardrails.

**** =CLAUDE.md= loading
:PROPERTIES:
:CUSTOM_ID: h:claude-md-loading
:ID:       D03891CF-C905-480A-9690-62B4F8A7273D
:END:

Claude Code loads =CLAUDE.md= files from several locations. When working in a project, it loads:

1. =~/.claude/CLAUDE.md= — user-level global conventions (symlinked to =dotfiles/claude/CLAUDE.md=)
2. =<project>/CLAUDE.md= — project-level instructions (if present)

Each file should contain distinct content. The global one defines behavioral conventions (verify work, fix root causes, commit immediately, etc.). Project-level ones define project-specific context.

**** Where to put things
:PROPERTIES:
:CUSTOM_ID: h:where-to-put-things
:ID:       A01A5164-CE5F-414D-9999-924B5C89339B
:END:

Use this table for Claude-specific placement. For reusable artifacts that should
work in both Claude and Codex, =../agents/README.org= is authoritative.

| What | Where | Why |
|------+-------+-----|
| New global Claude behavioral rule | =claude/CLAUDE.md= | Applies to all Claude projects; follow =../agents/README.org= when the rule should also apply elsewhere |
| New project-specific instruction | =<project>/CLAUDE.md= | Scoped to that project; follow =../agents/README.org= for counterpart policy |
| New reusable skill or hook | Follow =../agents/README.org= | Shared artifacts need a counterpart or explicit manifest exemption |
| New Claude global hook registration | =~/.claude/settings.json= | Fires in all Claude projects; mutable live config edited in place |
| New Claude project-specific hook registration | =<project>/.claude/settings.json= | Fires only in that project |
| New machine-specific permission | =~/.claude/settings.local.json= | E.g. brew, mpv, pip3 |
| New MCP server | =~/.claude.json=, then run =claude/bin/sync-mcp-servers.sh= | Syncs to account dirs |
| New MCP reference detail | =claude/context/mcp-servers.md= | Keeps MCP-specific instructions out of global memory and the long README |

When adding or changing a reusable Claude skill or hook, follow the counterpart
policy in =../agents/README.org= in the same session. The
=remind-ai-config-sync.sh= and =require-ai-config-sync.sh= hooks make this
visible but do not auto-edit the counterpart. The commit guard checks skill
content after tool-specific frontmatter normalization and only accepts
manifest-based bypasses when the manifest records an artifact-specific status or
exemption. See =../agents/README.org= for the full synchronization contract and
known non-portable cases.

*** Multi-account setup
:PROPERTIES:
:CUSTOM_ID: h:multi-account
:ID:       239E0918-2383-460C-8C9C-2ABC4E3D344D
:END:

Three OAuth accounts (personal, tlon, and epoch) are supported without re-authenticating. Each account uses a separate =CLAUDE_CONFIG_DIR= (=~/.claude-personal/=, =~/.claude-tlon/=, and =~/.claude-epoch/=) to keep its OAuth credentials and identity isolated, while sharing everything else via symlinks back to =~/.claude/=.

Shared (symlinked into each account dir):
- =settings.json=, =settings.local.json= — preferences and permissions
- =skills/= — custom skills
- =projects/= — trusted folders, project config, session logs
- =history.jsonl= — command history
- =plugins/= — installed plugins

Not symlinked:
- =CLAUDE.md= — loaded from =~/.claude/CLAUDE.md= at user level; account dirs do not have their own copy (to avoid loading the same instructions twice)

Separate per account:
- =.claude.json= — OAuth identity, user ID, and MCP servers (synced from =~/.claude.json=)
- Keychain credentials — the actual OAuth tokens

The active account is stored in =~/.claude-current-account=. From Emacs, =agents-claude-start-or-switch= uses the persisted account automatically (configured via =agents-claude-accounts=). Use =agents-claude-select-account= to change it. From the terminal, the shell aliases =claude-personal=, =claude-tlon=, and =claude-epoch= are available.

*Adding a new account:* add it to =agents-claude-accounts=, then run =M-x agents-claude-init-account= *before* invoking =claude= for that account. This creates the config dir and all shared symlinks up front; otherwise =claude= writes real empty =settings.json=/=history.jsonl=/etc. on first login, which divorces the new account from your permissions and history.  If this happens anyway, re-running =init-account= (or =select-account=) heals virgin-state files automatically; non-virgin files are left alone with a warning so you can migrate them by hand.

*1Password service account:* All =op://= references resolve silently via =OP_SERVICE_ACCOUNT_TOKEN= (a 1Password service account token with read-only access to the "Automations" vault).  The token is stored in =pass= at =epoch/1password-service-account-token= and exported in =.zshenv-secrets=.  No Touch ID prompts are needed.  To rotate: delete the old service account in the admin console, create a new one with =op service-account create "Epoch Ops Automation" --vault "Automations:read_items" --raw=, and update the =pass= entry.  Item names in the Automations vault must not contain parentheses (they break =op://= parsing).

*MCP servers:* Placement rules, current inventory, multi-account behavior, and account-specific MCP secret handling live in =context/mcp-servers.md=.

**** Chrome integration and multi-account
:PROPERTIES:
:ID:       E317AE6E-6C3F-4543-A66B-3EF1D50E7719
:END:

Chrome integration uses three Chrome profiles, one per Claude Code account. Browser automation from a given Claude Code session targets the Chrome profile its account is paired to, so agents working on a tlon project automatically drive the tlon browser, and likewise for epoch and personal.

*Profile ↔ account mapping:*

| Chrome profile | Google account (Chrome)      | Anthropic identity (Claude Code) | Claude Code account |
|----------------+------------------------------+----------------------------------+---------------------|
| Default        | =pablo.stafforini@gmail.com=   | =pablo@stafforini.com=             | personal            |
| Profile 2      | =pablo@epoch.ai=               | =pablo@epoch.ai=                   | epoch               |
| Profile 3      | =pablo@tlon.team=              | =tlon.shared@gmail.com=            | tlon                |

When opening Chrome manually for browser-only service flows, use =Default= for personal projects and =Profile 2= for Epoch/work projects. For example, open Epoch Google Cloud pages with =/Applications/Google Chrome.app/Contents/MacOS/Google Chrome --profile-directory="Profile 2" URL=.

The Google column is the account each Chrome profile is signed into (independent of Anthropic). The Anthropic identity column is the account used to authenticate the Claude Code CLI inside =~/.claude-<account>/= — each =CLAUDE_CONFIG_DIR= holds its own OAuth credentials for a distinct Anthropic account, so =/usage= and rate-limit tracking report each account separately.

*Setup:* make sure each target Chrome profile has the Claude extension installed and is signed into =claude.ai= using the Anthropic identity the table above assigns to that profile. That's the whole setup — no CLI-side pairing step is needed.

*How profile selection works:* when a Claude Code session starts with =--chrome=, it opens a websocket to =wss://bridge.claudeusercontent.com=. Each Claude extension instance in Chrome also connects to that bridge. The bridge matches a CLI session to an extension instance by shared Anthropic identity, so a session authenticated as =tlon.shared@gmail.com= automatically finds the Chrome extension signed into the same account (i.e. the one in Profile 3). No per-device pair record is required.

You may see =chromeExtension.pairedDeviceId= and =pairedDeviceName= populated in some accounts' =.claude.json= (e.g. epoch's =Epoch= device). These are leftovers from an older CLI flow and are not read by current Chrome integration — they can be safely ignored or deleted. The =hooks/check-chrome-profile-pair.sh= SessionStart hook used to warn when this field was missing; it's now a no-op placeholder because the warning was based on a wrong assumption.

*Native host manifest:* there is a single manifest file shared across all Chrome profiles. Its =path= field should point to the current account's =chrome-native-host= script (e.g. =~/.claude-epoch/chrome/chrome-native-host=). In practice, all account scripts are identical (they exec the same =claude= binary with =--chrome-native-host=), so the path doesn't affect functionality today — but it should be kept in sync for correctness.

*Emacs-specific:* non-interactive sessions (Emacs) require the =--chrome= flag, set via =(setq claude-code-program-switches '("--chrome"))=. Without it, Claude Code silently skips Chrome MCP startup.

*Known issues:*
- *Silent bridge failure:* the Chrome bridge can fail to connect at session startup (e.g. due to stale sockets or timing issues). When this happens, =/chrome= shows "Status: Disabled" even though =--chrome= was passed. The only fix is starting a new session. There is no mid-session reconnect mechanism.
- *Stale bridge sockets:* old sockets in =/tmp/claude-mcp-browser-bridge-pablostafforini/= are not automatically cleaned up and may interfere with new connections.
- *Session-account binding:* if you switch accounts mid-session (e.g. via =agents-claude-select-account=), Chrome will still use the original account. The running process retains its =CLAUDE_CONFIG_DIR=; you must start a fresh session under the new account — use =/handoff= to preserve context if needed.

*Architecture:*
- Chrome extension (ID =fcoeoabgfenejglbffodgkkbkcdhcgfn=) communicates via native messaging
- Native host manifest: =~/Library/Application Support/Google/Chrome/NativeMessagingHosts/com.anthropic.claude_code_browser_extension.json=
- Native host scripts: =~/.claude-{epoch,tlon}/chrome/chrome-native-host= (each execs the installed =claude= binary with =--chrome-native-host=)
- Bridge socket: =/tmp/claude-mcp-browser-bridge-pablostafforini/<pid>.sock=
- Keychain credentials per account: =Claude Code-credentials-<sha256(config_dir)[:8]>=

*** CLAUDE.md
:PROPERTIES:
:ID:       9B404E8C-F109-4188-9F3E-1F1A9A64A37D
:END:

The global =~/.claude/CLAUDE.md= (symlinked here) is the user-level memory loaded in every project. It defines conventions that apply across all projects:

- Never present guesses as facts; state uncertainty and verify claims that matter.
- Fix root causes, not symptoms; no silent fallbacks or workaround code unless explicitly labeled, justified, and approved.
- Verify fixes end-to-end before declaring them done; say exactly what could not be verified.
- Commit each logical change immediately; prefer amending when iterating on the same logical change (unless already pushed).
- Pre-commit checklist (hook-enforced): Elisp pre-commit checks (batch-test, stage =doc/*.org=) are in the =elisp-conventions= skill; the =README.org= staging rule is in the project =CLAUDE.md=.
- Use subagents and teams liberally; prefer the most powerful model.
- Use =trash= instead of =rm -rf=; never send signals to Emacs without confirmation.
- Anything the user is asked to paste — message, credential, URL, snippet — goes to the Emacs kill ring via =emacsclient= (and =pbcopy= as well when the paste target is a Chrome or native form). Multi-line or secret content uses the temp-file =insert-file-contents= → =kill-new= pattern to avoid Elisp string-escaping.
- For =/schedule=, always set email as the notification method.
- =git push= to the user's own remotes is allowed autonomously (once the underlying commits are in); other externally-visible actions (PRs, issues, Slack, email, Asana task creation/updates) still require explicit confirmation.
- Secrets handling is imported from =context/secrets.md= with =@context/secrets.md= so the short safety rules load with the global memory.
- MCP server instructions are referenced via =context/mcp-servers.md=, not imported, so agents load them only when MCP-specific context is needed.
- Service access tools are listed by service in the global memory.

Other agents do not consume this file directly. Counterpart instruction files
are documented in =../agents/README.org=, and =bin/ai-config-sync= audits
semantic equivalence after tool-specific normalization.

** Skills
:PROPERTIES:
:ID:       05017294-32AE-4465-9254-13C1BA5662CA
:END:

Skills are reusable prompt modules stored in =skills/<name>/SKILL.md=. The filename must be uppercase =SKILL.md=, since =file-expand-wildcards= is case-sensitive on the Google Drive FUSE mount. Some skills are user-invocable (triggered by typing =/<name>=), others provide context that Claude loads automatically when relevant.

Skill frontmatter descriptions should stay concise and routing-focused: include
the user phrasing, scope boundary, and important near-miss exclusions needed to
decide whether to load the skill, but keep implementation details in the skill
body.

The tracked Claude interactive skill bodies live under =claude/skills/=. The
live Claude skill root, =~/.claude/skills=, is a symlink to that tracked tree,
created by =bin/skill-prune install= or =claude/bin/install-skills.sh=.

Programmatic-only skills are not available in ordinary Claude Code sessions.
They live in =programmatic-skills/= directories so =agents-run-skill= can find
and run them explicitly from Emacs without adding them to every interactive
Claude session. The installer exposes those directories at
=~/.claude/programmatic-skills= because =agents-run-skill= discovers Claude
skills from that live path.

Run =bin/skill-prune apply --days 60= to disable stale paired skills. The scan
checks recent Claude and Codex transcripts and disables a skill only when both
tools lack usage inside the threshold window; recent use in either tool keeps
both tools enabled. It includes global paired skills and dynamically discovered
project-local skill pairs under =~/My Drive=; override the discovery roots with
=SKILL_PRUNE_LOCAL_ROOTS=, using the platform path separator for multiple roots.
Disabled skills are recorded in =skills-disabled.json= and applied to Claude as
=skillOverrides= entries set to ="off" in the live
=~/.claude/settings.local.json=. Re-enable a skill with
=bin/skill-prune enable NAME=; add =--keep= when a renamed or otherwise
intentionally retained skill should remain enabled despite missing history.

Project-local skills live with the project or workspace that owns their
workflow instead of this global dotfiles tree:

- Dotfiles maintenance: =~/My Drive/dotfiles/.claude/skills/=.
- Emacs-only maintenance: =~/My Drive/dotfiles/emacs/.claude/skills/=.
- Notes workflows: =~/My Drive/notes/.claude/skills/=.
- Batch-only notes workflows: the matching =programmatic-skills= roots under
  =~/My Drive/notes/.claude/=.
- Google Drive cleanup: =~/My Drive/.claude/skills/=.
- Epoch-only workflows: =~/My Drive/Epoch/.claude/skills/=.

Each project-local skill root documents its skills in that project's
=README.org= under a dedicated =Skills= section.

**** dotfiles-context
:PROPERTIES:
:ID:       3AB926D5-2116-4864-93F2-B6D33CE832C5

States where dotfiles and external Emacs packages should be edited: all non-=dotfiles= elpaca packages are edited in the active profile's =~/.config/emacs-profiles/<profile>/elpaca/sources/<package>/= clone, while =dotfiles= is edited only at =~/My Drive/dotfiles/= and the elpaca clone is a read-only mirror synced by git hooks. **** elisp-conventions
:PROPERTIES:
:ID:       EBC619A2-2402-43B0-8FBA-8A68EE29BB7D
:END:

Emacs Lisp coding standards: atomic functions, helper placement, docstring format. Defines the skill boundary for =.el= work, routes compiler/checkdoc lint requests to =lint-elisp= and documentation generation to =doc-elisp=, and documents the elpaca rebuild-wait-reload process enforced by the =load-elisp-after-edit= hook. Prohibits ad-hoc reload mechanisms (=load-file=, =eval-buffer=, =eval-defun=), including after shell edits that bypass edit hooks; those continue through batch-test, commit-triggered sync/rebuild, and targeted live =emacsclient -e= verification. Also documents the critical =emacs --batch= testing pattern: elpaca builds on load-path first, then =push= =emacs/extras= to the front so the edited source always loads instead of the stale =.elc= from elpaca builds. Includes pre-commit checks (batch-test and doc staging) and a transient menus section: after adding or modifying a =transient-define-prefix=, every suffix must be verified as an interactive command (transient defers validation to invocation time).

:PROPERTIES:

Audits code for actual defects: bugs, application-code security issues, error handling gaps, and correctness problems. :END:

Audits code for architectural quality: unnecessary complexity, missing or wrong abstractions, duplication, and refactoring opportunities. Pass =--accept= to auto-refactor high-confidence, behavior-preserving high-impact findings.

**** interpretability-audit
:PROPERTIES:
:ID:       800B3CAA-A993-4929-BB4A-7467A56FA70C
:END:

Audits code for reader comprehension: unclear names, magic values, missing
rationale, undocumented conventions, implicit phases, opaque score formulas,
duplicated explanatory logic, and type-safety escapes that obscure intent. Pass
=--accept= to apply high-confidence, behavior-preserving clarity fixes and
report risky renames, broad refactors, public API changes, or domain-judgment
calls instead of applying them.

**** dx-audit
:PROPERTIES:
:ID:       B2E4A6C8-1D3F-4E9A-8B5C-7F2D9A0E4C16
:END:

Audits a codebase from the AI agent's perspective: inventories existing verification infrastructure (tests, types, linting, CI), project instructions, code navigability, and automation, then produces prioritized recommendations for reducing friction. Each recommendation is concrete, justified from the AI's perspective, evidence-backed, and effort-estimated. Pass =--accept= to implement only high-confidence quick wins that are scoped, low-risk, and directly improve future agent orientation or verification.

**** pr-audit
:PROPERTIES:
:ID:       03187E6C-B3E2-47EB-821B-68121F4F883F
:END:

PR readiness verification for the current branch. It discovers the base/PR
scope, runs project build/test checks, reviews each commit, checks upstream
freshness, scans changed files for artifacts, secrets, whitespace, and line
endings, and polls remote CI only when the branch or PR is already pushed. It
reports not-run checks with concrete reasons and does not change remote state.

**** symptom-check
:PROPERTIES:
:ID:       CC534E25-3A8B-495E-A6BD-7FB05B4721F3
:END:

Forces an invariant-focused root-cause check before applying any non-trivial bug
fix, after enough debugging has identified a tempting local patch. It prevents
the failure mode where agents patch the immediate symptom while the underlying
invariant stays broken, and it explicitly avoids trivial typo/formatting fixes
and greenfield feature work. Workflow: restate the symptom, consult
=architectural-issues.md= at the repository root for similar prior entries,
draft the local fix, articulate the invariant in one sentence, search for other
manifestations with code evidence, then pick a disposition (=fix locally= /
=fix locally + flag= / =stop and refactor=). The "+ flag" and "stop and
refactor" dispositions produce a registry entry for review, or add it directly
only when registry updates were explicitly authorized; the workflow then reports
the reproduction, test, lint, build, or re-read verification that was performed.

:END:

Adapted from the third-party skill previously stored under the dotfiles

*** block-sensitive-read.sh (PreToolUse)
:PROPERTIES:
:ID:       8B4D91E7-7EA8-44FC-83E9-FD92097997BA
:END:

Guards against accidental ingestion of sensitive files into the context window. Fires on =Read= AND =Bash=, checking either the target =file_path= or the Bash command for the same sensitive-path set: =~/.password-store/=, =.zshenv-secrets=, =~/.ssh/id_*= (private keys), =~/.gnupg/=, =tokens.json= inside =~/.config/=, =~/.gmail-mcp-epoch/credentials/=, and =secret.json= / =client_secret*.json= inside =~/.config/=. The Bash branch allowlists operations that legitimately mention these paths without exposing content: auth-aware tools (=pass=, =op=, =git-crypt=, =security=), metadata commands (=ls=, =stat=, =file=, =basename=, =dirname=, =realpath=), =wc= with safe flags, =grep=/=rg=/=ripgrep= with non-content flags (=-l=, =-c=, =-q=, =-L=), and existence tests (=[ -f=, =[[ -f=, =test -f=). Anything else that touches a sensitive path is treated as content-extraction and escalated. Uses =permissionDecision: "ask"= to escalate to the user rather than hard-denying. Closes the structural gap that allowed e.g. =grep ANNAS .zshenv-secrets= to dump the secret value into the transcript via Bash, bypassing the original Read-only matcher.
