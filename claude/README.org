* Overview
:PROPERTIES:
:ID:       FC7EF2FA-CDB3-43C0-9EFC-545DEA807191
:END:

This directory contains my [[https://docs.anthropic.com/en/docs/claude-code][Claude Code]] configuration: settings, hooks, and custom skills. Claude Code is a CLI-based coding agent; I use it exclusively from the terminal. Indeed, these days I hardly do anything other than using Claude Code.

The =~/.claude/= directory symlinks its core files here, so changes are version-controlled:

- =~/.claude/settings.json= → =claude/settings.json=
- =~/.claude/settings.local.json= → =claude/settings.local.json=
- =~/.claude/CLAUDE.md= → =claude/CLAUDE.md=
- =~/.claude/skills/= → =claude/skills/=

** Multi-account setup
:PROPERTIES:
:CUSTOM_ID: h:multi-account
:END:

Two OAuth accounts (personal and work) are supported without re-authenticating. Each account uses a separate =CLAUDE_CONFIG_DIR= (=~/.claude-personal/= and =~/.claude-work/=) to keep its OAuth credentials and identity isolated, while sharing everything else via symlinks back to =~/.claude/=.

Shared (symlinked into each account dir):
- =CLAUDE.md=, =settings.json=, =settings.local.json= — instructions and preferences
- =skills/= — custom skills
- =memory/= — persistent memory
- =projects/= — trusted folders, project config, session logs
- =history.jsonl= — command history
- =plugins/= — installed plugins

Separate per account:
- =.claude.json= — OAuth identity, user ID, and MCP servers (synced from =~/.claude.json=)
- Keychain credentials — the actual OAuth tokens

*MCP servers:* Claude Code reads =.claude.json= from =$CLAUDE_CONFIG_DIR= (not =$HOME=) when that variable is set. This means user-level MCP servers must be synced to each account directory. The canonical source of truth is =~/.claude.json=; run =claude/bin/sync-mcp-servers.sh= after adding or removing any MCP server there.

From Emacs, =claude-code-extras-start-or-switch= uses the persisted account automatically (configured via =claude-code-extras-accounts=, persisted in =~/.claude-current-account=). Use =claude-code-extras-select-account= to change the active account. From the terminal, the shell aliases =claude-personal= and =claude-work= are available.

Project-level instructions referencing the =dotfiles-context= skill. The global =~/.claude/CLAUDE.md= (symlinked here) defines conventions that apply across all projects:

*** twitter

Reads and searches Twitter/X content via the =twitterapi-io= MCP server. Supports single tweet lookup, thread/reply retrieval, user timelines, keyword search, and user profile lookup. *** twitter-digest
:PROPERTIES:
:ID:       F677671C-6C97-4BC6-B2F5-A4E0E2457C92
:END:

Fetches recent tweets from a named list of accounts, triages them against the list's description (used as a relevance rubric), and surfaces only notable content. Uses a shell script (=fetch-tweets.sh=) that curls the twitterapi.io REST API directly and extracts only the fields needed for triage (~130 chars/tweet), bypassing MCP tools whose raw JSON responses (~120K chars/account) would consume enormous context. Lists are stored in =~/.claude/skills/twitter-digest/lists/= and track the last-run timestamp to avoid re-surfacing old tweets. Digests are saved to =digests/= for archival. Discovered accounts (surfaced via retweets) are automatically vetted using the shared =twitter-vet= procedure and auto-added to the list if they score >= 7. Commits all changed files (digests, last-run timestamps, list updates, vet registry) at the end. Pairs with =/loop= for periodic digests.

*** twitter-vet
:PROPERTIES:
:ID:       A7B3E1D4-5F92-4C8A-B6E0-9D2F4A1C7E53
:END:

Shared account vetting procedure used by =twitter-digest= and =twitter-discover=. Defines a two-phase evaluation: a free quick filter (follower bounds, bio relevance, tweet quality) and a full scoring pass (fetch 20 tweets, score 1-10 against the list's rubric). Accounts scoring above the threshold (default 7, overridable via =vet-threshold= frontmatter) are auto-added to the list; borderline scores are reported; low scores are skipped silently. Commits changed files (list updates, vet registry) when run standalone; defers to the caller's commit step when invoked from =twitter-digest=.

*** twitter-discover
:PROPERTIES:
:ID:       D3FF4B1E-429E-4999-951C-FBFB30F1E488
:END:

Iterative, graph-based discovery of high-value Twitter accounts in any niche. Interviews the user for criteria, acquires seed accounts via search, then expands using a "hot promise score" algorithm: accounts followed by multiple high-scoring seeds are profiled next, with scores recomputed after each round. Presents tiered results (must-follow, strong, situational) at checkpoints and optionally exports to a =twitter-digest= list.

*** slack-digest
:PROPERTIES:
:ID:       DF40872F-2CA3-4CAD-99F9-E684C175F772
:END:

Pulls all unread Slack messages via the =slack-epochai= MCP server, triages them into "action required" vs. "worth knowing" (discarding noise), and opens the digest as an org-mode buffer in Emacs with deep links back to Slack. Accepts =--mark-read= or =--no-mark-read= to control whether conversations are marked as read; if neither is passed, asks interactively.

* Hooks
:PROPERTIES:
:ID:       4D862953-7E8A-4154-BC16-919ED6B092A5
:END:

Seven hook scripts handle workflow invariants and Emacs integration.

** require-elisp-test-before-commit.sh (PreToolUse)
:PROPERTIES:
:ID:       4E0CDC5D-C436-4701-AC99-E29F950318B4
:END:

Blocks =git commit= when Elisp files (=*.el= or =emacs/config.org=) are staged but no test has been run in the current session. Forces the agent to verify changes before committing. Also detects the case where =git add= and =git commit= appear in the same bash command (which would otherwise bypass the staged-file check by running before staging occurs).

** track-elisp-test.sh (PostToolUse)
:PROPERTIES:
:ID:       85C6B926-9F3F-431A-9650-F504CEE12B4A
:END:

Creates a marker file (=/tmp/claude-elisp-tested-{SESSION_ID}=) whenever =emacs --batch= runs. This marker is what the pre-commit hook checks for. The marker is consumed (deleted) after a successful commit, so each commit requires a fresh test.

** symlink-project-memory.sh (PostToolUse)
:PROPERTIES:
:ID:       086B58F4-949F-4C88-B855-687EAD5F3B63
:END:

On the first Edit or Write in a project, moves Claude Code's memory directory from its default location (=~/.claude/projects/<encoded-path>/memory/=) into the project's =.claude/memory/= directory, replacing the original with a symlink. This keeps memory local and version-controllable.

** require-readme-update.sh (PreToolUse)
:PROPERTIES:
:ID:       F4D78958-4194-4DB3-9624-C04EC7790960
:END:

Blocks =git commit= when files in =claude/= (skills, hooks, settings, CLAUDE.md) are staged but =claude/README.org= is not. Also detects when =git add= and =git commit= appear in the same bash command (which would otherwise bypass the staged-file check). Ensures documentation stays in sync with configuration changes. Project-specific skills (under =.claude/skills/=) are excluded from this check, since they belong to their project's documentation, not the dotfiles README.

** require-doc-update.sh (PreToolUse)
:PROPERTIES:
:CUSTOM_ID: h:require-doc-update
:END:

Blocks =git commit= when =.el= files are staged but no =doc/*.org= or =README.org= manual update is included. Applies only in repos that have a =doc/= directory or a =README.org= at the root. Ensures documentation stays in sync with code changes.

** load-elisp-after-edit.sh (PostToolUse)
:PROPERTIES:
:CUSTOM_ID: h:load-elisp-after-edit
:END:

Fires on every Edit or Write of an =.el= file under =elpaca/sources/=. Identifies the elpaca package (by matching the file's basename against package names for extras, or by source directory prefix for standalone packages), then runs =elpaca-rebuild= + =elpaca-wait= + =elpaca-extras-reload= synchronously via =emacsclient=. This byte-compiles the source into =elpaca/builds/=, waits for completion, and reloads all features — so the running Emacs always has the latest byte-compiled code, matching what the user gets after a restart. Skips byte-compiled files and test files.

** elpaca-run.sh (utility)
:PROPERTIES:
:ID:       54F62C33-E882-4B75-A4F0-A8963E5AF345
:END:

Resolves the active elpaca profile dynamically (via =emacsclient -e 'init-current-profile'=) and runs a script from the specified elpaca package. Searches both =repos/= and =sources/= directories to handle different elpaca layouts. Used by the Notification and Stop hooks to invoke =claude-code-hook-wrapper= without hardcoding the profile name. The =emacsclient= call has a 5-second timeout to prevent hangs when the Emacs server is unresponsive.

** claude-code-hook-wrapper (Notification, Stop)
:PROPERTIES:
:ID:       44E20BF0-9B6B-420E-858F-98CBF18D13F8
:END:

Invoked via =elpaca-run.sh= on Notification and Stop events. Delegates to the =claude-code= Emacs package's hook wrapper, which updates the Emacs-side state (e.g. mode-line indicators) in response to Claude Code lifecycle events.
