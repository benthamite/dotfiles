#+title: Codex configuration
#+author: Pablo Stafforini

* Overview

This directory contains the dotfiles source of truth for the parts of my
[[https://developers.openai.com/codex/][Codex]] setup that are version-controlled.

The corresponding Claude Code setup remains the primary source of truth for
shared behavioral instructions.  Codex should consume those instructions
directly where it can, and use generated or wrapper files only where Codex's
discovery rules require Codex-specific filenames.

* What ports cleanly

- Project instructions via repository =CLAUDE.md= files, because
  =codex/config.toml= sets =project_doc_fallback_filenames = ["CLAUDE.md"]=.
- Repository =AGENTS.md= files also work when a project already uses that
  convention, but they are not required for my own repositories.
- Plain =SKILL.md= skills: the shared subset of Claude/Codex skills can be
  reused with thin registration or symlink layers.
- Trusted-project settings, model defaults, theme, and rules via
  =codex/config.toml= and =codex/rules/default.rules=.

* What ports partially

- Global instructions: Codex's global discovery reads only
  =~/.codex/AGENTS.override.md= or =~/.codex/AGENTS.md=.  Fallback filenames
  apply to project documents, not the Codex home directory.  If global Codex
  instructions must exactly mirror Claude, generate a flattened
  =~/.codex/AGENTS.md= from =claude/CLAUDE.md= rather than maintaining a
  hand-written fork.
- File imports: Claude expands =@path/to/file= imports in =CLAUDE.md= at
  session startup.  Codex =AGENTS.md= files are plain Markdown; Codex does not
  document support for =@= import expansion in instruction files.  Any
  instructions Codex must always see should be self-contained or generated into
  a flattened file.
- Hooks: Codex hooks are experimental and currently support =SessionStart=,
  =PreToolUse=, =PermissionRequest=, =PostToolUse=, =UserPromptSubmit=, and
  =Stop=.  =PreToolUse= and =PostToolUse= can match =Bash=, =apply_patch=,
  MCP tools, and the =Edit|Write= aliases, but the payload shape and available
  events differ from Claude's.  Shared hook scripts live under =claude/hooks/=
  and Codex calls them through =codex/hooks/run-claude-hook.sh=, which
  normalizes Codex hook JSON into the Claude-shaped fields the scripts expect.
- Skills with Claude-only frontmatter or runtime assumptions need Codex
  overlays.  Examples include =allowed-tools=, subagent execution, dynamic
  context injection, Claude environment variables, Chrome assumptions, and
  references to Claude-specific tool names.
- MCP server definitions use the same protocol but different config files.
  Treat the server inventory as shared policy and generate per-tool config if
  user-level MCPs return.
- Subagents are conceptually portable but not file-format portable.  Claude
  uses Markdown/YAML agent definitions; Codex custom agents are TOML config
  layers.
- Custom status line: no direct Codex equivalent.
- Claude project/global memory relocation: no stable Codex equivalent yet.
- Claude's allow-list permissions: Codex uses sandbox settings, approval
  policy, and prefix rules instead.

* Active port in this repo

- =project_doc_fallback_filenames = ["CLAUDE.md"]= lets Codex keep reading my
  existing =CLAUDE.md= files without migrating the source of truth to
  =AGENTS.md=.
- =.codex/hooks.json= ports the Bash-based commit hooks and the stop hook for
  this repository by calling shared =claude/hooks/= scripts through
  =codex/hooks/run-claude-hook.sh=.
- =codex/skills/= currently contains only the two Codex-specific ports needed
  for this repository: =dotfiles-context= and =elisp-conventions=.
- The remaining Claude hooks are not ported yet.  Edit/write-oriented hooks
  such as =block-elpaca-dotfiles-edit.sh=, =load-elisp-after-edit.sh=, and
  =remind-claude-readme-update.sh= should be re-evaluated against Codex's
  current =Edit|Write= hook matching before deciding whether they need
  wrappers, simplification, or should remain Claude-only.

* Global activation

To make global Codex instructions apply everywhere, symlink =~/.codex/AGENTS.md=
to =~/My Drive/dotfiles/codex/AGENTS.md= only if that file is kept as a real
Codex-readable entrypoint.  A bare pointer to =claude/CLAUDE.md= is useful for
humans but is not a guaranteed import mechanism.

If I later want a global hook layer, I can also symlink =~/.codex/hooks.json=
to a shared hooks file. Until then, the repository-local =.codex/hooks.json=
only affects this repository.

* Synchronization plan

- Keep =claude/CLAUDE.md= as the human-edited source of truth for shared
  behavioral instructions.
- Keep project =CLAUDE.md= files self-contained when Codex must consume them.
  If a =CLAUDE.md= uses Claude-only =@= imports, generate a flattened Codex
  instruction file or move that content into a portable skill.
- Classify Claude skills as =portable=, =codex-overlay-needed=, or
  =claude-only= before registering them for Codex.
- Share hook scripts where practical, but keep hook registration and JSON input
  adapters tool-specific.
- Add an audit command that reports unclassified Claude skills, divergent
  shared skill files, stale Codex overlays, and hook references to missing
  scripts.
