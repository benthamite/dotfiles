This manual describes the features and customization options for the Emacs Lisp file =pass-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=pass-extras.el= extends the =pass= package, which provides an Emacs interface to the Unix =pass= password manager. While =pass= already offers browsing and copying of password entries, this package adds several capabilities that streamline common workflows:

- Opening the URL associated with a password entry while simultaneously copying the password to the clipboard ([[#h:browsing-entries][Browsing entries]]).
- Generating passwords of various types and copying them to the clipboard with automatic clearing ([[#h:generating-passwords][Generating passwords]]).
- Storing external key files into the password store and exporting entries back out ([[#h:key-management][Key management]]).
- Unlocking =git-crypt= repositories using keys stored in =pass= ([[#h:git-crypt-integration][Git-crypt integration]]).

The package depends on =pass= and =password-generator=.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Browsing entries
:PROPERTIES:
:CUSTOM_ID: h:browsing-entries
:END:

#+findex: pass-extras-open-at-point
When you are reviewing your password entries in a =pass-mode= buffer, you may want to log in to the associated website. The command ~pass-extras-open-at-point~ combines two steps into one: it copies the password of the entry at point to the kill ring (via ~pass-copy~) and then opens the URL stored in the entry's =password-store-url-field= in your default browser. If the entry does not contain a URL field, the command signals an error.

#+findex: pass-extras-insert-generated-no-symbols
The command ~pass-extras-insert-generated-no-symbols~ creates a new password store entry with a randomly generated password that contains only alphanumeric characters (no special symbols). It delegates to ~password-store-generate-no-symbols~ for the actual generation and then refreshes the =pass-mode= buffer so the new entry appears immediately.

** Generating passwords
:PROPERTIES:
:CUSTOM_ID: h:generating-passwords
:END:

#+findex: pass-extras-generate-password
Sometimes you need a password without creating a new entry in the store, for instance when filling a form in a browser or registering for a service you do not intend to save. The command ~pass-extras-generate-password~ prompts you to choose a password type from several categories provided by the =password-generator= package:

- Simple (alphanumeric)
- Strong (with special characters)
- Numeric (PIN)
- Phonetic (memorable)
- Words (correct horse battery style)
- Custom (your own alphabet)

The generated password is copied to the kill ring and will be automatically cleared after the number of seconds specified by ~password-store-time-before-clipboard-restore~, just like a regular =pass= copy operation. This ensures that generated passwords do not linger in the clipboard.

** Key management
:PROPERTIES:
:CUSTOM_ID: h:key-management
:END:

#+findex: pass-extras-store-key
#+findex: pass-extras-export-key
Cryptographic key files (SSH keys, GPG keys, API tokens stored in files) can be imported into the password store for safekeeping. The command ~pass-extras-store-key~ prompts for a key file and a target entry name, then pipes the file contents into =pass insert --multiline= to store them as a multiline entry. Completion is offered for existing entry names.

The reverse operation is provided by ~pass-extras-export-key~, which prompts for an existing entry (with completion requiring a match) and a destination file path, then writes the entry contents to that file. This is useful when you need to restore a key file on a new machine.

** Git-crypt integration
:PROPERTIES:
:CUSTOM_ID: h:git-crypt-integration
:END:

#+findex: pass-extras-git-crypt-unlock
If you store =git-crypt= symmetric keys in =pass=, the command ~pass-extras-git-crypt-unlock~ can unlock a repository without ever writing the key to disk. It prompts for a =pass= entry containing the key (with completion) and runs =git-crypt unlock= with process substitution so the key is fed directly from =pass= into =git-crypt=. The command operates in the current ~default-directory~ unless an optional REPO argument is provided programmatically. It reports success or failure in the echo area.
