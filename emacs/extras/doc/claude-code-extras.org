This manual describes the features and customization options for the Emacs Lisp file =claude-code-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=claude-code-extras.el= extends =claude-code=, the Emacs package that runs Claude Code sessions inside an Eat terminal buffer. While =claude-code= provides the core session management and terminal integration, this extras package layers on several quality-of-life features that make extended Claude sessions more manageable.

The package provides eight main areas of functionality:

- *Session switching.* A Consult-powered interface for starting new Claude sessions or switching between existing ones, with live buffer previews ([[#h:session-switching][Session switching]]).
- *Status polling.* Integration with a statusline script that exports session metadata (model name, cost, context usage, token counts) as JSON, polled periodically and exposed through accessor functions ([[#h:status-polling][Status polling]]).
- *Alerts.* Configurable OS notifications (visual, audible, or both) when Claude needs attention, triggered by the CLI's Notification hook event for idle prompts, permission requests, and elicitation dialogs ([[#h:alerts][Alerts]]).
- *Buffer lifecycle.* Kill protection for active sessions, automatic buffer cleanup on process exit, a SIGWINCH workaround for terminal rendering glitches, and a fix for ~C-g~ escape handling with multiple sessions ([[#h:buffer-lifecycle][Buffer lifecycle]]).
- *Snippet insertion.* An adapter for ~consult-yasnippet~ that bypasses its direct buffer manipulation (which fails in Eat terminal buffers) and instead sends expanded snippet text through the terminal emulator ([[#h:snippet-insertion][Snippet insertion]]).
- *Copilot integration.* An adapter that enables Copilot ghost-text prose completions while typing prompts in Claude Code buffers, routing accepted text through the terminal emulator ([[#h:copilot-integration][Copilot integration]]).
- *Skill runner.* Run any user-invocable Claude Code skill non-interactively from Emacs with completion, argument prompting, and a result buffer ([[#h:skill-runner][Skill runner]]).
- *Batch TODO processing.* Sequential execution of org TODO entries via =claude -p= in non-interactive mode, with logged results and a summary buffer ([[#h:batch-todo-processing][Batch TODO processing]]).
- *Project audit.* One-command comprehensive audit that sequentially runs code, design, and interpretability audit skills with auto-fix on the current project ([[#h:project-audit][Project audit]]).

The package depends on =claude-code=, =consult= (for the session switcher), and =paths= (for the batch log directory default).

* User options
:PROPERTIES:
:CUSTOM_ID: h:user-options
:END:

** Buffer protection
:PROPERTIES:
:CUSTOM_ID: h:buffer-protection-option
:END:

#+vindex: claude-code-extras-protect-buffers
The user option ~claude-code-extras-protect-buffers~ controls whether killing a Claude Code buffer requires confirmation. When set to non-~nil~ (the default), any attempt to kill a buffer with a live Claude process triggers a =yes-or-no-p= prompt. This prevents accidental loss of an active session. When the process has already exited (e.g. via the =/exit= command inside Claude), the prompt is skipped.

The default value is ~t~. Set it to ~nil~ if you prefer to kill Claude buffers without confirmation.

** Diff display
:PROPERTIES:
:CUSTOM_ID: h:diff-display-fix
:END:

#+findex: claude-code-extras--display-diff-buffer
The function ~claude-code-extras--display-diff-buffer~ overrides ~monet--display-diff-buffer~ to display diff buffers in a bottom side window instead of using ~display-buffer-in-tab~. The upstream implementation calls ~display-buffer-in-tab~ from an async websocket callback, which switches tabs as a side effect and can corrupt the window layout when the fallback ~display-buffer-pop-up-window~ fails and Emacs resorts to ~display-buffer-use-some-window~ (which replaces the buffer in an existing window). The override avoids tab switching entirely.

** Terminal rendering
:PROPERTIES:
:CUSTOM_ID: h:terminal-rendering-option
:END:

#+vindex: claude-code-extras-sigwinch-delay
The user option ~claude-code-extras-sigwinch-delay~ specifies the delay (in seconds) before sending a =SIGWINCH= signal to the Claude process after startup. This works around a race condition where the TUI queries terminal dimensions before the Eat terminal window is fully laid out, resulting in a garbled banner. The default value is =0.5=. Increase it if you still see rendering artifacts on slower machines.

#+findex: claude-code-extras-disable-scrollback-truncation
The function ~claude-code-extras-disable-scrollback-truncation~ sets ~eat-term-scrollback-size~ to =nil= in Claude Code buffers. Eat's default scrollback limit of 8192 lines causes early output to be silently discarded in long sessions. This function is added to ~claude-code-start-hook~ automatically.

** Theme sync
:PROPERTIES:
:CUSTOM_ID: h:theme-sync-option
:END:

#+vindex: claude-code-extras-sync-theme
The user option ~claude-code-extras-sync-theme~ controls whether the Claude Code theme is kept in sync with the current Emacs theme. When non-~nil~, changing the Emacs theme (via ~enable-theme-functions~) updates =~/.claude/settings.json= and sends ~/theme~ to all running Claude Code sessions so they switch between light and dark mode automatically. The default value is ~nil~ (disabled).

Sessions that are idle (waiting for user input) receive the ~/theme~ command immediately. Sessions where Claude is still generating a response queue the theme change and apply it automatically when the turn finishes (via the Stop hook).

The theme sync is fully asynchronous: the ~enable-theme-functions~ hook returns immediately, and all terminal interactions are deferred to zero-delay timers.  This prevents reentrant timer execution (e.g. ~org-roam-db-sync~) from blocking the main thread during ~accept-process-output~.

** Alert options
:PROPERTIES:
:CUSTOM_ID: h:alert-options
:END:

#+vindex: claude-code-extras-alert-on-ready
#+vindex: claude-code-extras-alert-style
#+vindex: claude-code-extras-alert-sound
The user option ~claude-code-extras-alert-on-ready~ controls whether the user is notified when Claude needs attention. The default is ~nil~ (no alerts). Toggle it interactively with ~claude-code-extras-toggle-alert~ ([[#h:alert-toggle][Toggling alerts]]).

The user option ~claude-code-extras-alert-style~ determines the type of notification. It accepts one of three values: ~visual~ (an OS notification via the =alert= package), ~sound~ (an audio alert), or ~both~ (the default when alerts are enabled). This option only takes effect when ~claude-code-extras-alert-on-ready~ is non-~nil~.

The user option ~claude-code-extras-alert-sound~ specifies the path to the sound file played for audible alerts. The default is =/System/Library/Sounds/Glass.aiff=, a macOS system sound. Change this to any audio file that =afplay= can handle.

** Account options
:PROPERTIES:
:CUSTOM_ID: h:account-options
:END:

#+vindex: claude-code-extras-accounts
The user option ~claude-code-extras-accounts~ is an alist of =(NAME . CONFIG-DIR)= pairs for switching between multiple Claude OAuth accounts without re-authenticating. See [[#h:multi-account][Multi-account support]] for details and configuration examples. The default is ~nil~ (single-account mode).

** Copilot options
:PROPERTIES:
:CUSTOM_ID: h:copilot-options
:END:

#+vindex: claude-code-extras-copilot-enabled
The user option ~claude-code-extras-copilot-enabled~ controls whether Copilot ghost-text completions are enabled in Claude Code buffers. The default is ~nil~ (disabled). When set to non-~nil~, ~copilot-mode~ is automatically activated in new Claude Code buffers, and an adapter routes accepted completions through the Eat terminal emulator instead of inserting them directly into the buffer. Completions are suppressed while Claude is responding (i.e. when the buffer is read-only). The =copilot= package must be installed for this feature to work; if it is not available, the setup is silently skipped.

** Batch TODO processing options
:PROPERTIES:
:CUSTOM_ID: h:batch-todo-options
:END:

#+vindex: claude-code-extras-log-directory
The user option ~claude-code-extras-log-directory~ specifies the directory where batch processing output is saved. The default value is the =claude-logs= subdirectory under ~paths-dir-notes~.

#+vindex: claude-code-extras-batch-allowed-tools
#+vindex: claude-code-extras-batch-max-turns
#+vindex: claude-code-extras-batch-system-prompt
#+vindex: claude-code-extras-batch-model
The user option ~claude-code-extras-batch-allowed-tools~ specifies which tools are auto-allowed via =--allowedTools= for non-interactive execution (batch TODO processing, skill runner, and project audit). The default includes =Bash=, =Read=, =Write=, =Edit=, =Glob=, =Grep=, =WebFetch=, and =WebSearch=. Adjust this list to restrict or expand what Claude can do autonomously.

The user option ~claude-code-extras-batch-max-turns~ controls the maximum number of agentic turns per entry in non-interactive execution. The default is 30. Increase it for complex tasks that require many tool calls; decrease it to bound execution time.

The user option ~claude-code-extras-batch-system-prompt~ specifies an optional system prompt appended via =--append-system-prompt= to each =claude -p= invocation. The default is ~nil~ (no extra system prompt).

The user option ~claude-code-extras-batch-model~ specifies an optional model override via =--model=. The default is ~nil~ (use Claude's default model).

** Project audit options
:PROPERTIES:
:CUSTOM_ID: h:project-audit-options
:END:

#+vindex: claude-code-extras-audit-skills
The user option ~claude-code-extras-audit-skills~ specifies the list of skills to run when performing a project audit via ~claude-code-extras-audit-project~. Each entry is a skill name with a leading slash. The default value is ~("/code-audit" "/design-audit" "/interpretability-audit")~. Each skill is invoked with =--accept=, which instructs it to auto-fix all findings without prompting. Customize this list to add, remove, or reorder audit passes.

#+vindex: claude-code-extras-audit-project-directories
The user option ~claude-code-extras-audit-project-directories~ specifies the directories available for selection when running ~claude-code-extras-audit-project~. When the user enters a directory not already in this list, it is automatically appended and persisted via ~customize-save-variable~, so it will appear as a candidate in future invocations.

** Debug backtrace options
:PROPERTIES:
:CUSTOM_ID: h:debug-backtrace-options
:END:

#+vindex: claude-code-extras-debug-backtrace-model
The user option ~claude-code-extras-debug-backtrace-model~ sets the gptel model used to identify the offending package from a backtrace. The default is ~gemini-flash-lite-latest~, a fast and cheap model suited for this simple classification task.

#+vindex: claude-code-extras-debug-backtrace-backend
The user option ~claude-code-extras-debug-backtrace-backend~ sets the gptel backend name for backtrace analysis. The default is ~"Gemini"~.

** Status polling options
:PROPERTIES:
:CUSTOM_ID: h:status-polling-options
:END:

#+vindex: claude-code-extras-status-interval
The user option ~claude-code-extras-status-interval~ sets the polling frequency (in seconds) for reading the status JSON file. The default is 5. Lower values give more responsive modeline updates; higher values reduce filesystem polling overhead.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Session switching
:PROPERTIES:
:CUSTOM_ID: h:session-switching
:END:

#+findex: claude-code-extras-start-or-switch
The command ~claude-code-extras-start-or-switch~ is the main entry point for working with Claude sessions. When no Claude sessions are active, it starts a new one by calling ~claude-code~. When one or more sessions exist, it opens a Consult selection interface listing all active sessions plus a "[new session]" option.

Sessions are displayed by project name (extracted from the buffer name). When multiple sessions share the same project, the instance name is appended for disambiguation (e.g. "project:instance"). The interface provides live buffer previews as you navigate the candidates, so you can see the current state of each session before switching.

*** Multi-account support
:PROPERTIES:
:CUSTOM_ID: h:multi-account
:END:

#+vindex: claude-code-extras-accounts
When ~claude-code-extras-accounts~ is non-nil, starting a new session prompts for an account before launching Claude. Each account maps to a separate =CLAUDE_CONFIG_DIR=, so OAuth credentials, settings, and memory are kept independent. This avoids the need to log out and back in when switching between accounts (e.g. personal and work).

Configure accounts as an alist of =(NAME . CONFIG-DIR)= pairs:

#+begin_src emacs-lisp
(setopt claude-code-extras-accounts
        '(("personal" . "~/.claude-personal")
          ("work"     . "~/.claude-work")))
#+end_src

The first time you start a session with a given account, Claude will prompt for OAuth login. After that, switching is instant. If only one account is configured, it is selected automatically without prompting.

** Toggling alerts
:PROPERTIES:
:CUSTOM_ID: h:alert-toggle
:END:

#+findex: claude-code-extras-toggle-alert
The command ~claude-code-extras-toggle-alert~ toggles the buffer-local value of ~claude-code-extras-alert-on-ready~ and reports the new state. This is useful during long-running operations: enable alerts before submitting a complex prompt, then disable them when you return to interactive use. The alert style and sound are configured via the user options described in [[#h:alert-options][Alert options]].

** Skill runner
:PROPERTIES:
:CUSTOM_ID: h:skill-runner
:END:

#+findex: claude-code-extras-run-skill
The command ~claude-code-extras-run-skill~ runs a Claude Code skill non-interactively via =claude -p=. Interactively, it prompts for the skill name with completion (annotated with descriptions), then for arguments if the skill declares an =argument-hint= in its frontmatter, and finally for a working directory. The result is displayed in a =*Claude Skill: <name>*= buffer in org-mode with cost, duration, and session ID.

Only skills with =user-invocable: true= (or no =user-invocable= field) in their =SKILL.md= frontmatter are offered for completion. Skills are discovered from =~/.claude/skills/= (global) and the current project's =.claude/skills/= directory. Project skills shadow global skills with the same name.

The command uses the same non-interactive execution infrastructure as batch TODO processing, respecting ~claude-code-extras-batch-allowed-tools~, ~claude-code-extras-batch-max-turns~, ~claude-code-extras-batch-system-prompt~, and ~claude-code-extras-batch-model~ ([[#h:batch-todo-options][Batch TODO processing options]]).

** Batch TODO processing
:PROPERTIES:
:CUSTOM_ID: h:batch-todo-processing
:END:

#+findex: claude-code-extras-batch-todos
The command ~claude-code-extras-batch-todos~ processes org TODO entries sequentially via =claude -p= in non-interactive mode. It must be called from an org-mode buffer. Scope is inferred automatically: =region= if the region is active, =subtree= if the buffer is narrowed, =buffer= otherwise. The command then prompts for a working directory via ~project-prompt-project-dir~. A confirmation prompt is shown unless the scope is =region= (where the explicit selection already signals intent).

Each TODO's heading becomes the prompt (with body text appended if present). Entries are processed sequentially — each =claude -p= invocation must finish before the next starts. Errors do not abort the batch; each failure is recorded and processing continues.

Results are saved as JSON log files in a timestamped subdirectory under ~claude-code-extras-log-directory~ ([[#h:batch-todo-options][Batch TODO processing options]]). When all entries have been processed, a =*Claude Batch Results*= summary buffer is displayed in org-mode with per-entry status, cost, duration, log file links, and Claude's response text.

The command respects ~claude-code-extras-batch-allowed-tools~, ~claude-code-extras-batch-max-turns~, ~claude-code-extras-batch-system-prompt~, and ~claude-code-extras-batch-model~ ([[#h:batch-todo-options][Batch TODO processing options]]).

** Project audit
:PROPERTIES:
:CUSTOM_ID: h:project-audit
:END:

#+findex: claude-code-extras-audit-project
The command ~claude-code-extras-audit-project~ runs a comprehensive audit of a project. It prompts the user to select a directory from ~claude-code-extras-audit-project-directories~ ([[#h:project-audit-options][Project audit options]]) or enter a new one. New directories are automatically added to the list for future use. It then sequentially invokes each skill in ~claude-code-extras-audit-skills~ ([[#h:project-audit-options][Project audit options]]) with =--accept=, each as a separate non-interactive =claude -p= session. This means each audit starts with a fresh context window.

After each audit completes successfully, any uncommitted changes in the project directory are automatically committed with a message of the form =<skill>: apply audit recommendations=. This ensures each audit's changes are isolated in their own commit, making them easy to review individually.

The command reuses the batch processing infrastructure ([[#h:batch-todo-internals][Batch TODO internals]]), so results are logged and displayed in the same =*Claude Batch Results*= summary buffer with per-audit status, cost, duration, and output. The batch options (~claude-code-extras-batch-max-turns~, ~claude-code-extras-batch-model~, etc.) apply to audit runs as well.

** Debug backtrace
:PROPERTIES:
:CUSTOM_ID: h:debug-backtrace
:END:

#+findex: claude-code-extras-debug-backtrace
The command ~claude-code-extras-debug-backtrace~ automates the workflow of debugging an Emacs backtrace with Claude Code. When invoked from a =*Backtrace*= buffer, it:

1. Saves the backtrace to the downloads directory via ~debug-save-backtrace~.
2. Sends the backtrace to a light LLM (configured via ~claude-code-extras-debug-backtrace-model~ and ~claude-code-extras-debug-backtrace-backend~, [[#h:debug-backtrace-options][Debug backtrace options]]) to identify the package that caused the error.
3. Starts an interactive Claude Code session in the package's source directory (elpaca source directory if available, otherwise the directory of the library file on ~load-path~).
4. Once the session is ready, sends it the backtrace file path and asks it to identify and fix the bug.

* Functions
:PROPERTIES:
:CUSTOM_ID: h:functions
:END:

** Buffer lifecycle
:PROPERTIES:
:CUSTOM_ID: h:buffer-lifecycle
:END:

#+findex: claude-code-extras-protect-buffer
#+findex: claude-code-extras-setup-kill-on-exit
#+findex: claude-code-extras-fix-rendering
The function ~claude-code-extras-protect-buffer~ is a ~kill-buffer-query-functions~ member that prompts for confirmation before killing Claude buffers with live processes. It respects the ~claude-code-extras-protect-buffers~ user option ([[#h:buffer-protection-option][Buffer protection]]).

The function ~claude-code-extras-setup-kill-on-exit~ wraps the process sentinel so that the buffer is automatically killed when the Claude process exits. This keeps the buffer list clean after normal session termination via =/exit=.

The function ~claude-code-extras-fix-rendering~ sends a =SIGWINCH= signal to the Claude process after the delay specified by ~claude-code-extras-sigwinch-delay~ ([[#h:terminal-rendering-option][Terminal rendering]]). This forces the TUI to re-query terminal dimensions and redraw correctly.

#+findex: claude-code-extras--send-escape-in-current-buffer
The function ~claude-code-extras--send-escape-in-current-buffer~ is an =:around= advice on ~claude-code-send-escape~ that fixes ~C-g~ behavior when multiple Claude sessions share the same project directory.  The upstream implementation uses ~claude-code--with-buffer~, which re-resolves the target buffer via ~claude-code--get-or-prompt-for-buffer~.  When multiple sessions exist for the same project, this triggers a selection prompt instead of sending the escape to the current buffer.  The advice short-circuits the lookup: if the current buffer is already a Claude buffer, it sends the escape sequence directly; otherwise it falls through to the original function.

** Status polling
:PROPERTIES:
:CUSTOM_ID: h:status-polling
:END:

#+findex: claude-code-extras-start-status-polling
#+findex: claude-code-extras-stop-status-polling
The function ~claude-code-extras-start-status-polling~ begins periodic reading of the JSON status file written by the bundled statusline shell script. The parsed data is stored in the buffer-local variable ~claude-code-extras--status-data~ as a plist. Calling it multiple times is safe: any existing timer is canceled before a new one is created.

The function ~claude-code-extras-stop-status-polling~ cancels the timer and deletes the status file. As a safety net, the polling callback also cancels itself if the associated buffer has been killed.

** Status accessors
:PROPERTIES:
:CUSTOM_ID: h:status-accessors
:END:

#+findex: claude-code-extras-status-model
#+findex: claude-code-extras-status-cost
#+findex: claude-code-extras-status-context-percent
#+findex: claude-code-extras-status-token-count
#+findex: claude-code-extras-status-lines-added
#+findex: claude-code-extras-status-lines-removed
#+findex: claude-code-extras-status-duration-ms
#+findex: claude-code-extras-status-cache-read-tokens
#+findex: claude-code-extras-status-cache-total-tokens
The status accessor functions extract specific values from the parsed status data. Each returns ~nil~ if the status data is unavailable or the relevant key is missing.

- ~claude-code-extras-status-model~ returns the model display name (e.g. "Claude Opus 4.6").
- ~claude-code-extras-status-cost~ returns the total session cost in USD.
- ~claude-code-extras-status-context-percent~ returns the context window usage as a percentage.
- ~claude-code-extras-status-token-count~ returns the total input token count.
- ~claude-code-extras-status-lines-added~ and ~claude-code-extras-status-lines-removed~ return line-change statistics.
- ~claude-code-extras-status-duration-ms~ returns the total session duration in milliseconds.
- ~claude-code-extras-status-cache-read-tokens~ returns the cache read input token count for the current turn.
- ~claude-code-extras-status-cache-total-tokens~ returns the sum of all input token categories (input, cache creation, and cache read) for the current turn.

These functions are designed to be called from modeline constructors (e.g. =doom-modeline-extras.el=) to display live session statistics.

** Alerts
:PROPERTIES:
:CUSTOM_ID: h:alerts
:END:

#+findex: claude-code-extras-notify
#+findex: claude-code-extras-alert-indicator
#+findex: claude-code-extras--handle-notification
The function ~claude-code-extras--handle-notification~ is a ~claude-code-event-hook~ listener that handles =Notification= events from the CLI.  It parses the notification JSON to determine the type and dispatches alerts accordingly: =idle_prompt= (Claude finished and is waiting), =permission_prompt= (a tool needs approval), and =elicitation_dialog= (Claude is showing an input dialog).  For =idle_prompt=, it also sets ~claude-code-extras--waiting-for-input~ to ~t~.

The function ~claude-code-extras-notify~ combines the default modeline pulse with optional OS alerts.  When ~claude-code-extras-alert-on-ready~ is non-~nil~, it dispatches visual and/or audible alerts according to ~claude-code-extras-alert-style~ ([[#h:alert-options][Alert options]]).  It is called by the notification handler rather than installed directly as ~claude-code-notification-function~ (which is set to the default modeline-only callback to avoid double-alerting from the terminal bell).

The function ~claude-code-extras-alert-indicator~ returns a bell icon string reflecting the current alert state. It is intended for use in modeline segments or header lines to give a visual indication of whether alerts are active.

** Modeline integration
:PROPERTIES:
:CUSTOM_ID: h:modeline-integration
:END:

#+findex: claude-code-extras-set-modeline
The function ~claude-code-extras-set-modeline~ activates the =claude-code= doom-modeline segment for the current buffer. It also starts status polling if it is not already active ([[#h:status-polling][Status polling]]). This function is added to ~claude-code-start-hook~ automatically.

** Snippet insertion
:PROPERTIES:
:CUSTOM_ID: h:snippet-insertion
:END:

#+findex: claude-code-extras--consult-yasnippet
#+findex: claude-code-extras--expand-snippet-to-text
The function ~claude-code-extras--consult-yasnippet~ is an =:around= advice on ~consult-yasnippet~ that makes snippet insertion work in Eat terminal buffers (such as Claude Code sessions).  The upstream ~consult-yasnippet~ captures buffer positions at invocation time and manipulates them on return (via ~delete-region~, ~insert~, and ~goto-char~) for preview cleanup.  In an Eat buffer the terminal emulator modifies the buffer asynchronously, so those positions become stale and cause =args-out-of-range= errors.

The advice detects ~eat-mode~ and replaces the entire flow: it calls ~consult-yasnippet--candidates~ and ~consult--read~ without a =:state= function (no preview or buffer manipulation), then expands the selected template to plain text via ~claude-code-extras--expand-snippet-to-text~, which creates a temporary buffer, enables ~yas-minor-mode~, expands the snippet, commits all active snippet fields to their defaults, and returns the resulting string.  Newlines in the expanded text are converted to Alt+Enter sequences (~ESC CR~) so that Claude Code's TUI treats them as multi-line input.  The final text is sent through ~eat-term-send-string~.

In non-Eat buffers the advice delegates to the original function unchanged.

#+findex: claude-code-extras--try-expand-snippet-at-prompt
#+findex: claude-code-extras-snippet-tab
#+findex: claude-code-extras-setup-snippet-keys
The function ~claude-code-extras--try-expand-snippet-at-prompt~ enables TAB-triggered snippet expansion at the terminal prompt.  It searches backward from ~point-max~ for the =❯= prompt marker, extracts the user's input, and checks whether it ends with a yasnippet key.  When the input matches, the function sends DEL characters to erase the key from the terminal, then sends the expanded snippet text through ~eat-term-send-string~.  When multiple keys match, the longest one wins.

The minor mode ~claude-code-extras--snippet-keys-mode~ overrides TAB in Claude Code buffers to call ~claude-code-extras-snippet-tab~, which tries snippet expansion first and falls through to ~eat-self-input~ otherwise.  The function ~claude-code-extras-setup-snippet-keys~ activates this mode from ~claude-code-start-hook~ when Copilot integration is disabled.  When Copilot is enabled, the Copilot TAB handler (~claude-code-extras--copilot-tab~) chains through snippet expansion before falling through to eat, so a separate mode is not needed.

** Copilot integration
:PROPERTIES:
:CUSTOM_ID: h:copilot-integration
:END:

#+findex: claude-code-extras-setup-copilot
#+findex: claude-code-extras-teardown-copilot
The function ~claude-code-extras-setup-copilot~ enables Copilot in the current Claude Code buffer. It is added to ~claude-code-start-hook~ automatically and guards on ~claude-code-extras-copilot-enabled~ ([[#h:copilot-options][Copilot options]]), the presence of an Eat terminal, and the availability of the =copilot= package. When all conditions are met, it bridges three mechanical mismatches between Copilot and Eat terminal buffers:

1. *Language ID.* Installs =:around= advice on ~copilot--get-language-id~ that returns ="plaintext"= in Claude Code buffers.  The default language ID ="eat"= is unknown to the Copilot server.

2. *Document URI.* Sets ~buffer-file-name~ to a temporary file path so that ~copilot--get-uri~ returns a =file://= URI.  The Copilot server silently ignores the virtual =buffer://= URIs that non-file buffers produce.

3. *Acceptance routing.* Installs =:around= advice on ~copilot-accept-completion~ that sends accepted text through ~eat-term-send-string~ instead of using ~delete-region~/~insert~, which would corrupt the terminal state.  Newlines in accepted completions are converted to Alt+Enter sequences so that Claude Code's TUI treats them as multi-line input rather than prompt submission.  The advice also sends the appropriate Copilot telemetry for both full and partial acceptance.

Additionally, the setup installs a ~post-command-hook~ handler that schedules a debounced document sync and completion request after each user command.  Two issues make copilot's built-in triggers ineffective in Eat buffers: (a) Eat's process filter runs with ~inhibit-modification-hooks~ bound to t, so neither ~after-change-functions~ nor ~track-changes~ (which copilot uses for incremental document sync) detect buffer modifications; (b) the terminal echo from a keystroke arrives asynchronously after ~post-command-hook~ has already fired.  The handler works around both problems by waiting 0.3 seconds for the echo to land, then sending a full-document ~textDocument/didChange~ notification to the Copilot server before requesting completions.  The completion request is positioned at the TUI input prompt (identified by the =❯= marker) rather than at the terminal cursor, which may be on a status-bar or border line due to TUI rendering updates.

Copilot's ~track-changes~ handler is unregistered after ~copilot-mode~ enables, since it cannot function in Eat buffers and would otherwise cause "Recovering from confusing calls to before/after-change-functions" errors.  Copilot's own ~post-command-hook~ handler (~copilot--post-command~) is also removed, since it would call ~copilot-complete~ without syncing the document first.

An =:around= advice on ~copilot--show-completion~ relocates the overlay to the TUI prompt line after the original function creates it.  In eat buffers the line numbers in the server's range drift because the buffer changes between sync and response, so the overlay often ends up on a TUI chrome line.  The advice lets the original function create the overlay, then moves it to the prompt and re-renders the ghost text at the correct position, and finally moves point there so the Emacs cursor sits at the ghost text boundary.

Completions are truncated to a single line via =:around= advice on ~copilot--set-overlay-text~, preventing ghost text from extending past the TUI input area into the terminal chrome below.

A predicate is added to ~copilot-disable-predicates~ that suppresses completions while ~buffer-read-only~ is non-nil.

4. *TAB acceptance.*  Eat's semi-char-mode keymap binds TAB to ~eat-self-input~, which takes priority over copilot's keymap overlay.  A buffer-local minor mode (~claude-code-extras--copilot-keys-mode~) overrides TAB so that it accepts the visible completion when one exists, and falls through to ~eat-self-input~ otherwise.

The function ~claude-code-extras-teardown-copilot~ clears the buffer-local Copilot integration flag and cancels any pending change timer when the buffer is killed. It is added to ~kill-buffer-hook~ automatically.

** Non-interactive execution internals
:PROPERTIES:
:CUSTOM_ID: h:noninteractive-internals
:END:

#+findex: claude-code-extras--build-cli-args
#+findex: claude-code-extras--run-prompt
#+findex: claude-code-extras--batch-parse-stream-json
The function ~claude-code-extras--build-cli-args~ assembles the command-line arguments for =claude -p=, incorporating allowed tools, max turns, system prompt, and model override. It accepts keyword arguments that default to the corresponding =claude-code-extras-batch-*= customization variables.

The function ~claude-code-extras--run-prompt~ is the single-shot runner that spawns a =claude -p= process asynchronously. It takes a prompt string and keyword arguments =:dir=, =:callback=, =:allowed-tools=, =:system-prompt=, =:model=, and =:max-turns=. Any =CLAUDE_CODE= environment variables are stripped from ~process-environment~ to prevent nested-session errors. When the process finishes, the sentinel parses the stream-json output and calls the callback with a result plist containing =:exit-code=, =:duration=, =:cost=, =:text=, =:session-id=, and =:raw=. Both ~claude-code-extras-run-skill~ and ~claude-code-extras--batch-run-next~ delegate to this function.

The function ~claude-code-extras--batch-parse-stream-json~ parses the raw stream-json output from =claude -p= into a plist with =:text= (concatenated assistant messages), =:cost=, and =:session-id=.

** Skill runner internals
:PROPERTIES:
:CUSTOM_ID: h:skill-runner-internals
:END:

#+findex: claude-code-extras--parse-skill-frontmatter
#+findex: claude-code-extras--discover-skills
#+findex: claude-code-extras--skill-display-result
The function ~claude-code-extras--parse-skill-frontmatter~ reads a =SKILL.md= file and extracts the YAML frontmatter into a plist with keys =:name=, =:description=, =:argument-hint=, and =:user-invocable=.

The function ~claude-code-extras--discover-skills~ scans =~/.claude/skills/= and the current project's =.claude/skills/= for =SKILL.md= files, parses their frontmatter, and returns a list of skill plists. Project skills shadow global skills with the same name. Skills with =user-invocable: false= are excluded.

The function ~claude-code-extras--skill-display-result~ displays the skill result.  When the skill created its own buffer (e.g. via =emacsclient=), the metadata (cost, duration, session) is added to that buffer and no separate buffer is created.  Otherwise a =*Claude Skill: <name>*= buffer is created with the full output.

** Batch TODO internals
:PROPERTIES:
:CUSTOM_ID: h:batch-todo-internals
:END:

#+findex: claude-code-extras--batch-collect-todos
#+findex: claude-code-extras--batch-format-prompt
#+findex: claude-code-extras--batch-start
#+findex: claude-code-extras--batch-run-next
#+findex: claude-code-extras--batch-finish
The function ~claude-code-extras--batch-collect-todos~ uses ~org-map-entries~ to find entries with a TODO state in the given scope. For each entry, it extracts the heading text (without TODO keyword, priority, tags, or comment) and the body text (excluding property drawers and child subtrees).

The function ~claude-code-extras--batch-format-prompt~ combines an entry's title and body into a single prompt string. If the body is empty, the title alone is used.

The function ~claude-code-extras--batch-start~ initializes the processing state, creates a timestamped log subdirectory, and kicks off the async chain.

The function ~claude-code-extras--batch-run-next~ pops the next entry from the queue, formats the prompt, and delegates to ~claude-code-extras--run-prompt~. The callback saves raw output to a log file, records the result, optionally commits changes, and recurses to process the next entry.

The function ~claude-code-extras--batch-finish~ constructs and displays the =*Claude Batch Results*= summary buffer with aggregate statistics and per-entry details.

** Auto-setup
:PROPERTIES:
:CUSTOM_ID: h:auto-setup
:END:

#+findex: claude-code-extras-ensure-statusline-config
#+findex: claude-code-extras-ensure-stop-hook-config
The function ~claude-code-extras-ensure-statusline-config~ checks =~/.claude/settings.json= for a =statusLine= entry and adds one if absent, pointing to the bundled shell script. This ensures that the status JSON file is written for every Claude session without manual configuration.

The function ~claude-code-extras-ensure-stop-hook-config~ similarly checks for a =Stop= hook entry and adds one if absent, pointing to the bundled hook wrapper script. This enables the stop-event detection that powers scroll-to-bottom and pending theme application.

Both functions run automatically when the package is loaded.
