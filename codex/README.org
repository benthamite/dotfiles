* Overview

This directory contains the version-controlled parts of my [[https://developers.openai.com/codex/][Codex]] setup.
The cross-tool synchronization policy is documented in =../agents/README.org=.
This file covers the Codex-specific activation files and local conventions.

Claude and Codex artifacts are peer implementations of shared agent behavior.
Codex gets explicit counterpart files where Codex syntax, hook payloads,
discovery rules, or instruction-loading semantics require them.

* Current layout

#+begin_example
codex/
├── AGENTS.md              <- tracked global Codex instructions
├── config.toml            <- live ~/.codex/config.toml symlink target
├── hooks.json             <- live ~/.codex/hooks.json symlink target
├── bin/install-skills.sh  <- links ~/.codex/skills to tracked skills
├── hooks/                 <- Codex hook counterparts
├── README.org
├── rules/default.rules
└── skills/                <- Codex counterparts for paired skills

ai-config-sync.json        <- Claude/Codex sync manifest
bin/ai-config-sync         <- audit, reminder, and commit-guard checker
#+end_example

Repo-root =.codex/= is reserved for project-local Codex config. Do not use
=~/My Drive/dotfiles/.codex/= as a target for live global =~/.codex/= symlinks;
that makes Codex load the same file as both global and dotfiles-project-local
configuration. The topology table in =../agents/README.org= is the
authoritative cross-tool reference.

* Instructions

Codex reads global user instructions from =AGENTS.override.md= and =AGENTS.md=
in =$CODEX_HOME=, so the live =~/.codex/AGENTS.md= points to the tracked
=codex/AGENTS.md= file.

Project-level Codex instructions live in =AGENTS.md= files next to their
Claude =CLAUDE.md= peers. The old =project_doc_fallback_filenames = ["CLAUDE.md"]=
setting is intentionally not used, so every project-local =CLAUDE.md= should
have a sibling =AGENTS.md= counterpart. The Claude/Codex sync hooks treat these
instruction files as project-local peers in the git repo being edited or
committed.

The tracked =codex/AGENTS.md= is the Codex-side peer of =claude/CLAUDE.md=. It
uses Codex-safe syntax: the title is tool-specific, Claude's
=@context/secrets.md= import is expanded into an explicit secrets rule, and
relative context/tool paths are absolute so they resolve correctly from any
project. =bin/ai-config-sync= normalizes those tool-specific forms before
checking instruction equivalence.

It also records the manual Chrome profile routing used by browser-only service
flows: =Default= for personal projects and =Profile 2= for Epoch/work projects
(=pablo@epoch.ai=).

=codex/config.toml= enables experimental goals support under =[features]=. It
also sets =suppress_unstable_features_warning = true= so Codex does not print
the startup warning on every launch.

* Skills

=codex/skills/= contains one Codex counterpart for each paired global skill
under =claude/skills/=. Most counterparts share the same skill bodies and
auxiliary files. The sync check removes tool-specific frontmatter keys such as
=user-invocable=, =allowed-tools=, =argument-hint=, and =model= before
comparison. Tool-specific skills may
intentionally diverge when Claude Code and Codex store state differently; these
cases are recorded in =ai-config-sync.json=.

For paired skills that keep small state files in their skill directories, update
both tool copies to the same contents unless =ai-config-sync.json= records an
explicit tool-specific divergence. Project-local skill pairs follow the same
paired-state rule as global skills.

Codex discovers global skills from =~/.codex/skills/*/SKILL.md=. The live
directory is a symlink to the tracked Codex tree. Run:

#+begin_src sh
codex/bin/install-skills.sh
#+end_src

The script ensures =~/.codex/skills= points at =codex/skills/= and applies any
native skill-disable settings recorded by =bin/skill-prune=. Local reusable
skills belong in both =claude/skills/= and =codex/skills/=, not as untracked
directories under =~/.codex/skills=. Codex-managed hidden system skills, when
present, live under ignored =codex/skills/.system/= runtime data and are not
part of the reusable Claude/Codex skill parity model.

Skill frontmatter descriptions are loaded for routing, so keep them concise:
preserve user phrasing, scope boundaries, and important near-miss exclusions,
but move implementation details to the skill body.

To reduce context pressure from stale skills, run:

#+begin_src sh
bin/skill-prune apply --days 60
#+end_src

The prune command scans recent Claude and Codex transcripts. It disables only
paired skills for which neither tool shows usage inside the threshold window;
if either Claude or Codex used the skill recently, both tools keep it enabled.
It includes global paired skills and dynamically discovered project-local skill
pairs under =~/My Drive=; override the discovery roots with
=SKILL_PRUNE_LOCAL_ROOTS=, using the platform path separator for multiple roots.
Disabled skills are recorded in =skills-disabled.json= and applied through native tool settings: Claude
=skillOverrides= entries set to ="off" in the live
=~/.claude/settings.local.json=, and Codex =[[skills.config]]= entries with
=enabled = false= in =codex/config.toml=. Re-enable a disabled skill with
=bin/skill-prune enable NAME=; add =--keep= when a renamed or otherwise
intentionally retained skill should remain enabled despite missing history.

Project-specific skills belong beside their project instead of this global
tree. Dotfiles maintenance skills live in the dotfiles repo-root project-local
=.claude/skills/= and =.codex/skills/= roots so they are available while
working in the dotfiles tree without loading globally in unrelated projects.
For the Epoch workspace, the Claude copies live in
=~/My Drive/Epoch/.claude/skills/= and the Codex copies live in
=~/My Drive/Epoch/.codex/skills/=. The sync reminder and commit guard apply to
these project-local pairs too: changing one local skill reminds the agent to
update the other, and committing one side without an equivalent counterpart is
blocked.

The =doc-extras= skill is a paired Claude/Codex documentation workflow for
auditing, creating, and refreshing manuals under =emacs/extras/doc/= from the
top-level sources in =emacs/extras/=. It delegates the Denote-style manual
standard to =doc-elisp=, checks coverage for every top-level =.el= file,
verifies changed Org manuals and generated Texinfo before committing, and keeps
package documentation commits scoped to one package at a time.

The =nosync= skill moves one explicitly named generated directory from
=~/My Drive/= to =~/.drive-nosync/=, leaves a symlink in its place, refuses to
merge or overwrite an existing mirrored target, verifies the replacement
symlink, and reports any broken symlinks that cannot be repaired. Use it for a
specific problematic directory; use =fix-drive-errors= for broad scans.

The =fix-drive-errors= skill is the batch counterpart to =nosync= for Google
Drive sync cleanup. It scans =~/My Drive/= for generated dependency, build,
cache, and virtual environment directories; skips ambiguous source/user data;
moves confirmed offenders to =~/.drive-nosync/=; rewrites broken relative
symlinks after each move; reports conflicts or missing targets instead of
guessing; and restarts Google Drive only after changes.

=bin/codex-app-handoff= opens the current Codex thread in the desktop app via
the app's =codex://threads/<THREAD_ID>= deeplink. Use it from CLI/Emacs turns
when a task needs App-only Browser Use tools; the next turn must be sent from
the Codex App for the Browser plugin / Node REPL tools to appear. The command
also copies a short continuation prompt to the Emacs kill ring and macOS
clipboard, because the existing-thread deeplink opens the thread but does not
submit a new message.

When changing a skill:

1. Edit the side requested by the user.
2. Update the counterpart in the same session.
3. Stage both sides. For global skills, update =ai-config-sync.json= with a
   clear artifact-specific status or exemption if a counterpart is intentionally
   absent.

When renaming a global skill, rename both the Claude and Codex directories,
update each =name= frontmatter value, and update the skill entry in
=ai-config-sync.json=. When removing a global skill, remove both copies and
remove the skill's entry from =ai-config-sync.json=. Project-local replacements
should live in the target project's =.claude/skills/= and =.codex/skills/=
directories.

Skill files must be named uppercase =SKILL.md=.  This matters on the Google
Drive mount and for Codex skill discovery.

The =dotfiles-context= skill is a paired Claude/Codex context skill for routing
dotfiles and Emacs package edits to the correct canonical clone. It explains
the =dotfiles= exception to the normal elpaca source layout, active profile
lookup, profile-aware =emacs/config.org= tangling, README update obligations,
verification handoffs, and the standard =<scope>: <description>= commit format.

The =best-of-n= skill is a paired Claude/Codex workflow for querying several
models multiple times, selecting or merging the best samples, and synthesizing
the result.  Agents resolve the live tool-specific skill directory before
running, use =models.json= as the source of truth for preset details, exclude
deep-research/browser presets from the normal menu, and verify result files
before reporting completion.

The =proofread= skill is a paired Claude/Codex workflow for proofreading
Markdown files with aspell or Gemini Flash. It resolves the live tool-specific
skill directory, keeps API keys scoped to the skill =.env= file, writes reviewed
output to =*.proofread.md=, applies accepted suggestions to =*.final.md=, and
uses British English conventions by default.

The =triage-issues= skill is a paired Claude/Codex operations workflow for
open GitHub issues in =tlon-team/tlon.el=. It starts with a scope and mode gate
that verifies the current checkout, separates read-only triage, local fixes,
and externally visible closure, and requires explicit authorization before
GitHub comments, issue closure, or org archiving. It presents batches before
edits, verifies =TLON_AI_GITHUB_TOKEN= belongs to =tlon-ai= without creating
test comments, commits fixes issue-by-issue, and reports commit, check,
GitHub, and org-archive verification.

The =google-sheets-comments= skill is a paired Claude/Codex workflow for
processing Google Sheets comment review, replies, resolves, and reopens. It
makes Drive comments via =gdoc comments= the source of truth, treats Gmail
notifications only as pointers, excludes ordinary spreadsheet editing, requires
one-comment-at-a-time approval before externally visible actions, uses the
matching =gdoc reply=, =gdoc resolve=, or =gdoc reopen= command, and verifies
with =gdoc comment-info= before moving on.

The =twitter= skill is a paired Claude/Codex workflow for direct Twitter/X
lookup, search, and save requests via the active skill copy's
=lib/twitterapi.sh= curl wrapper around twitterapi.io. It supports tweet,
thread/reply, timeline, profile, follower/following, user-search, and keyword
search calls; documents the explicit Claude/Codex wrapper paths; protects API
keys from logging; separates direct lookup from =twitter-digest=,
=twitter-discover=, and =twitter-vet= workflows; and requires response,
pagination, and saved-file verification before reporting completion.

The =twitter-digest= skill is a paired Claude/Codex workflow for fetching recent
tweets from a named curated account list, triaging them against the list rubric,
saving an org digest, and updating last-run state. It uses the active
=fetch-tweets.sh= helper and active =twitterapi.sh= wrapper rather than MCP
Twitter tools, keeps API keys out of logs, writes digest archives and last-run
timestamps only into the current tool's runtime tree, keeps curated list files
synchronized in both skill trees, uses the shared Claude-side =twitter-vet=
registry, fails loudly when no listed account fetch succeeds, and verifies fetch
markers, saved digest, last-run timestamp, diff cleanliness, and staged paths
before committing digest, timestamp, list, and vet registry changes.

The =twitter-discover= skill is a paired Claude/Codex workflow for iterative,
graph-based discovery of high-value Twitter accounts in any niche. It
interviews the user for criteria, acquires seed accounts via search, expands
through hot promise scores from following-list overlap, uses the active
Claude/Codex skill root for wrappers and saved state, keeps API responses in a
per-session temporary directory, verifies its parsing helper, re-reads saved
logs/lists/registries, and optionally exports high-scoring accounts to a
=twitter-digest= list.

The =twitter-vet= skill is a paired Claude/Codex workflow for vetting, scoring,
ranking, or deciding whether Twitter/X accounts belong on curated lists. It
requires a handle, list name, rubric, and optional =vet-threshold=; uses the
active =twitter= skill wrapper for read-only tweet sampling; applies a quick
filter plus a full 1-10 scoring pass with the shared epistemic quality gate; and
keeps the shared vet registry under =~/.claude/skills/twitter-vet/vetted/= so
Claude and Codex callers do not split history. It auto-adds accounts only for
existing digest lists, returns scores to =twitter-discover= until its save
phase, and verifies registry/list placement plus clean diffs before standalone
commits.

The =handoff= skill saves next-session instructions to
=/tmp/codex-handoff.md= for Codex session transitions. It writes explicit user
next steps directly, but if next steps must be inferred it first prints the
full proposed prompt as a code block and asks for yes/no confirmation. Inferred
prompts include exact dates, file paths, command names, commit hashes, and
verification state; after writing, the skill re-reads the temp file, prints the
saved contents, and tells the user to run =! emacsclient -e '(ai-agent-codex-handoff)'=.

The =verify= skill is a paired Claude/Codex workflow for running an explicit
verification loop when the user asks to verify, close the loop, define success
criteria before work, or rigorously check an output whose correctness is not
automatically answerable. It generates concrete criteria before acting,
meta-verifies criteria quality, proceeds without a user checkpoint when criteria
are clear, asks only when criteria require user/domain judgment or change
scope/risk, evaluates each criterion, loops on failures, and reports
pass/fail/uncertain evidence. Ordinary coding and debugging use narrower
project checks unless this meta-verification workflow was explicitly requested.

The =walk-list= skill is a paired Claude/Codex workflow for strict item-by-item
processing of JSON arrays, newline-delimited JSON, or newline-separated text
files. It moves the source file into the protected walk-list store, leaves a
stub with quoted commands generated from the same =walk.py= path that started
the walk, validates positive concurrency and non-negative stale-claim
thresholds, and distinguishes strict isolation/concurrency work from ordinary
summarizing, filtering, sorting, or bulk list transformations.

The =code-audit= skill is a paired Claude/Codex development skill for finding
actual correctness, security, error-handling, and data-integrity defects in
application code. It requires agents to resolve scope, identify project-specific
verification, read real code paths, and report concrete failure modes. With
=--accept=, it fixes only confirmed in-scope findings that can be verified and
committed cleanly; broader readability, architecture, or security-posture
reviews route to the neighboring audit skills.

The =test-suite= skill is a paired Claude/Codex development workflow for adding
or expanding high-value tests. It triggers on requests to write tests, improve
coverage, bootstrap testing, or create regression tests, while routing ordinary
debugging, code review, lint cleanup, and one-off validation elsewhere. It
reads project instructions, existing runners, fixtures, dependencies, and
critical paths before writing behavior-focused tests by risk, then reports
commands run, coverage added, gaps left, bugs uncovered, and unresolved
failures.

The =pr-audit= skill is a paired Claude/Codex PR readiness workflow. It
discovers the branch and PR base, runs project build/test checks, reviews each
commit in the PR range, checks upstream freshness, scans changed files for
artifacts, secrets, whitespace, and line endings, and polls remote CI only when
the branch or PR is already pushed. It reports unavailable checks explicitly and
does not change remote state.

The =post-push-ci= skill is a paired Claude/Codex workflow for closing the
GitHub Actions loop after an authorized agent push. It runs =ci-after-push= to
wait for push-triggered CI on the pushed SHA, fetches failed logs, fixes root
causes, commits and pushes follow-up fixes, and repeats until the pushed commit
is green or a concrete blocker is identified. It does not grant permission to
push; it only applies after a push has already been authorized.

The =release= skill is a paired Claude/Codex operations workflow for releasing
the user's listed Emacs packages from the listed headings in
=my-emacs-packages.org=, excluding the unlisted section. It supports =/release

The =diagnose= skill is a paired Claude/Codex diagnostic mode modifier for
investigating errors, confusion, and process/tooling friction before patching.

requests, and requests to investigate failures without immediately changing
code. It requires the agent to verify which existing hooks, skills, configs, or
scripts should have caught the issue, classify the diagnosis, distinguish
structural gaps from behavioral lapses, and report before implementing any
remedy.

The =emacs-freeze= skill is a paired Claude/Codex operations skill for frozen,

per-PID diagnostics, tries to read =*Backtrace*= and =*Messages*= through the

steps separately. Recovery actions that change Emacs state are recommendations

The =design-audit= skill is a paired Claude/Codex development skill for

* Global Activation

=~/.codex/config.toml= points to =codex/config.toml=, and
=~/.codex/hooks.json= points to =codex/hooks.json=.  Updating the tracked files
therefore updates the live global Codex configuration.

=codex/config.toml= sets the default model and reasoning effort, enables Codex
hooks, enables the goals UI and terminal resize reflow, records trusted project
roots, records model-availability and hook-trust state written by Codex, and
sets the TUI theme.
Trusted project roots are Codex-local allowlist entries for directories that can
run without extra trust prompts; they do not have a Claude settings counterpart.
The tracked allowlist includes the nested =dotfiles/emacs= project because it is
opened independently from the dotfiles root during Emacs configuration work.

=~/.codex/AGENTS.md= points to =codex/AGENTS.md= so Codex gets a tracked global
instruction file with Codex-safe path references. This is needed for user-level
instructions; project-level Codex instructions are handled by sibling
=AGENTS.md= files.

=~/.codex/skills= points directly to =codex/skills/= via
=codex/bin/install-skills.sh= because Codex does not load arbitrary skill paths
from =codex/config.toml=. =bin/skill-prune= disables stale skills natively with
=[[skills.config]]= entries in =codex/config.toml= instead of moving skill
directories. Project-local Codex skills are stored under the project's own
=.codex/skills/= directory. Codex-managed hidden system skills, if present, are
ignored runtime data under =codex/skills/.system/=.
