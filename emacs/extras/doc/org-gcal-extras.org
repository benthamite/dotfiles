This manual describes the features and customization options for the Emacs Lisp file =org-gcal-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=org-gcal-extras.el= extends =org-gcal=, the package that synchronizes Org mode entries with Google Calendar events. The upstream package stores event timestamps as =SCHEDULED= properties and places descriptions inside a dedicated =:org-gcal:= drawer. This layout may not suit every workflow---for instance, if you use deadlines rather than scheduled dates for calendar events, or if you prefer event descriptions to live in the heading body rather than a drawer.

The package provides three categories of functionality:

- *Commands.* A command to open the Google Calendar event at point in a web browser, and a transient menu that groups all common =org-gcal= operations into a single dispatch interface ([[#h:commands][Commands]]).
- *Compatibility fixes.* Advice that prevents ~track-changes~ assertion failures and stale org-element cache errors during asynchronous synchronization on Emacs 30 and later ([[#h:compatibility-fixes][Compatibility fixes]]).
- *Patched synchronization behavior.* =el-patch= modifications that change how =org-gcal= reads and writes event data: using =DEADLINE= instead of =SCHEDULED=, placing descriptions in the heading body, and removing the =:org-gcal:= drawer ([[#h:patched-functions][Patched functions]]).

The package depends on =org-gcal=, =el-patch=, and =transient=.

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Opening events in Google Calendar
:PROPERTIES:
:CUSTOM_ID: h:open-at-point
:END:

#+findex: org-gcal-extras-open-at-point
When reviewing synchronized calendar events in Org mode, you may want to edit an event in the Google Calendar web interface directly. The command ~org-gcal-extras-open-at-point~ reads the =entry-id= property from the Org heading at point, constructs the corresponding Google Calendar edit URL (encoding the event ID in base64), and opens it in the default web browser via ~browse-url~. If the heading has no =entry-id= property, the command signals a user error.

** Transient menu
:PROPERTIES:
:CUSTOM_ID: h:transient-menu
:END:

#+findex: org-gcal-extras-menu
The =org-gcal= package provides many commands but no built-in dispatch interface. The command ~org-gcal-extras-menu~ defines a transient menu that groups all common operations into four columns:

- *Fetch:* ~org-gcal-fetch~ (fetch all calendars) and ~org-gcal-fetch-buffer~ (fetch only the current buffer's calendar).
- *Sync:* ~org-gcal-sync~ (sync all calendars) and ~org-gcal-sync-buffer~ (sync only the current buffer).
- *Act:* ~org-gcal-post-at-point~ (push local changes for the event at point), ~org-gcal-extras-open-at-point~ (open in browser, [[#h:open-at-point][Opening events in Google Calendar]]), and ~org-gcal-delete-at-point~ (delete the event).
- *Setup:* ~org-gcal--sync-unlock~ (release a stuck sync lock), ~org-gcal-sync-tokens-clear~ (clear stored sync tokens), ~org-gcal-toggle-debug~ (toggle debug logging), and ~org-gcal-reload-client-id-secret~ (reload OAuth credentials).

This menu is defined with ~transient-define-prefix~ and can be bound to a convenient key for quick access.

* Compatibility fixes
:PROPERTIES:
:CUSTOM_ID: h:compatibility-fixes
:END:

Emacs 30 introduced the ~track-changes~ facility, which can trigger assertion failures when =org-gcal= modifies buffers asynchronously during synchronization. Additionally, the org-element cache may enter a stale state during these async updates, causing "emergency" cache resets and subsequent errors.

=org-gcal-extras.el= addresses both issues with an =:around= advice on ~org-gcal--update-entry~ that wraps the original function in two protective measures: it sets ~inhibit-modification-hooks~ to ~t~ during the update (preventing ~track-changes~ assertions), and it resets the org-element cache both before and after the update (preventing stale cache errors). These fixes are applied automatically when the package is loaded.

* Patched functions
:PROPERTIES:
:CUSTOM_ID: h:patched-functions
:END:

** Reading event timestamps
:PROPERTIES:
:CUSTOM_ID: h:get-time-and-desc-patch
:END:

The internal function ~org-gcal--get-time-and-desc~ determines the start and end times of an event by reading Org heading properties. The upstream version prefers the =SCHEDULED= property; the =el-patch= in =org-gcal-extras.el= changes this to prefer the =DEADLINE= property instead. If neither property is present, the function falls back to timestamps found within the =:org-gcal:= drawer, as in the original. This change ensures that when events are pushed back to Google Calendar, the deadline-based timestamps are used.

** Writing event data
:PROPERTIES:
:CUSTOM_ID: h:update-entry-patch
:END:

The internal function ~org-gcal--update-entry~ writes Google Calendar event data into an Org heading during synchronization. The upstream version creates an =:org-gcal:= drawer, writes the event timestamp and description inside it, and uses the =SCHEDULED= property for the event date.

The =el-patch= in =org-gcal-extras.el= makes three changes to this behavior:

1. The =:org-gcal:= drawer is not created. There is no =:ORG-GCAL:= / =:END:= wrapper around the event content.
2. The event description is inserted directly into the heading body (after the properties drawer), making it visible without opening a drawer.
3. The event date is set using ~org-deadline~ instead of ~org-schedule~, so the timestamp appears as a =DEADLINE= property rather than a =SCHEDULED= property.

Together with the timestamp reading patch ([[#h:get-time-and-desc-patch][Reading event timestamps]]), these changes create a consistent workflow where Google Calendar events use deadlines throughout the round-trip synchronization.
