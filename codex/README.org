#+title: Codex configuration
#+author: Pablo Stafforini

* Overview

This directory contains the version-controlled parts of my [[https://developers.openai.com/codex/][Codex]] setup.
The cross-tool synchronization policy is documented in =../agents/README.org=.
This file covers the Codex-specific activation files and local conventions.

Claude and Codex artifacts are peer implementations of shared agent behavior.
Codex gets explicit counterpart files where Codex syntax, hook payloads,
discovery rules, or instruction-loading semantics require them.

* Current layout

#+begin_example
codex/
├── AGENTS.md              <- tracked global Codex instructions
├── config.toml            <- live ~/.codex/config.toml symlink target
├── hooks.json             <- live ~/.codex/hooks.json symlink target
├── bin/install-skills.sh  <- links ~/.codex/skills to tracked skills
├── hooks/                 <- Codex hook counterparts
├── README.org
├── rules/default.rules
└── skills/                <- Codex counterparts for paired skills

ai-config-sync.json        <- Claude/Codex sync manifest
bin/ai-config-sync         <- audit, reminder, and commit-guard checker
#+end_example

Repo-root =.codex/= is reserved for project-local Codex config. Do not use
=~/My Drive/dotfiles/.codex/= as a target for live global =~/.codex/= symlinks;
that makes Codex load the same file as both global and dotfiles-project-local
configuration. The topology table in =../agents/README.org= is the
authoritative cross-tool reference.

* Instructions

Project-level =CLAUDE.md= files are loaded by Codex because
=codex/config.toml= sets:

#+begin_example
project_doc_fallback_filenames = ["CLAUDE.md"]
#+end_example

That setting removes the need for project-level =AGENTS.md= symlinks.  It does
not apply to Codex's user-level instruction file in =$CODEX_HOME=. Codex reads
global user instructions from =AGENTS.override.md= and =AGENTS.md= in
=$CODEX_HOME=, so the live =~/.codex/AGENTS.md= points to the tracked
=codex/AGENTS.md= file.

There is no repo-root =AGENTS.md= manual counterpart. Project-level Codex
instructions come from =CLAUDE.md= through =project_doc_fallback_filenames=.
The Claude/Codex sync hooks therefore only treat =claude/CLAUDE.md= and
=codex/AGENTS.md= as a paired instruction file; other project =CLAUDE.md= or
=AGENTS.md= files are not paired by that hook.

The tracked =codex/AGENTS.md= is the Codex-side peer of =claude/CLAUDE.md=. It
uses Codex-safe syntax: the title is tool-specific, Claude's
=@context/secrets.md= import is expanded into an explicit secrets rule, and
relative context/tool paths are absolute so they resolve correctly from any
project. =bin/ai-config-sync= normalizes those tool-specific forms before
checking instruction equivalence.

It also records the manual Chrome profile routing used by browser-only service
flows: =Default= for personal projects and =Profile 2= for Epoch/work projects
(=pablo@epoch.ai=).

=codex/config.toml= enables experimental goals support under =[features]=. It
also sets =suppress_unstable_features_warning = true= so Codex does not print
the startup warning on every launch.

* Skills

=codex/skills/= contains one Codex counterpart for each paired global skill
under =claude/skills/=. Most counterparts share the same skill bodies and
auxiliary files. The sync check removes tool-specific frontmatter keys such as
=user-invocable=, =allowed-tools=, =argument-hint=, and =model= before
comparison. Tool-specific skills may
intentionally diverge when Claude Code and Codex store state differently; these
cases are recorded in =ai-config-sync.json=.

For paired skills that keep small state files in their skill directories, update
both tool copies to the same contents unless =ai-config-sync.json= records an
explicit tool-specific divergence. For example, =ai-journal= keeps its
=last-run.txt= cutoff synchronized under both =claude/skills/ai-journal/= and
=codex/skills/ai-journal/=.

Codex discovers global skills from =~/.codex/skills/*/SKILL.md=.  The live
skills directory is therefore a symlink to the tracked Codex tree.  Run:

#+begin_src sh
codex/bin/install-skills.sh
#+end_src

The script ensures =~/.codex/skills= points at =codex/skills/=. Local reusable
skills belong in both =claude/skills/= and =codex/skills/=, not as untracked
directories under =~/.codex/skills=. Codex-managed hidden system skills, when
present, live under ignored =codex/skills/.system/= runtime data and are not
part of the reusable Claude/Codex skill parity model.

Project-specific skills belong beside their project instead of this global
tree. For the Epoch workspace, the Claude copies live in
=~/My Drive/Epoch/.claude/skills/= and the Codex copies live in
=~/My Drive/Epoch/.codex/skills/=. The sync reminder and commit guard apply to
these project-local pairs too: changing one local skill reminds the agent to
update the other, and committing one side without an equivalent counterpart is
blocked.

The former dotfiles-project-local Claude skills =find-skills=,
=skill-creator=, and =systematic-debugging= now live in the global
=claude/skills/= and =codex/skills/= trees so they are available consistently
from both tools in every project.

The =doc-extras= skill is a paired Claude/Codex documentation workflow for
auditing, creating, and refreshing manuals under =emacs/extras/doc/= from the
top-level sources in =emacs/extras/=. It delegates the Denote-style manual
standard to =doc-elisp=, checks coverage for every top-level =.el= file,
verifies changed Org manuals and generated Texinfo before committing, and keeps
package documentation commits scoped to one package at a time.

Codex-only skills are allowed when they document a capability that exists only
in Codex. Record them in =ai-config-sync.json= with =status: codex-only=,
=claude: null=, and the tracked Codex skill path. The current example is
=codex-app-browser=, which captures the Codex App in-app browser handoff and
Browser Use bootstrap procedure; it intentionally has no Claude counterpart.
The skill defers to the bundled =browser-use:browser= skill for the current
plugin root and Browser API instead of hard-coding a versioned plugin path.

=bin/codex-app-handoff= opens the current Codex thread in the desktop app via
the app's =codex://threads/<THREAD_ID>= deeplink. Use it from CLI/Emacs turns
when a task needs App-only Browser Use tools; the next turn must be sent from
the Codex App for the Browser plugin / Node REPL tools to appear. The command
also copies a short continuation prompt to the Emacs kill ring and macOS
clipboard, because the existing-thread deeplink opens the thread but does not
submit a new message.

User prompts should not need to mention the transport details. The intended
shape is =\$codex-app-browser <task>=; the skill decides whether to hand off
from CLI/Emacs or run the App Browser preflight immediately.

When changing a skill:

1. Edit the side requested by the user.
2. Update the counterpart in the same session.
3. Stage both sides. For global skills, update =ai-config-sync.json= with a
   clear artifact-specific status or exemption if a counterpart is intentionally
   absent.

When renaming a global skill, rename both the Claude and Codex directories,
update each =name= frontmatter value, and update the skill entry in
=ai-config-sync.json=. When removing a global skill, remove both copies and
remove the skill's entry from =ai-config-sync.json=. Project-local replacements
should live in the target project's =.claude/skills/= and =.codex/skills/=
directories.

Skill files must be named uppercase =SKILL.md=.  This matters on the Google
Drive mount and for Codex skill discovery.

The =dotfiles-context= skill is a paired Claude/Codex context skill for routing
dotfiles and Emacs package edits to the correct canonical clone. It explains
the =dotfiles= exception to the normal elpaca source layout, active profile
lookup, profile-aware =emacs/config.org= tangling, README update obligations,
verification handoffs, and the standard =<scope>: <description>= commit format.

The =best-of-n= skill is a paired Claude/Codex workflow for querying several
models multiple times, selecting or merging the best samples, and synthesizing
the result.  Agents resolve the live tool-specific skill directory before
running, use =models.json= as the source of truth for preset details, exclude
deep-research/browser presets from the normal menu, and verify result files
before reporting completion.

The =gsc-indexing-triage= skill is a paired Claude/Codex operations skill for
=stafforini.com= Search Console Page indexing alerts.  It includes the
=scripts/extract-gsc-links.py= helper, a site-specific reference file, and a
workflow for reading GSC emails, fixing root causes, quick-deploying when
authorized, validating in the browser, archiving handled emails, and appending
to the site's persistent =logs/gsc-indexing.md= log.  Its frontmatter is kept
Codex-safe because Codex skips skills whose =description= field is not parsed as
a string.

The =google-sheets-comments= skill is a paired Claude/Codex workflow for
processing Google Sheets comments. It makes Drive comments via =gdoc comments=
the source of truth, treats Gmail notifications only as pointers, requires
one-comment-at-a-time approval before externally visible replies, posts through
=gdoc reply=, and verifies with =gdoc comment-info= before moving on.

The =skill-resolver= skill wraps =bin/agent-skill=, a deterministic resolver for
opening skills by name.  It searches project-local skills before global skills
and supports =path=, =cat=, =list=, and =roots= so agents do not need to manually
expand skill root aliases such as =r0= / =r1=.

The =skill-audit= skill is a paired Claude/Codex meta-skill for reviewing
skills and =SKILL.md= files. It checks trigger quality, workflow design,
progressive disclosure, bundled resources, verification, portability, and
maintainability, with special attention to whether the skill captures a
repeatable workflow, triggers on realistic user phrasing, gives ordered
operational guidance, defines output and verification expectations, and avoids
surprising side effects.

The =diagnose= skill is a paired Claude/Codex diagnostic mode modifier for
investigating errors, confusion, and process/tooling friction before patching.
It triggers on =/diagnose=, why-did-this-happen questions, root-cause analysis
requests, and requests to investigate failures without immediately changing
code. It requires the agent to verify which existing hooks, skills, configs, or
scripts should have caught the issue, classify the diagnosis, distinguish
structural gaps from behavioral lapses, and report before implementing any
remedy.

The =code-audit= skill is a paired Claude/Codex development skill for finding
actual correctness, security, error-handling, and data-integrity defects in
application code. It requires agents to resolve scope, identify project-specific
verification, read real code paths, and report concrete failure modes. With
=--accept=, it fixes only confirmed in-scope findings that can be verified and
committed cleanly; broader readability, architecture, or security-posture
reviews route to the neighboring audit skills.

The =emacs-freeze= skill is a paired Claude/Codex operations skill for frozen,
hung, beachballing, or otherwise unresponsive Emacs sessions. It collects
per-PID diagnostics, tries to read =*Backtrace*= and =*Messages*= through the
server socket with a macOS-safe timeout helper, samples the process, and reports
observations, inferences, unknowns, recovery options, and recurrence-prevention
steps separately. Recovery actions that change Emacs state are recommendations
until the user explicitly confirms that the agent should run them.

The =design-audit= skill is a paired Claude/Codex development skill for
architecture and maintainability reviews. It treats auditing as the default
mode and uses =--accept= only for high-confidence, behavior-preserving
high-impact refactors; risky or broad redesigns are reported instead of
half-applied.

The =claude-config-audit= skill is a paired Claude/Codex meta-skill for auditing
Claude Code configuration across CLAUDE.md, skills, memory, hooks, settings, and
documentation. It inventories instruction sources, redacts sensitive setting
values, extracts discrete rules, checks for default-behavior restatements,
conflicts, duplicates, reactive one-off fixes, and vague guidance, then produces
a cut list, conflict list, and cleaned-up CLAUDE.md. Pass =--accept= or
explicitly ask to apply cleanup to make high-confidence scope-preserving edits,
verify changed files and sync checks, and commit the result.

The =automate= skill is a paired Claude/Codex meta-skill for reusable AI
workflow design. It decides whether a request should become a Claude Code skill,
a Codex skill, a gptel directive or preset, a gptel tool, or a gptel command,
then builds in the canonical dotfiles paths after confirmation and verifies the
result before committing.

The =add-to-emacs-packages= skill is a paired Claude/Codex workflow for
registering one of the user's Emacs packages in =my-emacs-packages.org=,
creating its documentation note, and adding the GitHub profile README card when
that externally visible update has been explicitly authorized. It saves org IDs
after creating them, verifies the note/list/profile changes before committing,
and cleans temporary clones with =trash=.

The =convert-citations= skill is a paired Claude/Codex workflow for replacing
raw bibliographic references in org files with =[cite:@KEY]= citations from the
user's citar bibliography. It validates file, directory, and glob targets,
keeps raw-reference edits out of org metadata and prose, uses the current
agent's file-editing and web lookup tools, and verifies edited files for
Markdown citation tags or placeholder markers before reporting.

The =audit-mac-app= skill is a paired Claude/Codex static-first workflow for
auditing macOS applications before running them or granting sensitive
permissions. It covers trust verification, permission analysis, app type
detection, Electron source extraction and security checks, network endpoint
scanning, and spaces-safe binary analysis. Its bundled scanner uses =mktemp=
for temporary directories and moves extracted sources to Trash when possible.

The =build= skill is a paired Claude/Codex workflow for turning a rough
=spec.md= or explicit spec path into an implementation-ready plan. It interviews
the user in depth, records goals, design, edge cases, implementation order, and
verification in the spec, and only proceeds to implementation after explicit
confirmation.

The =ai-journal= skill is a paired Claude/Codex workflow for scanning recent
Claude Code and Codex sessions, surfacing candidate AI journal topics, and
drafting org-sourced posts for =stafforini.com/ai=. Its =find-sessions.py= and
=last-run.txt= auxiliary files are kept synchronized in both paired skill
directories; the workflow checks for cutoff drift before scanning so either
tool can run it without silently using stale state.

The =update-log= skill is a paired Claude/Codex end-of-session bookkeeping
workflow. The Codex copy treats =CLAUDE.md= as the session-log index unless the
project explicitly says otherwise, asks for first-run setup without relying on
Claude-only tools, uses the =record-decisions= skill or its local workflow when
applicable, reports any =context/post-update-log-hook.md= hooks, and stages only
intended bookkeeping paths before committing.

* Hooks

=codex/hooks.json= is tracked here and live globally through the symlink at
=~/.codex/hooks.json=.  It registers Codex-side scripts from =codex/hooks/=
instead of calling Claude hook scripts directly.

The Emacs =codex= package also auto-adds =codex-hook-wrapper= entries for its
own event handling.  Those entries coexist with the dotfiles hooks and should
not be hand-pruned unless the Emacs package configuration changes.  The wrapper
commands must point at the package source or repo tree, not =elpaca/builds/=:
the build directory contains compiled Elisp but not the =bin/codex-hook-wrapper=
script.  Managed wrapper commands include explicit =--emacsclient= and
=--socket-name= arguments so hook dispatch targets the active Emacs server
instead of relying on emacsclient's default server lookup.

The Codex hook counterparts cover:

- Bash safety and workflow gates: secret leakage, destructive commands,
  walk-list data access, Elisp test-before-commit, manual update checks
  (including staged generated Texinfo outputs), post-commit Elisp
  verification, and stale elpaca rebuild protection.
- Shared hook payload helpers such as =lib-codex-hook-json.sh=, which normalize
  Codex =tool_input= and =tool_response= whether Codex supplies them as objects
  or JSON strings.
- Shared hook helpers such as =lib-repo-root.sh=, which resolves an optional
  leading =cd <path> &&= prefix, otherwise uses structured =workdir= / =cwd=
  hook payload fields when present, and only then falls back to the hook cwd.
- MCP guardrails through =guard-external-actions.sh=.
- =apply_patch=/=Edit|Write= guards for secret leakage, generated Texinfo
  files, elpaca dotfiles mirror edits, and walk-list protected data.
- Post-edit behavior for Elisp reloads, generated manual regeneration, README
  reminders, and Claude/Codex sync reminders. Manual regeneration runs
  =ox-texinfo= with batch lockfile creation disabled and only regenerates the
  exact =.texi= returned by the org exporter plus its existing matching
  =.info= file.
- Codex lifecycle forwarding to Emacs through =codex-hook-wrapper= entries in
  =codex/hooks.json=.

Codex PostToolUse hooks must write either no stdout or valid hook JSON.  Reminder
hooks therefore return
={"hookSpecificOutput":{"hookEventName":"PostToolUse","additionalContext":...}}=
instead of plain text; plain stdout, missing =hookEventName=, and unsupported
fields such as =hookSpecificOutput.message= are treated as invalid hook protocol
output by Codex.

Some Claude hooks are intentionally not paired one-for-one because Codex does
not expose the same event in this setup:

- Claude =Read=/=Grep=/=Glob= hooks such as =block-sensitive-read.sh=.
- Claude =Skill= hooks such as =require-elisp-skill.sh= and
  =track-elisp-skill.sh=.
- Claude Notification-specific Emacs hooks.  Codex routes lifecycle and
  notification-equivalent events through =codex-hook-wrapper= entries in
  =codex/hooks.json= instead.

* Sync Guard

=ai-config-sync.json= is the sync manifest.  It lists the global instruction
pair, every global skill, hook pairing status, live registration files, and the
current unsupported or Codex-only cases.

Two hook pairs keep the model simple:

- =claude/hooks/remind-ai-config-sync.sh= and
  =codex/hooks/remind-ai-config-sync.sh= run after edits and print the
  counterpart that should be updated.
- =claude/hooks/require-ai-config-sync.sh= and
  =codex/hooks/require-ai-config-sync.sh= run before =git commit= and block
  one-sided Claude/Codex config changes unless the counterpart is synchronized
  or, for global dotfiles artifacts, the manifest records an explicit
  artifact-specific exemption.

The same hooks also handle project-local skills and hooks in the git repo being
committed. Local pairs use =.claude/skills/= with =.codex/skills/=,
=.claude/hooks/= with =.codex/hooks/=, and =.claude/settings.json= with
=.codex/hooks.json=. Local skills are compared with the same tool-specific
frontmatter normalization used for global skills.

The Codex-only README gate, =require-codex-readme-update.sh=, is scoped to this
dotfiles repository. That prevents unrelated repositories with paths such as
=codex/= or =bin/ai-config-sync= from being treated as Codex integration
changes for these dotfiles.

Manual audit:

#+begin_src sh
bin/ai-config-sync audit
#+end_src

The audit is deterministic; it does not edit files.  Its job is to make drift
visible. See =../agents/README.org= for the full synchronization contract and
the audit scope.

* Global Activation

=~/.codex/config.toml= points to =codex/config.toml=, and
=~/.codex/hooks.json= points to =codex/hooks.json=.  Updating the tracked files
therefore updates the live global Codex configuration.

=codex/config.toml= sets the default model and reasoning effort, uses
=CLAUDE.md= as the project-doc fallback, enables Codex hooks, enables the goals
UI and terminal resize reflow, records trusted project roots, and sets the TUI
theme.
Trusted project roots are Codex-local allowlist entries for directories that can
run without extra trust prompts; they do not have a Claude settings counterpart.

=~/.codex/AGENTS.md= points to =codex/AGENTS.md= so Codex gets a tracked global
instruction file with Codex-safe path references. This is needed for user-level
instructions; project-level =CLAUDE.md= loading is handled by
=project_doc_fallback_filenames= instead.

=~/.codex/skills= points directly to =codex/skills/= via
=codex/bin/install-skills.sh= because Codex does not load arbitrary skill paths
from =codex/config.toml=. This keeps the live Codex skill inventory aligned
with the tracked global Claude/Codex skill counterparts. Project-local Codex skills
are stored under the project's own =.codex/skills/= directory. Codex-managed
hidden system skills, if present, are ignored runtime data under
=codex/skills/.system/=.
