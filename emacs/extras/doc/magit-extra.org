This manual describes the features and customization options for the Emacs Lisp file =magit-extra.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=magit-extra.el= extends =magit=, the popular Git interface for Emacs, with convenience commands and workflow automation. The package depends on =magit= and =paths=.

The extensions fall into several categories:

- *Streamlined commit workflows.* Commands that stage, commit, and push in a single step, reducing the friction of routine Git operations ([[#h:stage-commit-push][Stage, commit, and push]]).
- *Repository maintenance.* Batch updates across multiple repositories and automatic submodule management ([[#h:repository-maintenance][Repository maintenance]]).
- *Safety guards.* Dirty-repository warnings and confirmation prompts before pushing to production branches ([[#h:dirty-repo-detection][Dirty repository detection]]).
- *Asynchronous operations.* Background pulling for repositories that need periodic synchronization ([[#h:async-pull][Asynchronous pull]]).
- *Customized dispatch menu.* A modified Magit dispatch transient that overrides the default, providing a tailored command palette ([[#h:dispatch-menu][Dispatch menu]]).
- *Submodule transient suffixes.* Additional entries for the Magit submodule transient menu ([[#h:transient-suffixes][Transient suffixes]]).

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Stage, commit, and push
:PROPERTIES:
:CUSTOM_ID: h:stage-commit-push
:END:

#+findex: magit-extras-stage-commit-and-push
The most common Git workflow---stage modifications, write a commit message, and push---typically requires several separate steps in Magit. The command ~magit-extras-stage-commit-and-push~ collapses this into a single action. When called interactively, it first displays the unstaged diff for review, then prompts for a commit message.

The command accepts two optional arguments. When FILE is non-nil, only that file is staged rather than all modified files. When NO-PUSH is non-nil, the push step is skipped, which is useful when you want to batch several commits before pushing. The command is adapted from a workflow by Sacha Chua.

#+findex: magit-extras-stage-commit-and-push-all-repos
For users who maintain multiple Git repositories, ~magit-extras-stage-commit-and-push-all-repos~ iterates over all directories listed in ~paths-dir-all-repos~ and performs a "midnight update" on each one. This is intended for automated, periodic synchronization---for example, from a timer or a shutdown hook. Each repository receives a commit with the message "Midnight update".

#+findex: magit-extras-with-editor-finish-and-push
When composing a commit message in a =with-editor= buffer, you may want to finalize and push immediately. The command ~magit-extras-with-editor-finish-and-push~ calls ~with-editor-finish~ to complete the editing session and then invokes ~magit-push-current-to-pushremote~ to push the new commit.

** Tag and submodule checkout
:PROPERTIES:
:CUSTOM_ID: h:tag-submodule-checkout
:END:

#+findex: magit-extras-checkout-tag-with-submodules
Projects that use Git submodules often need to check out a specific tag consistently across the main repository and all submodules. The command ~magit-extras-checkout-tag-with-submodules~ prompts for a tag (or the "main" branch) via ~completing-read~ and then performs a force checkout.

When "main" is selected, the command checks out the =main= branch in every submodule recursively and pulls the latest changes. When a tag is selected, it resets all submodules to match the commit recorded in the superproject and reinitializes them. This distinction ensures that tag checkouts produce a deterministic state while branch checkouts pull the latest upstream changes.

** Dirty repository detection
:PROPERTIES:
:CUSTOM_ID: h:dirty-repo-detection
:END:

#+findex: magit-extras-warn-if-repo-is-dirty
#+findex: magit-extras-repo-is-dirty-p
Before performing automated operations on a repository, it is often wise to verify that it has no uncommitted changes. The command ~magit-extras-warn-if-repo-is-dirty~ checks whether the repository at REPO-PATH (or the current repository if nil) has any modifications and emits an alert via the =alert= package if so.

The predicate ~magit-extras-repo-is-dirty-p~ performs the underlying check using ~magit-anything-modified-p~. It returns ~t~ when the directory exists, is a Git repository, and contains uncommitted changes. Both functions are useful in hooks or timers that perform automated Git operations.

** Dispatch menu
:PROPERTIES:
:CUSTOM_ID: h:dispatch-menu
:END:

#+findex: magit-extras-dispatch
The function ~magit-extras-dispatch~ defines a customized transient prefix that replaces the default ~magit-dispatch~ via =advice-add=. The layout largely mirrors the standard Magit dispatch but includes the Forge dispatch under =n= and a push binding under =.=, among other adjustments. The menu is organized into three groups: transient and DWIM commands, applying changes (visible only in Magit buffers), and essential commands.

Because this function overrides ~magit-dispatch~, invoking the standard Magit dispatch keybinding will display this custom menu.

** Asynchronous pull
:PROPERTIES:
:CUSTOM_ID: h:async-pull
:END:

#+findex: magit-extra-async-pull
The function ~magit-extra-async-pull~ starts a background =git pull= process for REPO without blocking Emacs. It creates an asynchronous process and reports errors via ~message~ if the pull does not finish cleanly. This is useful for keeping repositories synchronized in the background, for instance from a timer or init hook.

** Automatic submodule pull
:PROPERTIES:
:CUSTOM_ID: h:auto-submodule-pull
:END:

#+findex: magit-extras-pull-including-submodules
The function ~magit-extras-pull-including-submodules~ is installed as =:after= advice on ~magit-pull~. Whenever you pull in a repository that contains a =.gitmodules= file, this function automatically updates all submodules recursively by calling ~magit-extras-pull-all-submodules~. This saves you from having to remember to update submodules manually after each pull.

** Production branch guard
:PROPERTIES:
:CUSTOM_ID: h:production-guard
:END:

Accidentally pushing to a production branch can have serious consequences. The internal function ~magit-extras--guard-production~ is installed as =:before= advice on both ~magit-push-current-to-pushremote~ and ~magit-push-current-to-upstream~. When the target branch name contains the string "production", the function prompts for confirmation with ~yes-or-no-p~ before allowing the push to proceed. If the user declines, the push is cancelled with a ~user-error~.

* Transient suffixes
:PROPERTIES:
:CUSTOM_ID: h:transient-suffixes
:END:

#+findex: magit-extras-update-all-submodules
#+findex: magit-extras-pull-all-submodules
The package defines two transient suffix commands intended for inclusion in Magit's submodule transient menu:

The suffix ~magit-extras-update-all-submodules~ runs =git submodule update --init= with optional =--recursive= arguments. This initializes any uninitialized submodules and checks out the exact commit recorded in the superproject. It does /not/ pull the latest changes from the submodule remotes, making it suitable for reproducing a known state.

The suffix ~magit-extras-pull-all-submodules~ runs =git submodule foreach --recursive git pull=, which fetches and merges the latest changes from each submodule's default upstream branch. Unlike ~magit-extras-update-all-submodules~, this updates submodules to their latest development state regardless of what the superproject records.

Both suffixes accept the standard ~magit-submodule-arguments~ for the =--recursive= flag and run asynchronously.

** URL and file helpers
:PROPERTIES:
:CUSTOM_ID: h:url-file-helpers
:END:

#+findex: magit-extras-parse-url
#+findex: magit-extras-get-commit-file
#+findex: magit-extras-get-commit-heading
#+findex: magit-extras-get-unstaged-files
#+findex: magit-extras-track-file
#+findex: magit-extras-move-point-to-start
#+findex: magit-extras-midnight-update
The function ~magit-extras-parse-url~ converts a Git remote URL in SSH format (e.g., =git@github.com:user/repo.git=) into an HTTPS URL. It passes HTTP URLs through unchanged.

The function ~magit-extras-get-commit-file~ extracts the name of the first file listed under "Changes to be committed" in the current buffer. It accepts an optional PATH argument: ~'full~ returns the absolute path, ~'sans-dir~ returns only the filename, and ~nil~ (the default) returns the path relative to the repository root.

The function ~magit-extras-get-commit-heading~ retrieves the Org mode heading associated with the code being committed. It parses the diff in the commit buffer to find the first changed line number, visits the file at that line, and returns the enclosing heading via ~org-entry-get~. This is useful for generating commit messages from Org structure.

The function ~magit-extras-get-unstaged-files~ returns a list of unstaged file names using =git diff --name-only --diff-filter=d=. The function ~magit-extras-track-file~ stages a single file with =git add=. The function ~magit-extras-move-point-to-start~ moves point to the beginning of the buffer after a brief delay, which can be useful in Magit hooks.

The function ~magit-extras-midnight-update~ performs a stage-commit-push cycle in the given DIRECTORY with the commit message "Midnight update". It is called by ~magit-extras-stage-commit-and-push-all-repos~ ([[#h:stage-commit-push][Stage, commit, and push]]).
