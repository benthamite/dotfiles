This manual describes the features and customization options for the Emacs Lisp file =forge-extras.el=.

* Overview
:PROPERTIES:
:CUSTOM_ID: h:overview
:END:

=forge-extras.el= extends =forge=, the Emacs package that integrates Git
forges (GitHub, GitLab, and others) into Magit.  While =forge= handles
the fundamentals---pulling topics, viewing issues and pull requests,
posting comments---=forge-extras.el= adds higher-level workflows on top
of that foundation.

The package addresses two broad areas:

- *Everyday issue and notification management.*  Commands for toggling
  issue state, navigating between posts in a topic buffer, copying
  message bodies, storing Org links for selected commits, tracking new
  repositories in the Forge database, and keeping notification read
  status in sync between Emacs and GitHub via the REST API
  ([[#h:notification-and-read-status][Notification and read-status synchronization]]).

- *GitHub Projects v2 integration.*  A suite of commands that interact
  with the GitHub GraphQL API through the =gh= command-line tool,
  allowing you to set project status and estimates, list project items in
  board order, discover field and option Node IDs, and insert Markdown
  links to issues---all without leaving Emacs ([[#h:github-project-integration][GitHub Project integration]]).

The package requires =forge= (version 0.3.1 or later) and =shut-up=.
The GitHub Project features additionally require the =gh= CLI to be
installed and authenticated.

* User options
:PROPERTIES:
:CUSTOM_ID: h:user-options
:END:

All user options belong to the ~forge-extras~ customization group.  Most
of them configure the GitHub Projects v2 integration; see
[[#h:github-project-setup][Setting up GitHub Project integration]] for a
guided walkthrough.

** Project owner and number
:PROPERTIES:
:CUSTOM_ID: h:project-owner-and-number
:END:

#+vindex: forge-extras-project-owner
#+vindex: forge-extras-project-number
The user option ~forge-extras-project-owner~ specifies the GitHub owner
(organization or user name) that owns the target project.  Its type is a
string and its default value is ~nil~.

The user option ~forge-extras-project-number~ specifies the project
number (the integer visible in the project's URL).  Its type is an
integer and its default value is ~nil~.

These two options are used by ~forge-extras-gh-get-issue-fields~ to
construct the GraphQL query that retrieves project data for a given
issue.  You must set both before using any of the project-related
commands ([[#h:github-project-integration][GitHub Project integration]]).

** Project Node ID
:PROPERTIES:
:CUSTOM_ID: h:project-node-id
:END:

#+vindex: forge-extras-project-node-id
The user option ~forge-extras-project-node-id~ holds the global Relay
Node ID of the target GitHub Project.  Its type is a string and its
default value is ~nil~.

This is the primary identifier used by most project commands.  See
[[#h:github-project-setup][Setting up GitHub Project integration]] for
instructions on how to obtain it.

** Status field Node ID
:PROPERTIES:
:CUSTOM_ID: h:status-field-node-id
:END:

#+vindex: forge-extras-status-field-node-id
The user option ~forge-extras-status-field-node-id~ stores the global
Relay Node ID of the "Status" single-select field in the target project.
Its type is a string and its default value is ~nil~.

This ID is required by ~forge-extras-set-project-status~
([[#h:set-project-status][Setting project status]]) and by the discovery
commands that inspect status options ([[#h:discover-project-ids][Discovering project IDs]]).
Use ~forge-extras-get-project-field-ids~ to find the correct value
([[#h:get-project-field-ids][Getting project field IDs]]).

** Estimate field Node ID
:PROPERTIES:
:CUSTOM_ID: h:estimate-field-node-id
:END:

#+vindex: forge-extras-estimate-field-node-id
The user option ~forge-extras-estimate-field-node-id~ stores the global
Relay Node ID of the "Estimate" numeric field in the target project.
Its type is a string and its default value is ~nil~.

This ID is required by ~forge-extras-set-project-estimate~
([[#h:set-project-estimate][Setting project estimates]]).  Use
~forge-extras-get-project-field-ids~ to find the correct value
([[#h:get-project-field-ids][Getting project field IDs]]).

** Status option IDs alist
:PROPERTIES:
:CUSTOM_ID: h:status-option-ids-alist
:END:

#+vindex: forge-extras-status-option-ids-alist
The user option ~forge-extras-status-option-ids-alist~ maps
human-readable status names (such as "Todo", "In Progress", "Done") to
their corresponding GraphQL Option IDs for the "Status" single-select
field.  Its type is an alist mapping strings to strings, and its default
value is ~nil~.

When ~forge-extras-set-project-status~ prompts you for a status, the
completion candidates come from the keys of this alist.  You can
populate it manually or use the output of
~forge-extras-discover-project-status-options~
([[#h:discover-status-options][Discovering status options]]), which
formats the data for easy insertion into your configuration.

** Issue link state
:PROPERTIES:
:CUSTOM_ID: h:issue-link-state
:END:

#+vindex: forge-extras-issue-link-state
The user option ~forge-extras-issue-link-state~ controls which issues
~forge-extras-insert-issue-markdown-link~ retrieves from GitHub.  Its
type is a choice among the strings ~"open"~, ~"closed"~, and ~"all"~.
The default value is ~"open"~.

If you frequently need to reference closed issues, change this to
~"all"~.  The value is passed directly to =gh issue list --state=.

** Debug flag
:PROPERTIES:
:CUSTOM_ID: h:debug-flag
:END:

#+vindex: forge-extras-debug
The variable ~forge-extras-debug~, when set to a non-nil value, enables
verbose debugging messages throughout =forge-extras.el=.  These messages
are emitted via ~forge-extras-message-debug~ and appear in the
=*Messages*= buffer.  The default value is ~nil~.

This is a plain ~defvar~, not a ~defcustom~, so set it in your
configuration or toggle it interactively with =M-: (setq
forge-extras-debug t)=.  It is useful for diagnosing failures in the
GraphQL API calls used by the project integration features
([[#h:github-project-integration][GitHub Project integration]]).

* Commands
:PROPERTIES:
:CUSTOM_ID: h:commands
:END:

** Storing Org links for commits and topics
:PROPERTIES:
:CUSTOM_ID: h:store-org-links
:END:

#+findex: forge-extras-orgit-store-link
A common workflow when writing notes or documentation is to link back to
specific commits or forge topics.  The command
~forge-extras-orgit-store-link~ extends the standard ~org-store-link~ to
handle two Magit/Forge contexts:

- When multiple commit sections are selected in a Magit log or status
  buffer (via the region), it iterates through each selected commit and
  stores an Org link for every one of them.  This is useful when you
  want to reference a range of commits without invoking ~org-store-link~
  repeatedly.

- When there is no active region of commit sections, the command
  determines whether point is on a pull request or an issue.  It visits
  the corresponding topic buffer and stores a link from there, so you
  get a properly formatted orgit link even when invoked from the topic
  list.

** Browsing the GitHub notification inbox
:PROPERTIES:
:CUSTOM_ID: h:browse-github-inbox
:END:

#+findex: forge-extras-browse-github-inbox
The command ~forge-extras-browse-github-inbox~ opens the GitHub
notifications page in your default browser.  It is a thin wrapper around
~browse-url~ and is handy as a quick escape hatch when you want to
triage notifications in the web interface rather than in Forge.

** Toggling issue or pull request state
:PROPERTIES:
:CUSTOM_ID: h:toggle-state
:END:

#+findex: forge-extras-state-set-dwim
The command ~forge-extras-state-set-dwim~ provides a single key binding
that toggles the state of the current topic: if the issue or pull
request is open, it closes it (setting the state to ~completed~); if it
is already completed, it reopens it (setting the state to ~open~).

The command operates on the topic at point or, if none is found, the
topic associated with the current buffer.  You can also pass an ISSUE
argument programmatically.

** Pulling notifications
:PROPERTIES:
:CUSTOM_ID: h:pull-notifications
:END:

#+findex: forge-extras-pull-notifications
The command ~forge-extras-pull-notifications~ calls
~forge-pull-notifications~ silently, suppressing all output via
=shut-up= and =with-no-warnings=.  It also guards against running when
an Elfeed update is in progress (detected via
~elfeed-extras-auto-update-in-process~), because concurrent database
access was found to cause problems.

This command is primarily intended for use in timers or hooks (for
example, to refresh notifications periodically in the background) rather
than direct interactive invocation.

*** Re-entrancy protection
:PROPERTIES:
:CUSTOM_ID: h:reentrant-protection
:END:

Forge queries involve synchronous GPG decryption (via ~epg-wait-for-status~)
and network waits (via ~url-retrieve-synchronously~), both of which call
~accept-process-output~.  This services the Emacs event loop, allowing
timers to fire re-entrantly.  Each re-entrant ~forge--query~ blocks on
GPG again, creating unbounded nesting that can freeze Emacs.

Two guards prevent this.  The command-level guard
~forge-extras--pull-in-progress~ prevents re-entrant calls through
~forge-extras-pull-notifications~ itself.  The lower-level guard
~forge-extras--query-active~, installed as ~:around~ advice on
~forge--query~, catches re-entrant calls that enter forge through any
other path (such as callback hooks or other timers).

** Graceful handling of unsupported notification types
:PROPERTIES:
:CUSTOM_ID: h:massage-notification-gracefully
:END:

#+findex: forge-extras-massage-notification-gracefully
GitHub occasionally introduces new notification types (such as
CheckSuite) that Forge does not yet know how to parse.  When
~forge--ghub-massage-notification~ encounters one of these, it signals
an error, which aborts the entire notification pull.

The function ~forge-extras-massage-notification-gracefully~ is installed
as ~:around~ advice on ~forge--ghub-massage-notification~.  It wraps the
original function in a ~condition-case~: if the call succeeds, its
return value is passed through unchanged; if it signals an error, the
advice returns ~nil~ instead.  Because ~forge-pull-notifications~ feeds
the results through ~seq-keep~, a ~nil~ return causes the unsupported
notification to be silently filtered out rather than crashing the pull.

** Notification and read-status synchronization
:PROPERTIES:
:CUSTOM_ID: h:notification-and-read-status
:END:

#+findex: forge-extras-sync-read-status
When you visit a topic in Forge, GitHub may still show it as unread
because Forge only updates local notification state, not GitHub's.  The
function ~forge-extras-sync-read-status~ bridges this gap by calling the
GitHub REST API (~PATCH /notifications/threads/{thread-id}~) to mark
the notification thread as read.

The function only acts when the topic at point has an ~unread~ status in
Forge.  It looks up the corresponding ~forge-notification~ object to
obtain the GitHub thread ID, then issues the API call asynchronously
via ~forge--rest~.  On success, it refreshes the notification count in
Doom Modeline (if ~doom-modeline-github~ is enabled).

#+findex: forge-extras-sync-unread-from-github
The complementary function ~forge-extras-sync-unread-from-github~
handles the reverse direction: it ensures that notifications marked as
unread on GitHub's web interface are also shown as unread in Forge.
This is needed because marking a notification as unread on GitHub does
not update its ~updated_at~ timestamp, so Forge's incremental pull
(which uses a ~since~ filter based on that timestamp) skips those
notifications entirely.

The function is installed as ~:after~ advice on
~forge--ghub-update-notifications~, so it runs automatically at the end
of every ~forge-pull-notifications~ call.  It queries the GitHub REST
API for currently-unread notifications via the =gh= CLI, then sets the
local topic status to ~unread~ for any that Forge missed.

** Tracking repositories
:PROPERTIES:
:CUSTOM_ID: h:track-repository
:END:

#+findex: forge-extras-track-repository
When you clone a new repository or want Forge to know about one that
predates your Forge setup, the command ~forge-extras-track-repository~
adds a directory's Git repository to the Forge database.  It accepts an
optional DIR argument (defaulting to ~default-directory~), verifies that
the directory is a Git repository (using ~vc-extras-is-git-dir-p~ from
=vc-extras.el=), retrieves the remote URL, inserts the repository into
the Forge database with ~forge-get-repository~ using the =:insert!=
keyword, pulls all topics via ~forge--pull~, and opens a Magit status
buffer for the directory.

If you have a URL or local path rather than a directory with an
existing checkout, the supporting function
~forge-extras-track-repo-all-topics~ handles the lower-level logic of
resolving URLs and pulling topics.

** Navigating messages in a topic buffer
:PROPERTIES:
:CUSTOM_ID: h:navigate-messages
:END:

#+findex: forge-next-message
#+findex: forge-previous-message
When reading a long discussion thread in a Forge topic buffer, it is
convenient to jump between posts without scrolling.  The command
~forge-next-message~ moves point to the start of the next post section,
and ~forge-previous-message~ moves to the previous one.

If point is not currently on a post section, ~forge-next-message~ jumps
to the first post and ~forge-previous-message~ jumps to the last.  When
there is no further post in the requested direction, a message is
displayed in the echo area.

Both commands delegate to the internal function ~forge--goto-message~,
which navigates by inspecting Magit section siblings of type ~post~.

** Copying a message body
:PROPERTIES:
:CUSTOM_ID: h:copy-message
:END:

#+findex: forge-extras-copy-message-at-point-as-kill
The command ~forge-extras-copy-message-at-point-as-kill~ copies the body
of the Forge post at point to the kill ring, preserving its original
formatting.  This is useful when you want to quote or reuse a comment
from a GitHub discussion in another context, such as an email or an Org
document.

** Inserting Markdown links to GitHub issues
:PROPERTIES:
:CUSTOM_ID: h:insert-issue-link
:END:

#+findex: forge-extras-insert-issue-markdown-link
The command ~forge-extras-insert-issue-markdown-link~ fetches issues from
one or more GitHub repositories, presents them in a completing-read
prompt, and inserts a Markdown-formatted link at point.  This is
particularly useful when writing commit messages, pull request
descriptions, or documentation that needs to reference specific issues.

When called interactively, the command prompts for repositories as a
space- or comma-separated list of ="owner/repo"= strings.  It then calls
the =gh= CLI to fetch issues whose state matches
~forge-extras-issue-link-state~ ([[#h:issue-link-state][Issue link state]]),
up to 300 per repository.

The completion candidates are displayed as ="owner/repo #NUMBER: TITLE"=,
sorted by repository name (ascending) and then by issue number
(descending, so the most recent issues appear first within each
repository).  Upon selection, a Markdown link of the form
=[owner/repo#NUMBER](URL)= is inserted at point.

#+findex: forge-extras-test-fetch-issues
The companion command ~forge-extras-test-fetch-issues~ is a debugging
aid.  It fetches issues for a single repository and echoes the count,
helping you verify that the =gh= CLI is authenticated and working.

* GitHub Project integration
:PROPERTIES:
:CUSTOM_ID: h:github-project-integration
:END:

The GitHub Projects v2 features in =forge-extras.el= let you manage
project board metadata---status, estimates, and item listings---directly
from Emacs.  All of these commands communicate with the GitHub GraphQL
API by shelling out to the =gh= CLI, which must be installed and
authenticated.

** Setting up GitHub Project integration
:PROPERTIES:
:CUSTOM_ID: h:github-project-setup
:END:

Before you can use any of the project commands, several user options need
to be configured.  The process involves discovering various Node IDs from
GitHub's GraphQL API:

1. *Set the project owner and number.*  Configure
   ~forge-extras-project-owner~ ([[#h:project-owner-and-number][Project owner and number]])
   with the organization or user name, and ~forge-extras-project-number~
   with the integer from the project URL.

2. *Obtain the project Node ID.*  Use the =gh= CLI:
   #+begin_src sh
   # For an organization-owned project:
   gh api graphql -f query='query{organization(login:"YOUR_ORG"){projectV2(number:NUMBER){id}}}'

   # For a user-owned project:
   gh api graphql -f query='query{user(login:"YOUR_USER"){projectV2(number:NUMBER){id}}}'
   #+end_src
   Set ~forge-extras-project-node-id~ ([[#h:project-node-id][Project Node ID]])
   to the returned ID string.

3. *Discover field Node IDs.*  Run =M-x forge-extras-get-project-field-ids=
   ([[#h:get-project-field-ids][Getting project field IDs]]).
   From the resulting buffer, copy the Node IDs for the "Status" and
   "Estimate" fields into ~forge-extras-status-field-node-id~
   ([[#h:status-field-node-id][Status field Node ID]]) and
   ~forge-extras-estimate-field-node-id~
   ([[#h:estimate-field-node-id][Estimate field Node ID]]).

4. *Discover status option IDs.*  Run =M-x forge-extras-discover-project-status-options=
   ([[#h:discover-status-options][Discovering status options]]).
   The output buffer contains an alist ready for insertion into your
   configuration as the value of ~forge-extras-status-option-ids-alist~
   ([[#h:status-option-ids-alist][Status option IDs alist]]).

** Setting project status
:PROPERTIES:
:CUSTOM_ID: h:set-project-status
:END:

#+findex: forge-extras-set-project-status
The command ~forge-extras-set-project-status~ sets the "Status"
single-select field for the current topic within the configured GitHub
Project.  When called interactively, it defaults ISSUE to
~forge-current-topic~ and prompts for a status from the keys of
~forge-extras-status-option-ids-alist~
([[#h:status-option-ids-alist][Status option IDs alist]]).  The current
project status of the issue is offered as the default in the prompt.

If the issue is not yet a member of the target project, the command
automatically adds it before setting the status.  After a successful
update, if you are in a ~forge-topic-mode~ buffer, the buffer is
refreshed to reflect the change.

You can also call this command programmatically with explicit ISSUE and
STATUS arguments, bypassing the prompt.

** Setting project estimates
:PROPERTIES:
:CUSTOM_ID: h:set-project-estimate
:END:

#+findex: forge-extras-set-project-estimate
The command ~forge-extras-set-project-estimate~ sets a numeric estimate
(story points, hours, or whatever your project uses) for the current
topic within the configured GitHub Project.  It reads the current
estimate from the project data and offers it as the default when
prompting.

Like ~forge-extras-set-project-status~
([[#h:set-project-status][Setting project status]]), this command
automatically adds the issue to the project if it is not already present.
It requires ~forge-extras-estimate-field-node-id~
([[#h:estimate-field-node-id][Estimate field Node ID]]) to be configured.

** Discovering project IDs
:PROPERTIES:
:CUSTOM_ID: h:discover-project-ids
:END:

*** Getting project field IDs
:PROPERTIES:
:CUSTOM_ID: h:get-project-field-ids
:END:

#+findex: forge-extras-get-project-field-ids
The command ~forge-extras-get-project-field-ids~ queries the GitHub
GraphQL API for all fields defined in the project identified by
~forge-extras-project-node-id~ and displays their names and Node IDs in
a formatted buffer named =*GitHub Project Fields*=.

This is the primary tool for populating
~forge-extras-status-field-node-id~
([[#h:status-field-node-id][Status field Node ID]]) and
~forge-extras-estimate-field-node-id~
([[#h:estimate-field-node-id][Estimate field Node ID]]).

*** Discovering status options
:PROPERTIES:
:CUSTOM_ID: h:discover-status-options
:END:

#+findex: forge-extras-discover-project-status-options
The command ~forge-extras-discover-project-status-options~ fetches all
available option names and Node IDs for the "Status" single-select field
configured via ~forge-extras-status-field-node-id~.  The results are
displayed in a =*GitHub Project Status Options*= buffer, formatted as an
alist that you can copy directly into your configuration for
~forge-extras-status-option-ids-alist~
([[#h:status-option-ids-alist][Status option IDs alist]]).

Both ~forge-extras-project-node-id~ and ~forge-extras-status-field-node-id~
must be set before running this command.

*** Looking up a single status option ID
:PROPERTIES:
:CUSTOM_ID: h:get-status-option-id
:END:

#+findex: forge-extras-get-project-status-option-id
If you already know the name of a status option (for example, "Waiting")
and need its Node ID, the command
~forge-extras-get-project-status-option-id~ provides a shortcut.  It
prompts for the status name, fetches all options for the configured
status field, and displays the matching Option ID in the echo area.  The
search is case-insensitive.

If no match is found, the error message lists all available option names
so you can correct a typo.

** Listing project items
:PROPERTIES:
:CUSTOM_ID: h:list-project-items
:END:

*** Listing items for a specific repository
:PROPERTIES:
:CUSTOM_ID: h:list-items-by-repo
:END:

#+findex: forge-extras-list-project-issues-by-repo-ordered
The command ~forge-extras-list-project-issues-by-repo-ordered~ fetches
issues and pull requests from a single repository within the configured
project and displays them in the order they appear on the project board.
It prompts for the repository in ="owner/repo"= format and shows the
results in a =*Project Issues for owner/repo*= buffer.

This command includes both open and closed items and returns the list of
item plists for programmatic use.

*** Listing all project items
:PROPERTIES:
:CUSTOM_ID: h:list-all-items
:END:

#+findex: forge-extras-list-project-items-ordered
The command ~forge-extras-list-project-items-ordered~ fetches all issues
and pull requests across every repository in the configured project,
handling pagination automatically.  Items are returned in project board
order.

The command's behavior depends on whether a prefix argument is provided:

- *Without a prefix argument:* closed and merged items are excluded, and
  results are displayed in a =*All Project Items (Ordered by Board)*=
  buffer.
- *With a prefix argument* (=C-u=): closed and merged items are included,
  and the results buffer is not displayed.  This is useful for
  populating the cache programmatically.

Programmatic callers can pass a third argument USE-CACHE-P.  When
non-nil and the cache variable ~forge-extras--cached-project-items~ is
populated, the cached data is returned immediately without a network
request.  Note that in this case, the INCLUDE-CLOSED-P argument does not
re-filter the cached items---it only applies when fresh data is fetched.

The fetched (or cached) items are always stored in
~forge-extras--cached-project-items~ for subsequent use.

* Functions
:PROPERTIES:
:CUSTOM_ID: h:functions
:END:

This section documents public non-interactive functions that are useful
for customization, hooks, or extending the package.

** Repository tracking
:PROPERTIES:
:CUSTOM_ID: h:repo-tracking-functions
:END:

#+findex: forge-extras-track-repo-all-topics
The function ~forge-extras-track-repo-all-topics~ is the workhorse
behind ~forge-extras-track-repository~
([[#h:track-repository][Tracking repositories]]).  It accepts a URL or
local path, resolves it to a repository, inserts it into the Forge
database, and pulls all topics.  You can call it directly when you have a
URL rather than a local directory.

** GraphQL API functions
:PROPERTIES:
:CUSTOM_ID: h:graphql-api-functions
:END:

#+findex: forge-extras-gh-get-issue-fields
#+findex: forge-extras-gh-parse-issue-fields
#+findex: forge-extras-gh-get-project-fields
#+findex: forge-extras-gh-parse-project-fields
#+findex: forge-extras-gh-add-issue-to-project
#+findex: forge-extras-gh-update-project-item-status-field
#+findex: forge-extras-gh-update-project-item-estimate-field
The GitHub Project integration is built on several public functions that
wrap the GraphQL API.  These can be useful if you want to build custom
workflows:

- ~forge-extras-gh-get-issue-fields~ fetches raw project field data for
  a given ISSUE-NUMBER and REPO-NAME.
- ~forge-extras-gh-parse-issue-fields~ parses that raw data into a
  property list with keys like ~:issue-node-id~, ~:status~, ~:effort~,
  ~:project-item-id~, and more.
- ~forge-extras-gh-get-project-fields~ fetches all field definitions for
  a given PROJECT-NODE-ID.
- ~forge-extras-gh-parse-project-fields~ parses the result into a list
  of =(NAME . ID)= cons cells.
- ~forge-extras-gh-add-issue-to-project~ adds an issue to a project and
  returns the new project item Node ID.
- ~forge-extras-gh-update-project-item-status-field~ updates a
  single-select field (typically "Status") on a project item.
- ~forge-extras-gh-update-project-item-estimate-field~ updates a numeric
  field (typically "Estimate") on a project item.
