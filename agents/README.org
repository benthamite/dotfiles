#+title: Agent configuration synchronization
#+author: Pablo Stafforini

* Overview

This document describes how the Claude Code and Codex configurations in this
repo stay aligned. Tool-specific setup details remain in
=claude/README.org= and =codex/README.org=.

This file is the authoritative cross-tool topology reference. If a tool-specific
README conflicts with this file about where a shared artifact belongs, follow
this file.

The synchronization model is intentionally manual, not auto-generated. When an
agent changes a reusable Claude or Codex artifact, whether global or
project-local, hooks remind it to update the counterpart during the same
session. Commit guards then block one-sided commits unless the changed artifact
is synchronized. For global dotfiles artifacts, =ai-config-sync.json= can also
record an explicit artifact-specific status or exemption.

* Topology

| Path | Role | Tracked? | Live consumer |
|------+------|----------|---------------|
| =claude/CLAUDE.md= | Claude global instructions | Yes | =~/.claude/CLAUDE.md= symlink |
| =codex/AGENTS.md= | Codex global instructions | Yes | =~/.codex/AGENTS.md= symlink |
| =CLAUDE.md= and =AGENTS.md= | Dotfiles project instructions | Yes | Loaded only when working in this repo |
| =claude/skills/= and =codex/skills/= | Global reusable skills | Yes | =~/.claude/skills= and =~/.codex/skills= symlinks |
| =claude/hooks/= and =codex/hooks/= | Hook script implementations | Yes | Referenced by live hook registrations |
| =~/.claude/settings.json= | Claude global settings and hook registration | No | Real mutable file, edited in place |
| =~/.claude/settings.local.json= | Claude machine-local permissions | No | Real mutable file, edited in place |
| =codex/hooks.json= | Codex global hook registration | Yes | =~/.codex/hooks.json= symlink |
| =codex/config.toml= | Codex global settings | Yes | =~/.codex/config.toml= symlink |
| =<project>/.claude/= and =<project>/.codex/= | Project-local tool config | Project-dependent | Loaded only for that project |

Repo-root =.claude/= and =.codex/= paths are project-local by definition. Do
not use them as targets for global =~/.claude/= or =~/.codex/= symlinks. In this
dotfiles repo, reusable skills belong in =claude/skills/= and =codex/skills/=.
Other projects may use =.claude/skills/= with =.codex/skills/=, and
=.claude/hooks/= with =.codex/hooks/=, for project-specific behavior. The sync
reminder and commit guard treat those local pairs like global pairs inside the
target project repo.

* Peer Model

Claude and Codex artifacts are peer implementations of shared agent behavior.
Either side may be edited first. When the tools need different syntax,
discovery rules, hook payloads, or activation files, both files are kept as
explicit counterparts and =bin/ai-config-sync= checks equivalence under
tool-specific normalization.

The tracked =claude/CLAUDE.md= and =codex/AGENTS.md= files are paired
user-level instruction files. The live =~/.codex/AGENTS.md= symlink points to
the tracked Codex file.

Project-level instructions are also paired. Every project-local =CLAUDE.md=
should have a sibling =AGENTS.md= counterpart. The sync checker treats
=CLAUDE.md= and =AGENTS.md= pairs anywhere in the command's target git
repository as project-local instruction peers, just as it treats =.claude/skills/= and
=.codex/skills/= as project-local skill peers. Codex does not use
=project_doc_fallback_filenames= to read =CLAUDE.md=; project Codex
instructions live in =AGENTS.md=.

Instruction equivalence checks normalize the Claude-only =@context= syntax and
Codex-safe path references before comparing paired instruction files.

* Skills

=claude/skills/<name>/= and =codex/skills/<name>/= should contain equivalent
skill bodies and auxiliary files. The content check removes tool-specific
frontmatter keys such as =allowed-tools=, =argument-hint=, =model=,
=user-invocable=, and argument metadata from both sides before comparison.
Runtime output directories such as =digests/=, =last-run/=, and
=twitter-vet/vetted/= are intentionally ignored.

Project-local skills follow the same rule inside their own repository:
=.claude/skills/<name>/= and =.codex/skills/<name>/= should contain equivalent
skill bodies and auxiliary files. The post-edit reminders and commit guard use
the same frontmatter normalization for local skills.

Codex discovers global skills from =~/.codex/skills/*/SKILL.md=. Claude
discovers global skills from =~/.claude/skills/*/SKILL.md=. The tracked skill
trees are activated by:

#+begin_src sh
codex/bin/install-skills.sh
#+end_src

The installers keep =~/.claude/skills= and =~/.codex/skills= linked to the
tracked skill trees. =skills-disabled.json= records stale global skills that
=bin/skill-prune= disables through native tool settings: Claude
=skillOverrides= in the live =~/.claude/settings.local.json= and Codex
=[[skills.config]]= entries in =codex/config.toml=. Run
=bin/skill-prune apply --days 60= to disable paired skills that have no recent
usage in either Claude or Codex; a recent use in either tool keeps both tools
enabled. Local reusable skills should live in both =claude/skills/= and
=codex/skills/= instead of living only under a live home-directory symlink.
Hidden Codex-managed system skills may exist under ignored
=codex/skills/.system/= runtime data, but they are not part of the reusable
skill parity model.

* Hooks

Hook scripts are paired when the two tools expose compatible events and payloads.
The Codex counterparts live in =codex/hooks/= and are registered from the tracked
=codex/hooks.json= file, which is live globally through the
=~/.codex/hooks.json= symlink. Claude hooks live in =claude/hooks/= and are
registered in the mutable live =~/.claude/settings.json=.

Project-local hook scripts are paired the same way: =.claude/hooks/<name>=
with =.codex/hooks/<name>= inside the target repo. Project-local registration
files are paired as activation state: =.claude/settings.json= with
=.codex/hooks.json=.

Some hooks cannot be synchronized one-for-one:

- Claude =Read=/=Grep=/=Glob= hooks do not have equivalent Codex events here.
- Claude =Skill= hooks do not have equivalent Codex skill hook events here.
- Claude Notification hooks are replaced by Codex lifecycle forwarding through
  =codex-hook-wrapper= entries in =codex/hooks.json=.
- Chrome-specific Claude hooks have no Codex counterpart.

These cases are recorded in =ai-config-sync.json= rather than treated as drift.

* Guard Rails

=bin/ai-config-sync= provides three kinds of checks:

- =remind-claude= and =remind-codex= read hook JSON and print the counterpart to
  update after relevant global or project-local edits.
- =guard-commit= blocks one-sided commits. It inspects staged files and common
  one-shot shell forms such as =git add ... && git commit=, =git add -A=,
  =git commit -a=, quoted paths, absolute paths, and symlinked =~/.claude= paths.
  The guard resolves the git repo targeted by the commit command, so it checks
  project-local sibling =CLAUDE.md=/=AGENTS.md=, =.claude/=, and =.codex/= pairs in that
  repo as well as global dotfiles pairs. For skills, it checks content parity
  after tool-specific frontmatter normalization rather than accepting any
  arbitrary counterpart file. Project-local skill changes must also update the
  same repo's =README.org= =Skills= section. Tool-specific README files document
  only that tool's behavior and paths; project-local README files document only
  the local project behavior, not the other agent's directory layout.
- =audit= verifies the manifest, topology invariants, live symlinks, global
  instruction equivalence, skill content parity after tool-specific frontmatter normalization,
  auxiliary skill files, hook file inventory, hook registrations, the live skill
  symlinks, and Codex activation files. Topology checks include
  forbidden dotfiles-local =.codex/hooks.json= files, live symlinks resolving
  into project-local config, undocumented tool-home symlinks, and dotfiles
  project-local skills. It also checks the dotfiles repo-root
  =CLAUDE.md=/=AGENTS.md= pair.

Run the deterministic audit with:

#+begin_src sh
bin/ai-config-sync audit
#+end_src

* When Changing Config

1. Edit the side requested by the user.
2. Update the counterpart in the same session.
3. For global dotfiles artifacts, if no counterpart should exist, update
   =ai-config-sync.json= with the explicit artifact-specific status or
   exemption.
4. Run =bin/ai-config-sync audit= before committing global dotfiles config.

* Limits

The system can keep the shared artifacts synchronized, but it cannot make Claude
Code and Codex fully behaviorally identical. Their global settings formats,
permission systems, hook event surfaces, tool names, model settings, MCP/plugin
systems, and lifecycle events differ. The sync target is therefore explicit
parity where possible plus documented non-parity where the tools differ.
