*** Directory topology
:PROPERTIES:
:CUSTOM_ID: h:directory-topology
:ID:       EB7422C8-E371-4F9B-AE1F-D6B6473F11BB
:END:

Three layers of directories work together: this dotfiles directory (source of truth), the shared =~/.claude/= hub, and per-account overlay directories.

*Source of truth* (version-controlled in this repo):

#+begin_example
~/My Drive/dotfiles/claude/           ← global config source of truth
├── CLAUDE.md                         ← global conventions for all projects
├── settings.json                     ← hooks, model, permissions (no theme; symlinked from ~/.claude/)
├── settings.local.json               ← machine-specific permissions (mpv, brew, pip3…)
├── hooks/                            ← 24 hook scripts
├── skills/                           ← 50 skill directories
├── bin/batch-test.sh                 ← elpaca-aware emacs --batch test runner
├── bin/sync-mcp-servers.sh           ← MCP server sync utility
└── README.org                        ← this file
#+end_example

*Shared hub* (=~/.claude/=), mostly symlinks into the above:

#+begin_example
~/.claude/
├── CLAUDE.md          → dotfiles/claude/CLAUDE.md
├── settings.json      → dotfiles/claude/settings.json
├── settings.local.json → dotfiles/claude/settings.local.json
├── skills/            → dotfiles/claude/skills/
├── projects/          ← trust entries, session logs (managed by Claude Code)
├── history.jsonl      ← command history (managed by Claude Code)
└── plugins/           ← installed plugins (managed by Claude Code)
#+end_example

*Per-account overlays* (=~/.claude-tlon/=, =~/.claude-epoch/=): see [[*Multi-account setup][multi-account setup]] below.

*Per-project config* (=<project>/.claude/=):

#+begin_example
<project>/.claude/
├── settings.json       ← project-level hooks (optional)
├── settings.local.json ← project-specific permissions (accumulated interactively)
└── skills/             ← project-specific skills (optional)
#+end_example

*MCP servers* are defined in =~/.claude.json= (canonical) and synced to account directories by =claude/bin/sync-mcp-servers.sh=.

*** Settings hierarchy
:PROPERTIES:
:CUSTOM_ID: h:settings-hierarchy
:ID:       ABB0C5CA-1B40-47BF-8746-BFFE1F4E420D
:END:

Claude Code loads settings from multiple levels, merged together. Each level can define permissions, hooks, and other options. Permissions are additive across levels. Hooks run from all levels that define them.

| Level | =settings.json= | =settings.local.json= | Purpose |
|-------+------------------+-----------------------+---------|
| User (=~/.claude/=) | Model, theme, global hooks, global permissions | Machine-specific tool whitelists | Applies to all projects |
| Project (=<project>/.claude/=) | Project-specific hooks | Project-specific permissions | Applies only in that project |

=settings.json= is version-controlled; =settings.local.json= is gitignored (machine-specific or accumulated interactively).

The global =settings.json= uses =auto= mode (Anthropic's classifier-backed permission mode that approves most actions while flagging risky ones) inside the =permissions= object. All MCP servers are explicitly listed in =permissions.allow= as a workaround for [[https://github.com/anthropics/claude-code/issues/5140][anthropics/claude-code#5140]]. Safety hooks (=guard-external-actions.sh=, =block-destructive-command.sh=, etc.) provide additional guardrails.

*** =CLAUDE.md= loading
:PROPERTIES:
:CUSTOM_ID: h:claude-md-loading
:ID:       D03891CF-C905-480A-9690-62B4F8A7273D
:END:

Claude Code loads =CLAUDE.md= files from several locations. When working in a project, it loads:

1. =~/.claude/CLAUDE.md= — user-level global conventions (symlinked to =dotfiles/claude/CLAUDE.md=)
2. =<project>/CLAUDE.md= — project-level instructions (if present)

Each file should contain distinct content. The global one defines behavioral conventions (verify work, fix root causes, commit immediately, etc.). Project-level ones define project-specific context.

| New MCP server | =~/.claude.json=, then run =claude/bin/sync-mcp-servers.sh= | Syncs to account dirs |

** Multi-account setup
:PROPERTIES:
:CUSTOM_ID: h:multi-account
:ID:       239E0918-2383-460C-8C9C-2ABC4E3D344D
:END:

Two OAuth accounts (tlon and epoch) are supported without re-authenticating. Each account uses a separate =CLAUDE_CONFIG_DIR= (=~/.claude-tlon/= and =~/.claude-epoch/=) to keep its OAuth credentials and identity isolated, while sharing everything else via symlinks back to =~/.claude/=.

Shared (symlinked into each account dir):
- =settings.json=, =settings.local.json= — preferences and permissions
- =skills/= — custom skills
- =projects/= — trusted folders, project config, session logs
- =history.jsonl= — command history
- =plugins/= — installed plugins

Not symlinked:
- =CLAUDE.md= — loaded from =~/.claude/CLAUDE.md= at user level; account dirs do not have their own copy (to avoid loading the same instructions twice)

Separate per account:
- =.claude.json= — OAuth identity, user ID, and MCP servers (synced from =~/.claude.json=)
- Keychain credentials — the actual OAuth tokens

The active account is stored in =~/.claude-current-account=. From Emacs, =claude-code-extras-start-or-switch= uses the persisted account automatically (configured via =claude-code-extras-accounts=). Use =claude-code-extras-select-account= to change it. From the terminal, the shell aliases =claude-tlon= and =claude-epoch= are available.

*1Password service account:* All =op://= references resolve silently via =OP_SERVICE_ACCOUNT_TOKEN= (a 1Password service account token with read-only access to the "Automations" vault).  The token is stored in =pass= at =epoch/1password-service-account-token= and exported in =.zshenv-secrets=.  No Touch ID prompts are needed.  To rotate: delete the old service account in the admin console, create a new one with =op service-account create "Epoch Ops Automation" --vault "Automations:read_items" --raw=, and update the =pass= entry.  Item names in the Automations vault must not contain parentheses (they break =op://= parsing).

*Account-specific MCP secrets:* When an MCP server needs different credentials per account, place =op://= references (or plain text values) directly in each per-account =.claude.json= file's MCP server =env= block.  Claude Code resolves =op://= natively.  The sync function deep-merges =mcpServers= per-server, preserving per-account =env= entries.  To add a new account-specific secret: (1) add the =env= entry to each per-account =.claude.json= (e.g. =~/.claude-tlon/.claude.json=, =~/.claude-epoch/.claude.json=), (2) leave the canonical =~/.claude.json= with an empty =env= for that server.

*MCP servers:* Claude Code reads =.claude.json= from =$CLAUDE_CONFIG_DIR= (not =$HOME=) when that variable is set. The canonical source of truth is =~/.claude.json=; run =claude/bin/sync-mcp-servers.sh= after adding or removing any MCP server there.

*** Chrome integration and multi-account

Chrome integration uses two Chrome profiles, each with the Claude in Chrome extension installed and logged into a different Anthropic account. This allows =/chrome= to work from both the epoch and tlon Claude Code accounts without switching extension logins.

*Chrome profiles:*

| Chrome profile | Google account              | Extension logged into    | Claude Code account |
|----------------+-----------------------------+--------------------------+---------------------|
| Default        | =pablo.stafforini@gmail.com=  | =tlon.shared@gmail.com=    | tlon                |
| Profile 2      | =pablo@epoch.ai=              | =pablo@epoch.ai=           | epoch               |

Each Claude Code account stores its own =chromeExtension.pairedDeviceId= and =pairedDeviceName= in its =.claude.json=. These are *not* synced across accounts (unlike =claudeInChromeDefaultEnabled=, which is in =claude-code-extras--shared-claude-json-keys=) because each account pairs with a different Chrome profile's extension instance.

*How profile selection works:* when a Claude Code session starts with =--chrome=, it connects to the Chrome extension instance that matches the session's OAuth account. The extension in each Chrome profile authenticates independently via native messaging. The =pairedDeviceId= links a specific Claude Code instance to a specific extension instance.

*Native host manifest:* there is a single manifest file shared across all Chrome profiles. Its =path= field should point to the current account's =chrome-native-host= script (e.g. =~/.claude-epoch/chrome/chrome-native-host=). In practice, all account scripts are identical (they exec the same =claude= binary with =--chrome-native-host=), so the path doesn't affect functionality today — but it should be kept in sync for correctness.

*Emacs-specific:* non-interactive sessions (Emacs) require the =--chrome= flag, set via =(setq claude-code-program-switches '("--chrome"))=. Without it, Claude Code silently skips Chrome MCP startup.

*Known issues:*
- *Silent bridge failure:* the Chrome bridge can fail to connect at session startup (e.g. due to stale sockets or timing issues). When this happens, =/chrome= shows "Status: Disabled" even though =--chrome= was passed. The only fix is starting a new session. There is no mid-session reconnect mechanism.
- *Stale bridge sockets:* old sockets in =/tmp/claude-mcp-browser-bridge-pablostafforini/= are not automatically cleaned up and may interfere with new connections.
- *Session-account binding:* if you switch accounts mid-session (e.g. via =claude-code-extras-select-account=), Chrome will still use the original account. The running process retains its =CLAUDE_CONFIG_DIR=; you must start a fresh session under the new account — use =/handoff= to preserve context if needed.

*Architecture:*
- Chrome extension (ID =fcoeoabgfenejglbffodgkkbkcdhcgfn=) communicates via native messaging
- Native host manifest: =~/Library/Application Support/Google/Chrome/NativeMessagingHosts/com.anthropic.claude_code_browser_extension.json=
- Native host scripts: =~/.claude-{epoch,tlon}/chrome/chrome-native-host= (each execs the installed =claude= binary with =--chrome-native-host=)
- Bridge socket: =/tmp/claude-mcp-browser-bridge-pablostafforini/<pid>.sock=
- Keychain credentials per account: =Claude Code-credentials-<sha256(config_dir)[:8]>=

** settings.json
:PROPERTIES:
:ID:       C60E07A6-7A22-4272-8875-45FE9B44F185
:END:

- *Permission mode*: =auto= — a background classifier allows most actions while blocking risky ones (force pushes, mass deletion, sending secrets to external endpoints, etc.).
- *Agent teams*: enabled.
- *Effort level*: high.

*Symlink breakage risk:* =~/.claude/settings.json= is a symlink to =dotfiles/claude/settings.json=. Claude Code (Node.js) may use atomic writes (temp file + rename) when updating settings, which replaces the symlink with a regular file. If =~/.claude/settings.json= stops being a symlink, the tracked source and the deployed file will silently diverge. To detect this, check =ls -la ~/.claude/settings.json= — it should show =->= pointing to the dotfiles path. To fix: =trash ~/.claude/settings.json && ln -s "/Users/pablostafforini/My Drive/dotfiles/claude/settings.json" ~/.claude/settings.json=.

** CLAUDE.md
:PROPERTIES:
:ID:       9B404E8C-F109-4188-9F3E-1F1A9A64A37D
:END:

Project-level instructions referencing the =dotfiles-context= skill. The global =~/.claude/CLAUDE.md= (symlinked here) defines conventions that apply across all projects:

- Fix root causes, not symptoms; no silent fallbacks. Prefer robust solutions over hacks, even if they take longer.
- Verify fixes before declaring them done.
- Commit all changes immediately; prefer amending when iterating on the same logical change (unless already pushed).
- Pre-commit checklist (hook-enforced): Elisp pre-commit checks (batch-test, stage =doc/*.org=) are in the =elisp-conventions= skill; the =README.org= staging rule is in the project =CLAUDE.md=.
- Use subagents and teams liberally; prefer the most powerful model.
- Use =trash= instead of =rm -rf=; never send signals to Emacs without confirmation.
- Never take actions visible to others (push, PR, post) without explicit confirmation.

** block-destructive-command.sh (PreToolUse)

Blocks or gates destructive shell commands that are hard to reverse. Hard-blocks (deny): =rm -rf= (suggests =trash=), =git push --force=, =git reset --hard=, =git clean -f=, =git checkout -- .=, and =git branch -D=. Prompts for confirmation (ask): =git push= (all pushes), =git clone= / =gh repo clone=. Each pattern includes a specific remediation suggestion. Users can run blocked commands manually via =!=.

** guard-external-actions.sh (PreToolUse)

Deny-by-default guard for all MCP tools. Fires on all =mcp__= tools. Read-only tools are explicitly allowlisted (pattern-matched on verb prefixes like =read_*=, =list_*=, =get_*=, =search_*=, plus specific tool names per server). Everything else exits 2 to require user confirmation. This ensures new MCP servers and tools are guarded automatically without needing to enumerate every dangerous action.

** require-elisp-skill.sh (PreToolUse)

Blocks edits to =.el= files until the =elisp-conventions= skill has been loaded in the current session. Checks for a session marker created by the companion =track-elisp-skill.sh= hook. If no marker exists, denies the edit with a message telling the agent to load the skill first.

** track-elisp-skill.sh (PostToolUse)

Creates a session marker (=/tmp/claude-elisp-skill-{SESSION_ID}=) when the =elisp-conventions= skill is loaded via the Skill tool. This marker allows =require-elisp-skill.sh= to permit =.el= edits. Cleans up stale markers older than 2 hours.

** require-elisp-verify-after-commit.sh (PreToolUse)

Blocks all Bash commands (except =emacsclient= and =batch-test.sh=) after committing =.el= files until verification via =emacsclient -e= has been run. Uses a session marker created by the companion =track-elisp-verify.sh= hook. Ensures the agent verifies that committed Elisp loads correctly in the running Emacs before moving on.

** track-elisp-verify.sh (PostToolUse)

Creates a verification marker (=/tmp/claude-elisp-verify-needed-{SESSION_ID}=) after a successful =git commit= that includes =.el= files. Clears the marker when =emacsclient -e= runs. Skips test files (=*-test.el=, =*-tests.el=). Cleans up stale markers older than 2 hours.

** notify-emacs-notification.sh (Notification)

Forwards Claude Code notifications to Emacs via =claude-code-handle-hook=. Runs through =fire-and-forget.sh= so the CLI doesn't block on =emacsclient=.

** fire-and-forget.sh (utility)

Wrapper that runs a hook command in the background. Reads stdin first to avoid broken pipes, then execs the command detached. Used by the Notification and Stop hooks so the CLI doesn't block waiting for =emacsclient=.

** lib-repo-root.sh (utility)

Shared helper sourced by other hooks. Resolves the git repo root from the command context. PreToolUse hooks run before the bash command executes, so the hook's cwd may differ from the command's effective cwd. If the command starts with =cd <path> &&=, this helper resolves the repo root from that path and cds into it so subsequent git commands operate on the correct repo.

** lib-staged-files.sh (utility)

Shared helper sourced by other hooks. Lists staged files and handles the =--amend= flag correctly by comparing against =HEAD~1= when amending (since amending replays the previous commit's changes).
